## Per-ticket innovation notes — WHY each choice

### BON-602 — Init idempotency

**Floor coverage** (line 111-121): `bonfire.toml` content marker survives re-init.

**Innovation 1 — Parametrized scaffold-dir marker test** (`test_init_preserves_user_files_in_scaffolded_dirs`).
- The narrow ticket asks for `.bonfire/` and `agents/` directory + content
  idempotency. I parametrized over `[".bonfire", "agents"]` with the SAME marker
  shape so the policy intent is surfaced once, not duplicated per dir. A future
  regression that nukes only one of the two scaffolded dirs is still caught.
- **WHY parametrize:** the policy is symmetric across both dirs. Two near-
  identical tests would drift; one parametrized test reads as "every scaffolded
  dir is user territory."

**Innovation 2 — Recursive non-empty `agents/` tree preservation**
(`test_init_preserves_nonempty_agents_tree`).
- Models the realistic upgrade scenario: user has populated `agents/my-team/scout/`
  with custom agent files, then re-runs `bonfire init` after upgrading
  `bonfire-ai`. A naive `mkdir(exist_ok=True)` already passes this — but the
  test LOCKS that contract against a future regression where someone
  "fixes" idempotency by `rmtree` + recreate.
- **WHY this case:** real-world users WILL populate `agents/`; that's the
  point of the scaffold. The contract must explicitly cover the populated
  case, not just the empty-shell case.

**Innovation 3 — bonfire.toml mtime stability** (`test_init_does_not_thrash_existing_bonfire_toml_mtime`).
- The policy "no overwrite" is one shape of the wider policy "no write at
  all". Even an idempotent overwrite (writing identical bytes) bumps mtime
  and triggers downstream filewatchers (editors, build tools, CI cache
  invalidation). Today's `if not toml_path.exists(): write` already passes
  this, but the test locks the no-write semantic against future "I'll just
  always write to be safe" refactors.
- **Load-bearing vs nice-to-have:** I think this is **nice-to-have**. The
  no-clobber-content contract (#1, #2) is the load-bearing piece; mtime
  is a secondary invariant.

### BON-601 — `Traversable` type assertion

**Floor coverage** (line 180-236): mock `PersonaLoader`, assert `builtin_dir`
kwarg's `str()` contains `"persona"` and `"builtins"`.

**Innovation 4 — `isinstance(builtin_dir, Traversable)`**
(`test_loader_builtin_dir_is_traversable`).
- The narrow ticket spec is exactly this: assert kwarg is
  `importlib.resources.abc.Traversable` (not plain `pathlib.Path`). I
  implemented it canonically: `import importlib.resources.abc` + `isinstance`.
- **WHY canonical:** `pathlib.Path` IS-A `Traversable` (via stdlib
  `Traversable.register(Path)` in CPython), so the check passes for both the
  editable-install case (returns `PosixPath`) and the wheel case (returns a
  zip-aware Traversable). The check rejects `str`, `os.PathLike`, and any
  hand-rolled wrapper.

**Innovation 5 — Live Traversable interface probe**
(`test_loader_builtin_dir_supports_traversable_interface`).
- Beyond static `isinstance`, walk the live API: `is_dir()`, `joinpath()`,
  `iterdir()`, and assert each returns/behaves as Traversable.
- **WHY this case:** an `isinstance` check passes for a closed/stale handle
  too. A regression where someone wraps the value in a class that registers
  as Traversable but raises on `iterdir()` would slip past static type checks.
- **Load-bearing vs nice-to-have:** I think the static `isinstance` check
  (#4) is **load-bearing** — that's the literal ticket ask. The live-probe
  (#5) is **nice-to-have** drift insurance and may be Sage-prunable.

### BON-600 — Ctrl-C cleanup contract

**Floor coverage**: zero — no test exercises KeyboardInterrupt mid-`server.stop()`.

**Innovation 6 — Parametrized interrupt arrival points**
(`test_scan_prints_cleanup_on_interrupt_during_run`).
- Parametrized over `[interrupt_at_start, interrupt_after_partial_work]` —
  two `KeyboardInterrupt` shapes (no-arg vs with-message) raised from
  `_run_scan`. Both must surface `"Front Door closed."`.
- **WHY parametrize:** the cleanup contract is "interrupt at any point →
  visible cleanup". Hard-coding one interrupt shape would let a regression
  that depends on the exception's `args` slip through.

**Innovation 7 — Mid-`server.stop()` interrupt simulation**
(`test_scan_prints_cleanup_on_interrupt_during_server_stop`).
- The narrow ticket case: KeyboardInterrupt arrives WHILE `server.stop()`
  is awaiting. Simulated by a fake `_run_scan` coroutine whose `finally`
  block raises `KeyboardInterrupt` (mirroring how the real `_run_scan` calls
  `await server.stop()` in `finally`).
- **WHY this shape:** asyncio's interrupt-during-finally semantics are
  subtle. The fake coroutine collapses the asyncio-loop edge case into a
  deterministic, reproducible test. Without this, the ticket would ship
  with no concrete RED.
- **Load-bearing vs nice-to-have:** **load-bearing** — this is the literal
  ticket case.

**Innovation 8 — Cleanup text format stability**
(`test_scan_cleanup_text_format_is_stable`).
- Locks the EXACT phrase `"\nFront Door closed."` — leading newline,
  capitalized "Front Door" (proper-noun branding for the onboarding feature).
- **WHY:** the cleanup line is the user's only signal that cleanup ran.
  Drift to `"Closed."` or `"front door closed"` would pass any substring
  test that uses `result.output.lower()`.
- **Load-bearing vs nice-to-have:** **nice-to-have** — guards UX polish,
  not the core cleanup behavior. Sage may downgrade.

### BON-599 — TOML decode warning

**Floor coverage**: zero — `_get_active_persona` silently swallows
`TOMLDecodeError | OSError` per `persona.py:38`.

**Innovation 9 — Parametrized over four real TOML failure shapes**
(`test_get_active_persona_warns_on_any_toml_failure`).
- Parametrized over: truncated table, unclosed string, invalid escape,
  duplicate key. All four MUST emit a `WARNING`-or-higher log record.
- **WHY parametrize:** the policy intent of BON-599 is "any TOML parse
  failure → warning, not silent fallback." A test for one specific shape
  invites a fix that handles only that shape. Parametrizing surfaces the
  policy as "all TOML decode failures, no exceptions."
- **Load-bearing vs nice-to-have:** **load-bearing** — this IS the policy.

**Innovation 10 — Warning text names the file**
(`test_get_active_persona_warning_names_the_file`).
- Asserts the warning message contains the substring `"bonfire.toml"`.
- **WHY:** a warning that says only "TOML parse failed" is unactionable.
  The user must know WHICH file to edit. Innovation over a bare
  `caplog.records` count check.
- **Load-bearing vs nice-to-have:** **load-bearing** — debuggability is
  the entire reason warnings beat silent fallback. Without naming the
  file, the warning is barely better than silence.

**Innovation 11 — OSError parity** (`test_get_active_persona_warns_on_oserror`).
- The implementation today catches `(TOMLDecodeError, OSError)` together.
  The narrow ticket only names TOMLDecodeError; OSError gets the same
  silent-swallow regression site. Symmetry argues for the same warning
  treatment. Patches `tomllib.load` to raise `OSError` (avoids
  cross-platform permission-mode flakiness).
- **WHY this case:** if the implementer fixes only the TOMLDecodeError
  branch, OSError silently falls through. This test forces both branches
  to emit warnings.
- **Load-bearing vs nice-to-have:** **load-bearing** — the same silent-
  swallow site means the same fix should cover both.

## Contract questions for Sage

1. **xfail xpass policy.** Several innovation tests xpass today because the
   underlying impl already satisfies the wider contract (BON-602/601/600).
   Should those drop the `xfail` marker (Sage stripping pre-merge), or
   stay as xfail "RED state placeholders" until a deliberate impl-change
   commit graduates them?

2. **BON-602 mtime contract.** Today's `init` writes `bonfire.toml` only
   when missing — mtime is preserved by accident. Is mtime stability part
   of the locked contract, or implementation incidental? My test treats
   it as a contract; a Sage view that says "no — only content matters"
   would prune #3.

3. **BON-599 warning level.** I asserted `WARNING`-or-higher via
   `caplog.at_level(logging.WARNING)`. Is `WARNING` the locked level, or
   would `INFO` satisfy the policy? `WARNING` matches the `loader.py`
   pattern (`logger.warning("persona %r: persona.toml is malformed: %s")`)
   so I assumed yes — but Sage should confirm.

4. **BON-599 OSError treatment.** The narrow ticket names only
   `TOMLDecodeError`. Should OSError silently fall back (current behavior),
   warn (my proposal), or RAISE (it's a true IO error)? My #11 asserts
   warn. A "raise on OSError, warn on TOMLDecodeError" policy would
   require pruning #11 and adding a different shape.

5. **BON-600 cleanup text.** Is `"Front Door closed."` the locked phrase,
   or is the contract "any user-visible cleanup signal"? My #8 locks
   exact text + leading newline. If the contract is broader, #8 prunes.

6. **Knight A convergence.** I have not seen Knight A's tests. If
   Knight A also produces parametrized BON-599 shapes with overlapping
   inputs, Sage should pick one parametrize set (mine: 4 shapes) and
   merge non-overlapping cases.
