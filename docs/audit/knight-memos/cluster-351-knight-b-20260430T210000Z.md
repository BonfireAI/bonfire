## 3 — Why each test is RED today

1. **`test_pipeline_envelope_model_is_empty_string_sentinel`** — the
   pipeline at `src/bonfire/engine/pipeline.py:456` still writes
   `spec.model_override or self._config.model` into the envelope. With
   `PipelineConfig(model="pipeline-config-default")` and no
   `spec.model_override`, the test expects `envelope.model == ""` but the
   current code produces `envelope.model == "pipeline-config-default"`.
2. **`test_pipeline_uses_resolve_dispatch_model_helper`** — the pipeline
   has no `resolve_dispatch_model` symbol imported at
   `bonfire.engine.pipeline`, so `monkeypatch.setattr(...,
   raising=False)` silently no-ops; the dispatch site uses the inline
   3-tier `or` chain, so `captured_kwargs` stays empty.
3. **`test_executor_uses_resolve_dispatch_model_helper`** — same shape:
   no `resolve_dispatch_model` symbol at `bonfire.engine.executor`; the
   inline `or` chain at `executor.py:271-282` runs unmocked.
4. **`test_executor_uses_load_settings_or_default_factory`** — there is
   no `bonfire.engine.factory` module today (verified via `ls
   src/bonfire/engine/`); the executor's `settings=None` branch at
   `executor.py:92` calls `_BonfireSettings()` directly. The patch
   target doesn't resolve, the mock isn't invoked, `call_count["n"]`
   stays 0, and the assertion fails.

After the Warrior phase lands the two new modules + the four call-site
edits enumerated in Sage memo §F.2 + §G.2, all four tests flip GREEN.

   `test_pipeline_uses_resolve_dispatch_model_helper` patches
   `bonfire.engine.pipeline.resolve_dispatch_model` (the import the
   pipeline performs at module scope). If the Warrior chooses to import
   `resolve_dispatch_model` lazily inside the dispatch path instead of at
   module scope, the patch target must change. Sage memo §F.2 item 2
   says "Add import" without specifying scope; Knight B assumed
   module-scope import per the existing `from bonfire.agent.tiers import
   resolve_model_for_role` precedent at `executor.py:26` and
   `pipeline.py:` (similar). If Sage prefers lazy import, flip the patch
   targets to the lazy site and re-roll.

2. **Factory patch target stability.** The factory test patches
   `bonfire.engine.factory.load_settings_or_default` (the canonical
   import path). The executor must import via `from bonfire.engine.factory
   import load_settings_or_default` AND call the bound name at
   construction — not inline-import inside `__init__` from a different
   alias — for the patch to apply. Sage memo §G.2 says "swap" without
   specifying; Knight B assumed the canonical import-then-call pattern.

3. **`xfail(strict=True)` vs `xfail` permissive.** Knight B used `strict`
   so the suite fails loud if these tests accidentally pass before the
   Warrior phase (e.g. if a partial helper module already exists in some
   other branch). If Sage / Wizard prefer non-strict (lets tests flip
   green silently when ready), drop `strict=True`. Knight B's bias is
   strict — TDD discipline says the contract is exact.
