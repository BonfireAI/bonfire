## Innovation-lens choices

These are the deliberate departures from the simplest-possible
conservative implementation. Each choice is here to give Sage code-synth
something to compare.

1. **`Final` constants for sentinels.**
   - `model_resolver._EMPTY_MODEL_SENTINEL: Final[str] = ""` — names the
     empty-string return so future readers see the architectural intent
     ("all three precedence layers were empty") rather than a bare
     `""` literal that could be mistaken for an oversight.
   - `factory._WARNING_MESSAGE_TEMPLATE: Final[str] = "..."` — locks
     the `"Failed to load BonfireSettings"` substring that the test
     contract asserts on (Sage §H.2.2 / §H.2.3). If a future maintainer
     softens the wording, the `Final` annotation is the docstring +
     type-checker hint that this string is a stable contract.

2. **Module-level docstrings explaining the architectural seam.** Both
   new modules open with a multi-paragraph docstring that documents
   *why the seam exists*, not just *what it does*. The factory
   docstring spells out the three problems the seam fixes (silent disk
   reads in a constructor, silent `ValidationError` propagation,
   three-site copy-paste drift); the resolver docstring spells out
   the public-vs-internal contract that motivated Decision D. This
   feeds the eventual `docs/architecture.md` updates and helps any
   external Apache-2.0 contributor read the seam without having to
   chase the Sage memo.

3. **Defensive guards log type AND message.** The factory's
   `except Exception as exc` block logs
   `type(exc).__name__` AND `exc` in the same warning record:
   ```
   "Failed to load BonfireSettings from bonfire.toml/env (%s: %s);
   falling back..."
   ```
   The type alone is too cryptic for an end-user diagnosing a malformed
   TOML; the message alone hides whether the failure was a
   `TOMLDecodeError`, a `ValidationError`, or an `OSError`. Both
   together are the actionable signal. (BON-519 / BON-513 type
   discipline precedent.)

4. **Provenance comments at every call site.** Each replaced inline
   `or` chain carries a comment of the form
   `# Per cluster-351 Sage memo §F.<n>` explaining what was replaced
   and why. The pipeline's deleted dead-write site also carries an
   explanation comment citing Decision D + the cost-telemetry path.
   Future readers (or the v0.2 Sage trying to promote `settings=` to
   required) get the architectural context without leaving the file.

5. **Helper kept inline (no `_explicit_or_empty` sub-function).** The
   resolver body is short enough (10 lines) that extracting a
   sub-function would add cognitive cost without gain. The
   provenance comments inside the helper (`# Per cluster-351 Sage
   memo §C.1 layer N`) carry the same documentary load a sub-function
   name would.

6. **Public-tree ADR-001 module names.** Both new files use the
   `bonfire/engine/` directory and singular module names
   (`model_resolver.py`, `factory.py`) per ADR-001's module-rename
   discipline. No `vault/`, `costs/`, `workflows/`, `cartographer/`,
   `front_door/`, or `scanners/` private-tree leaks.

## Open contract questions for Sage code-synth

1. **Old `TestModelResolution` tests vs Axis 1 helper.** Knight B's
   cherry-picked test files preserve the prior
   `TestModelResolution` classes that patch
   `bonfire.engine.{pipeline,executor}.resolve_model_for_role`. After
   Axis 1, the dispatch path no longer reads through that local
   binding — it reads through `bonfire.agent.tiers.resolve_model_for_role`
   inside the helper. Two of the three preserved tests still pass
   (override wins, role passed through helper), but
   `test_pipeline_config_model_wins_when_resolver_empty` and
   `test_executor_config_model_wins_when_resolver_empty` will fail
   because the patch no longer reaches the resolver invocation.
   Sage code-synth should decide whether to:
   - (a) Update those tests to patch
     `bonfire.engine.model_resolver.resolve_model_for_role` instead,
     OR
   - (b) Have the helper re-read the local binding from its caller's
     module (gnarly), OR
   - (c) Delete those tests as obsolete after Axis 1.
   - (d) Accept the test breakage as known-and-expected and ratify in a
     follow-up retrospective wave.
   Warrior B's preserved `# noqa: F401` imports are a hedge toward (a) —
   the patch target still resolves to a valid binding, so a one-line
   test edit (`bonfire.engine.pipeline.resolve_model_for_role` ->
   `bonfire.engine.model_resolver.resolve_model_for_role`) revives them.

2. **Engine `__init__.py` re-exports.** Sage memo §C.1 says
   "Re-export from `bonfire.engine.__init__` is **NOT** required for
   v0.1." Warrior B respected that. Both helpers are reachable via
   `from bonfire.engine.model_resolver import resolve_dispatch_model`
   and `from bonfire.engine.factory import load_settings_or_default`,
   matching the patch targets in the new tests. Sage code-synth might
   want to add re-exports if a v0.2 promotion is imminent, but that's a
   separate ticket.

3. **`Final` annotations vs runtime-readability.** Warrior B used
   `Final[str]` on the two sentinel constants. They have no runtime
   effect (PEP 591) but improve type-checker friendliness and signal
   "these strings are part of the contract." If Sage code-synth prefers
   the conservative-warrior version's plain string literals, drop the
   `Final` import. No behavioral difference.

4. **Wide `except Exception` in factory.** Sage §C.2 lists the caught
   types as `(ValidationError, TOMLDecodeError, OSError)` but the
   sample code uses `except Exception as exc: # noqa: BLE001`. Knight A
   §H.2.4 parametrizes the never-raise contract with `RuntimeError`,
   `ValueError`, `OSError` — none of which are in the §C.2 narrow tuple.
   Warrior B chose the wide `Exception` catch (matches Knight A test
   contract). The factory docstring documents both the caught families
   AND the defensive `Exception` rationale. If Sage code-synth wants
   the narrow tuple, the test for `RuntimeError` would have to be
   removed.
