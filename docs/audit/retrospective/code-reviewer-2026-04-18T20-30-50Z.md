## Findings

### CR-01 (Important) — CheckpointManager._load_all not corrupt-file tolerant

**File:** `src/bonfire/engine/checkpoint.py:150-158`

`_load_all` iterates `self._dir.glob('*.json')` and calls `json.loads(path.read_text())` without try/except. A single corrupt file in the checkpoint directory will raise `JSONDecodeError` (or `ValidationError`), propagating out of `latest()` and `list_checkpoints()` even when valid checkpoints exist alongside it.

**Test gap:** `TestCorruptFileHandling::test_corrupt_file_raises_on_direct_load` only asserts direct `load('bad')` raises. No test verifies `latest()` / `list_checkpoints()` tolerate a corrupt file alongside valid ones.

**Contract friction:** The review brief's checkpoint discipline item 4(e) explicitly requires "Corrupt-file tolerance: handled with ErrorDetail, not exception." `load()` raising is a reasonable direct-API contract. But `latest()` / `list_checkpoints()` are listing operations — one corrupt file poisoning the directory scan is a resilience regression. Users cannot resume ANY session after a single crashed save pollutes the dir.

**Recommendation:** Follow-up ticket. Wrap per-file read in `_load_all` with try/except (JSONDecodeError, ValidationError, OSError) + `logger.warning` + skip. Add test: 2 valid + 1 corrupt → `latest()` returns the most recent VALID one. Not a merge blocker — save() atomicity and direct-load behavior are correct.

### CR-02 (Important) — Defensive last-resort Envelope construction relies on outer try/except

**Files:** `src/bonfire/engine/pipeline.py:522-525`, `src/bonfire/engine/executor.py:280-285`

`Envelope(task=spec.name, ...)` used as fallback when `last_envelope is None` can raise `ValidationError` if `spec.name` violates Envelope field constraints (e.g., empty string). `WorkflowPlan` validators reject empty names today, so this is latent.

**Mitigation present:** `PipelineEngine.run`'s outer try/except (line 123-132) catches all exceptions and returns a failed `PipelineResult`. Never-raises shell is intact.

**Recommendation:** Use `task=spec.name or "<unnamed>"` or pre-validate. Removes reliance on a single outer catch for inner defensiveness; cheaper to maintain.

### CR-03 (Suggestion) — Pipeline._execute_stage duplicates StageExecutor logic

**File:** `src/bonfire/engine/pipeline.py:400-537`

PipelineEngine keeps its own inlined stage execution (~140 LOC) identical in semantics to `StageExecutor._execute_single_inner`. The `StageExecutor` import at line 32 is annotated `# noqa: F401 -- public re-export for patch/discover`.

The duplication is intentional for this ticket (minimal V1 transfer, separate patch seams in tests) but will drift. Follow-up refactor: `PipelineEngine` composes `StageExecutor`.

### CR-04 (Suggestion) — Budget check is group-boundary, not mid-group

**File:** `src/bonfire/engine/pipeline.py:212-230`

Budget check runs after each group completes. A runaway parallel group of N stages can overshoot by sum of concurrent costs before detection. V1-parity; cancellation-aware watchdog is non-trivial under TaskGroup semantics. Acceptable at v0.1 scope; document as known limitation.

### CR-05 (Suggestion) — GateChain.__init__ typed as untyped `list`

**File:** `src/bonfire/engine/gates.py:129`

`def __init__(self, gates: list) -> None:` — use `list[QualityGate]` under `TYPE_CHECKING`. Trivial type-hygiene cleanup.
