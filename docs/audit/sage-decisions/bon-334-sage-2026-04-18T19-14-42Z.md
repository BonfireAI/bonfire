## Adjudications

### D1 — Shim pattern: per-test lazy imports (Knight-A) WINS

**Decision:** Canonical suite uses **per-test lazy imports** inside each test
body — `from bonfire.engine.X import Y` sits inside the test function. No
module-level `pytest.fail` autouse fixture.

**Rationale:** With Knight-B's module-scope `pytest.fail(pytrace=False)` at
import time, the Warrior's `pytest` output collapses the file into a single
collection ERROR, hiding per-test RED counts. With per-test lazy imports,
each test RED is reported individually — the Warrior sees "143 RED → 120 RED
→ 0 RED" as impl lands, which is the point of TDD-driven GREEN.

**Verification:** Full canonical run prior to impl reports `323 failed`
(granular) rather than `7 errors during collection`. Confirmed.

### D2 — `bonfire.git.path_guard` absence: DROP PATH-GUARD CONTRACT

**Decision:** Option (b) — **drop** the path-validation contract from public
`ContextBuilder`. No canonical test asserts `PathGuard.contains_absolute_paths`
behavior. Follow-up ticket can re-introduce when `bonfire.git` lands.

**Rationale:** Public v0.1 has no `bonfire.git` module (`src/bonfire/git/` is
empty). Porting PathGuard inside an engine-transfer ticket is scope creep.
A trivial stub (option c) leaves a behavioral gap between public v0.1 and
V1 that tests cannot lock, so it adds noise without value.

**Test asserting it:** None by design. The canonical `test_engine_context.py`
does not call `build()` with paths that would trigger the V1 guard.

### D3 — Compiler kwarg: DROP ENTIRELY

**Decision:** Option (a) — `PipelineEngine.__init__` and
`StageExecutor.__init__` do NOT accept a `compiler` kwarg in v0.1. Tests
explicitly assert `TypeError` when it is passed.

**Rationale:** Matches spec "compiler interface dropped" in the brief. V1's
uses of `compiler` all relate to role-prompt composition and role-tool
lookup (`compose_task_prompt`, `get_role_tools`) — features that are
out-of-scope for v0.1. A future wave can re-introduce with a dedicated
design ticket rather than a silent kwarg on the engine constructor.

**Tests asserting it:**
- `test_engine_executor.py::TestConstructor::test_rejects_compiler_kwarg`
- `test_engine_pipeline.py::TestPipelineEngineConstructor::test_rejects_compiler_kwarg`

### D4 — VaultAdvisor shape: PROTOCOL-ONLY, OPTIONAL IN EXECUTOR

**Decision:** Option (a) — ship `VaultAdvisor` against the public
`bonfire.protocols.VaultBackend` Protocol; any structural backend works. v0.1
does NOT wire a default concrete backend. `StageExecutor.vault_advisor` is
optional and defaults to `None`. When `None`, no pre-dispatch vault check
is performed.

**Rationale:** Public v0.1 has no concrete VaultBackend. A stub
`NullVaultBackend` (option b) would ship dead code. Skipping the advisor
entirely (option c) would drop a feature V1 exercises. Shipping the Protocol
bridge + opt-in wiring matches V1 semantics while keeping v0.1 self-contained.

**Test asserting it:**
- `test_engine_advisor.py` — every test uses a structural `_StubVaultBackend`
  class, not a concrete import.
- `test_engine_executor.py::TestConstructor::test_accepts_vault_advisor_kwarg`
  locks optional wiring.

### D5 — Gate exception safety: PIPELINE outer-try CATCHES; GateChain DOES NOT WRAP

**Decision:** Option (a) — `GateChain.evaluate_all` does NOT wrap individual
`gate.evaluate` calls in try/except. A raising gate propagates out of
`evaluate_all`. The `PipelineEngine.run()` outer try/except (V1 pipeline.py
lines 121-130) catches it and returns `PipelineResult(success=False)`. Gate
exception safety is the Pipeline's responsibility, not GateChain's.

**Rationale:** Mirrors V1 semantics exactly. Option (b) would be stricter,
but changes behavior and adds surface area that v0.1 does not need.

**Tests asserting it:**
- `test_engine_pipeline.py::TestNeverRaise::test_gate_exception_does_not_crash_pipeline`
  — explicitly locks the outer-try invariant (an exploding gate yields a
  `PipelineResult`, never propagates).

### D6 — Budget clamp at zero: LOCKED

**Decision:** YES. `budget_remaining_usd = max(0, plan.budget_usd - total_cost)`
when passed to `ContextBuilder.build()`. Never negative.

**Test asserting it:**

### D7 — Bounce recovery: SINGLE BOUNCE, NO RECURSION

**Decision:** Locked as canonical. On-gate-failure runs the target stage,
re-runs the original stage, re-evaluates gates ONCE, then halts regardless

**Tests asserting it:**

### D11 — NEW DISCOVERY: `PipelineConfig.dispatch_timeout_seconds` DOES NOT EXIST IN V0.1

**Decision:** The Warrior MUST NOT read `self._config.dispatch_timeout_seconds`
in the backend-dispatch path. Pass `timeout_seconds=None` (or omit the kwarg)
to `execute_with_retry`. The backend dispatch does not enforce per-attempt
timeout in v0.1 — the executor's `max_iterations` loop is the only retry
mechanism at the engine layer; the dispatch runner's own retry/backoff is
out-of-band.

**Rationale:** V1's `PipelineConfig` has `dispatch_timeout_seconds: float = 300.0`;
public v0.1's `PipelineConfig` has only `tier`, `model`, `max_turns`,
`max_budget_usd`, `persona`. Adding the field mid-engine-transfer is scope
creep. Follow-up ticket can re-introduce if v0.1 users request it.

**Tests asserting it:** No test locks the exact `timeout_seconds=` value passed
to `execute_with_retry`. `test_engine_executor.py::TestBackendDispatch` locks
that `execute_with_retry` is CALLED with a `DispatchOptions` and that
`event_bus=bus` is wired, but does NOT assert any particular timeout value.

### D12 — NEW DISCOVERY: Empty-stages STANDARD plan is REJECTED BY PUBLIC WorkflowPlan

**Decision:** Knight-B's `test_run_with_empty_plan_returns_result` was excluded
from the canonical suite. Public `WorkflowPlan._validate_dag` raises
`ValueError("stages must not be empty")` for `WorkflowType.STANDARD` with
empty stages, so the test cannot exercise the engine's "empty plan" path —
it fails at plan construction, not at engine execution.

**Follow-up note:** If the Warrior wants to test "engine called with an empty
plan," they must use `WorkflowType.SINGLE` (which allows empty stages per
V1 `plan.py` lines 87-89). But this is NOT a contract the canonical suite
enforces.

### D13 — `max_iterations` semantics: EXACTLY N ATTEMPTS

**Decision:** On persistent failure, the executor calls the backend EXACTLY
`spec.max_iterations` times. Knight-A asserted "at least N"; Knight-B asserted
"exactly N". V1 executor uses `for _iteration in range(spec.max_iterations):`
which is exactly N iterations. Lock Knight-B's stricter semantic.

**Test asserting it:** `test_engine_executor.py::TestIteration::test_persistent_failure_exhausts_exact_iterations`.

## Deviation 7 — handler exception hardening (positive)

During W3.1 Warrior implementation, exception hardening was added in two sites
over V1's bare-invocation pattern:

- `src/bonfire/engine/pipeline.py` lines 483-485 — wraps `handler.handle()` in
  `try / except Exception`.
- `src/bonfire/engine/executor.py` lines 105-113 — wraps `_execute_single_inner`
  in `try / except Exception`.

This is recorded as a positive deviation. V1 leaves handler raises uncaught at
the `_execute_stage` level, relying on the engine's outer try/except. The
transferred public engine adds an inner try/except at each handler call site,
converting raises into FAILED envelopes inline. This hardens against handler
bugs that V1 silent-fails, and is recorded here per BON-486 §BON-334 W-02 INFO
finding.
