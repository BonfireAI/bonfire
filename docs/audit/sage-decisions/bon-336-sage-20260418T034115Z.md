## Divergences resolved

### 1. Duplicate import tests (merge)

**Lens-A:** split imports across `TestPackageSurface` (5 tests ‚Äî `test_target_is_plural_workflows`, `test_singular_stub_is_not_the_target`, `test_all_exports_present`, `test_dunder_all_is_sorted_and_complete`, `test_all_public_factories_are_callable`).
**Lens-B:** flat `TestImports` class (7 tests ‚Äî one per name).

**Decision:** Merge into single `TestPackageSurface` class (12 tests). Keep Lens-A's 5 surface tests (plural target, singular stub disavowal, dir listing, `__all__`, callability) AND Lens-B's 7 individual import tests. Rationale: the 7 import tests isolate which specific name is missing when the transfer lands partially ‚Äî diagnostic value at low cost.

### 2. Parametrized vs combined forbidden-imports test

**Lens-A:** 1 combined test asserting all 5 forbidden prefixes in a nested loop.
**Lens-B:** `@pytest.mark.parametrize` with 5 entries (one test per forbidden prefix).

**Decision:** Use Lens-B's parametrized form. Rationale: test output names which specific prefix was violated (e.g., `test_workflows_does_not_import_forbidden[bonfire.events]`), accelerating diagnosis.

### 3. `TestFactoryInvariants` (Lens-A exclusive)

Lens-B has no equivalent of the cross-cutting invariants (frozen, unique names, no self-bounce, no dangling deps, workflow_type matches enum, StageSpec instances, factory name == plan name, fresh instance per call).

**Decision:** Keep 100% of Lens-A's `TestFactoryInvariants` (12 tests). Rationale: these enforce the DAG-validation guarantee from BON-331 across every factory. Without them, a factory could drift from plan-level invariants and the suite would miss it.

### 4. `TestParallelScoutTopology` (Lens-A exclusive)

Lens-A's parametrized topology tests assert that dual/triple/spike all share one parallel group, the sage fans in on every scout, and the sage itself is NOT in a parallel group.

**Decision:** Keep 100% of Lens-A's 9 tests. Rationale: composition invariants that ride across all three research factories ‚Äî worth locking explicitly, especially `test_sage_is_not_in_parallel_group` (a subtle V1 invariant).

### 5. Default-registry isolation (Lens-A exclusive)

Lens-A added `test_fresh_instance_per_call`, `test_mutation_does_not_leak`, `test_default_factories_round_trip_through_registry`.

**Decision:** Keep all three. Rationale: V1 builds a fresh registry per call (`registry = WorkflowRegistry()` inside the factory). Mutation-leak would be a hidden-global bug; these tests lock the V1 behavior.

### 6. `test_project_strategist_not_registered` (new, Sage addition)

Neither Knight added an explicit **negative** assertion that `project_strategist` is absent from the default registry. V1 registers it; public v0.1 must not.

**Decision:** Add `TestGetDefaultRegistry.test_project_strategist_not_registered` and tighten `test_registered_names_exact_set` to assert the exact five-name set. Rationale: a Warrior copy-pasting V1 `registry.py` without editing would accidentally register the sixth factory; this test catches that regression.

### 7. `TestStandardBuild` detail density

**Lens-A:** 7 tests (shape-focused: stage count, order, warrior retries, warrior no-self-bounce, prover/wizard bounces, no parallel groups).
**Lens-B:** 27 tests (per-stage role/gate/handler/dependency assertions).

**Decision:** Use Lens-B as the base (27 tests ‚Äî exhaustive per-stage assertions) and add Lens-A's `test_no_parallel_groups` + `test_dag_validation_passes` (30 tests total). Rationale: when the reference pipeline breaks, knowing *which stage and which attribute* shaves minutes off triage.

### 8. `TestDebug.test_warrior_depends_on_scout` (Lens-A exclusive)

Lens-B checked `no_bounce_back` and `no_gates` but not that warrior's `depends_on` names the scout.

**Decision:** Include. Rationale: a debug plan where warrior has no dependencies would be a structurally valid DAG but not the V1 contract (`_stage("warrior", "warrior", depends_on=["scout"])` in V1 `standard.py:106`). Explicit assertion required.

### 9. Fixture scoping: class fixture vs module helper

**Lens-A:** fixture-free; uses `_call(factory_name)` helper + per-class `@pytest.fixture() def plan`.
**Lens-B:** per-class `@pytest.fixture() def plan` but calls factory imports directly.

**Decision:** Canonical uses Lens-A's `_call(factory_name)` helper everywhere (routes through the package surface, so a Warrior mistakenly re-exporting from a non-public submodule is caught). Keep per-class `plan` fixtures where per-stage lookups are heavy (standard_build, debug, research factories). Rationale: routing through `bonfire.workflows` rather than `bonfire.workflows.standard` enforces the public surface.

### 10. Forbidden-imports scope

**Lens-A:** 7 prefixes (added `bonfire.agent`, `bonfire.session` on top of the shared 5).
**Lens-B:** 5 prefixes ‚Äî matching V1 test.

**Decision:** Use Lens-B's 5 (matches V1 exactly). Rationale: `bonfire.agent` and `bonfire.session` are not part of the V1 C9 constraint. Adding them would test behavior not in the V1 contract. If a future wave locks those down, a follow-up ticket can tighten this parametrize list.
