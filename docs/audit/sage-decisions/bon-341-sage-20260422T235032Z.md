## D3 — Design Decisions Locked

### D3.1 — Default `vault_backend`: **`"memory"`**

`get_vault_backend(backend="memory", ...)` in public v0.1 `knowledge/__init__.py`.
- Rationale: `pip install bonfire-ai` (no extras) must not raise `ImportError` on factory call. `[knowledge]` extra is opt-in.
- **Evidence:** `pyproject.toml:36` confirms `knowledge = ["lancedb>=0.15", "ollama>=0.4"]` extra exists (verified Phase 2d).
- Diff from v1: v1 `vault/__init__.py:18` hard-codes `backend="lancedb"`. Public flips to `"memory"`.

### D3.2 — Default `embedding_provider`: **`"mock"`** (via the factory argument default)

`get_vault_backend(embedding_provider="mock", ...)`.
- Rationale: no Ollama server assumed on first install. `MockEmbedder` is zero-dep and deterministic. Matches `get_embedder(provider="mock")` default at v1 `vault/embeddings.py:23` (kept verbatim).
- Diff from v1: v1 `vault/__init__.py:20` defaults `embedding_provider="ollama"`. Public flips to `"mock"`.

### D3.3 — Factory function name: **KEEP `get_vault_backend`** (do NOT rename to `get_knowledge_backend`)

Machinist Option 2. Artisan-endorsed via "minimal diff." Reasoning:
- Zero public call sites today (`grep -rn "get_vault_backend" src/ tests/` → 0 hits).
- On-disk LanceDB table identifier is `"vault"` (v1 `vault/backend.py:~33`); keeping the factory name consistent with the persistent identifier avoids cross-layer drift.
- `VaultBackend` protocol + `VaultEntry` type already keep `"Vault"` in their names (see `bonfire/protocols.py:78, 135`) — the public API surface already speaks "vault" for the concept. `knowledge/` is the module location; `VaultBackend` is the contract.
- Future rename to `get_knowledge_backend` is a 1-liner if desired; deferring that decision costs nothing.

### D3.4 — Class rename `TechFingerprinter → TechScanner` + file rename `fingerprinter.py → tech_scanner.py`

Both locked per ADR-001 line 45 (directory rename row) + ADR-001 line 61 (class rename row).
- File rename: keeps file stem matching class name ("Matches module name" rationale from ADR-001).
- Class rename: Linear spec verbatim names "TechScanner" not "TechFingerprinter"; the spec IS the rename.

### D3.5 — Three scanner concepts stay distinct

**DO NOT merge.** They are semantically different:
- `bonfire.knowledge.scanner.ProjectScanner` — two-pass file-tree + AST scanner, feeds chunker → VaultEntry. Lives in `knowledge/` because it is a vault-ingestion dependency.
- `bonfire.scan.tech_scanner.TechScanner` — project dependency/language detection (pyproject.toml, package.json, etc.) → `entry_type="tech_fingerprint"` VaultEntry records.
- `bonfire.scan.decision_recorder.DecisionRecorder` — markdown-ADR extractor → `entry_type="decision_record"` VaultEntry records.
- A future `onboard/scanners/` (v1 `front_door/scanners/project_structure.py` pattern) is NOT in BON-341 scope; streaming wrappers come with a later wave.
