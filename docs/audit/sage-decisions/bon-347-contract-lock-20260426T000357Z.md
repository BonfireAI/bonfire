### Innovation 4 — `test_v1_blob_rejection_message_format` (test_analysis_schema.py)

**Knight B claim:** Parametrized over `[0, 1, 99]`: each `_v1_blob(version=...)` raises `ValidationError` AND `str(wrong_version) in str(excinfo.value)` — to drift-guard the validator's f-string `{v}` placeholder.
**Sage memo citation:** §D4 ProjectAnalysis lock — validator body locked verbatim including `raise ValueError(f"study_schema_version must be 2, got {v}")`.
**v1 source citation:** `bonfire/src/bonfire/project/cartographer/models.py:212-220` — `_require_v2_schema` validator body; line 219 raises `ValueError(f"study_schema_version must be 2, got {v}")`.
**Floor overlap?** Floor `test_v1_cache_blob_raises_validation_error_from_bytes` covers a single fixed `version=1` case AND only checks ValidationError raises. Innovation sweeps three values AND checks message content. Strict superset on rejection sweep; orthogonal on message check.

**Verdict:** MERGE

**Reasoning (adversarial):** The PARAMETRIZED REJECTION SWEEP is genuinely additive over floor — three wrong values (0, 1, 99) all reject. ADOPT that part.

But the assertion `str(wrong_version) in str(excinfo.value)` is **too loose to actually pin the `{v}` f-string placeholder** that Knight B's docstring claims it drift-guards. Pydantic v2's `ValidationError.__str__` ambiently embeds `[type=value_error, input_value=<v>, input_type=int]` regardless of the validator's own message format. So a Warrior who refactored line 219 to a static `raise ValueError("study_schema_version must be 2")` (dropping `{v}`) would STILL produce a ValidationError whose `str(...)` contains `"0"`, `"1"`, or `"99"` from the ambient `input_value=<v>` tail. The substring digit check would pass spuriously — wrong-reason GREEN.

**Tightening (the MERGE):** Replace `assert str(wrong_version) in str(excinfo.value)` with `assert f"study_schema_version must be 2, got {wrong_version}" in str(excinfo.value)`. The exact-phrase substring match precisely pins the validator's locked f-string format. Dropping `{v}` would change the literal phrase to `"study_schema_version must be 2"` and the test fails with the locked phrase no longer present (the ambient `input_value=` tail does not reproduce the locked phrase). Reordering args, changing the verb, etc. all caught by the same assertion.

**Effect:** test name preserved; structure preserved; parametrize cells preserved (3); only the assertion mechanism tightened. Adopted as MERGE — innovation kept, assertion strengthened to actually do what its docstring claims.

## Adversarial findings

**ONE wrong-reason-GREEN risk caught in Innovation 4** (assertion-too-loose).

Knight B's `test_v1_blob_rejection_message_format` asserts `str(wrong_version) in str(excinfo.value)`. The docstring claims this drift-guards the f-string `{v}` placeholder. But Pydantic v2's ValidationError representation ambiently embeds `input_value=<v>` regardless of whether the validator's own message includes the value. Trace:

- v1 validator: `raise ValueError(f"study_schema_version must be 2, got {v}")` → ValidationError str contains both the locked phrase AND the ambient `input_value=` tail.
- Hypothetical drift: `raise ValueError("study_schema_version must be 2")` (dropping `{v}`) → ValidationError str contains only the static phrase BUT STILL contains the ambient `input_value=<wrong_version>` tail.

So `str(wrong_version) in str(excinfo.value)` passes for both, NOT discriminating the locked f-string from the drift. The exact-phrase match `f"study_schema_version must be 2, got {wrong_version}" in str(excinfo.value)` discriminates correctly: only the v1-verbatim format produces that phrase, the static-message drift produces `"study_schema_version must be 2"` without the `, got <v>` suffix.

**Resolution:** Adopted Innovation 4 with the assertion tightened. Test name, parametrize cells, structure, and intent unchanged. Documented in canonical file's docstring header (and inline in the new assertion expression).

This is the BON-347 equivalent of the BON-344 Innovation 1 row-4 catch — a Knight-B drift-guard whose CLAIM was sound but whose IMPLEMENTATION was insufficient to prove the claim. Caught at the contract-lock stage, exactly where the adoption-filter pattern is meant to catch it.

**Other innovations: clean.**
- Innovation 1: 5-tuple ordering verified against v1 models.py:200–210 verbatim.
- Innovation 2: gzip magic + roundtrip pattern verified against §D4 + v1 models.py:222–246; equality holds because all five fields are immutable Pydantic types.
- Innovation 3: `field.annotation is int` is identity-checked; Pydantic's `ModelField.annotation` for a plain `int = Field(default=2)` declaration is exactly `int` (not `Optional[int]`, not `Annotated[int, ...]`).

**Knight B's deferred 5th innovation (`test_schema_field_required` ValidationError on omission):** Knight B correctly skipped this — §D4 locks `study_schema_version: int = Field(default=2)`, so omitting the field is silently permitted (default fires). A test asserting omission raises ValidationError would force Warriors away from §D4. Good adversarial discipline by Knight B.
