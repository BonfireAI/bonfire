## Section 1 — Structural Decision (TYPE_CHECKING)

### The divergence

Knight A and Knight B disagreed on how `bonfire.onboard.protocol.ScanUpdate` should be imported in scanner test files. The shared `ScanUpdate` symbol is referenced in helper-function type annotations (`def _collect_events(emit: AsyncMock) -> list[ScanUpdate]:`) at module level.

| Pattern | Knight A | Knight B |
|---|---|---|
| Module-level import of `bonfire.onboard.protocol.ScanUpdate` | NO — kept in `if TYPE_CHECKING:` block | YES — eager, with `# noqa: TC001 - runtime-anchor for RED collection` |
| Module-level import of `bonfire.onboard.scanners.X` | NO — lazy inside test functions | NO — lazy inside test functions |
| Collection errors at RED | 5 (3 files collect, fail at runtime) | 8 (all files fail at collection) |
| Collected-but-deferred-RED tests | 54 | 0 |
| Source-of-truth memory | `feedback_conservative_wins_execution.md` | `feedback_tc003_breaks_pydantic_pytest.md` |

### Verdict — Knight B's pattern wins for the canonical

**Reasoning:**

1. **The TC001/TC003 anti-pattern memory was written for exactly this scenario.** `feedback_tc003_breaks_pydantic_pytest.md` says: "Never move type-only imports to TYPE_CHECKING in Pydantic model or pytest fixture files. Use `# noqa: TC003`." `ScanUpdate` is a Pydantic frozen model used in fixture-adjacent helper return-type annotations. Even with `from __future__ import annotations` deferring evaluation, `AsyncMock`'s `call_args_list` returns runtime objects, and helper functions iterating over `ScanUpdate` instances need the symbol at runtime when ruff's TC001 collapse path is taken. Knight B's `# noqa: TC001` comment explicitly opts out.
2. **Cleaner RED demonstration.** 8 collection errors all surfaced at the same stage means the Wizard preflight at GREEN time has a single delta to verify (8 collection errors → 0). Knight A's split (5 collection + 3 runtime ImportError) creates a half-collected state where some Wizard-stage failures look like Warrior bugs when they're really import-deferral artefacts.
3. **Verbatim port is bound to SOURCE files (Warrior phase), not test files.** Every test file is already non-verbatim because of the package rename `bonfire.front_door` → `bonfire.onboard`. The conservative-wins precedent applies cleanly to source porting (where v1 line numbers are line-by-line load-bearing). Test file headers are reorganised in v0.1 to match v0.1 idioms.
4. **`Path` stays in TYPE_CHECKING.** Knight B's pattern is principled, not blanket-flat — `from pathlib import Path` is genuinely type-only (no runtime use in helpers) and stays in `if TYPE_CHECKING:`. Only `ScanUpdate` (a runtime Pydantic class anchored in helper return types) is lifted. That nuance matters and is preserved verbatim from Knight B.
5. **Empirical confirmation.** `pytest --collect-only` against the canonical produced exactly 8 collection errors with `ModuleNotFoundError: No module named 'bonfire.onboard.X'` — clean RED, no syntax errors, no lint hits.

### What this means at GREEN time

When Warriors port the source files, all 8 collection errors collapse to 0. The 132 floor + 59 expanded innovation cells (= 191 total) all execute. Delta-based preflight (per Sage §D10 with adopted-innovation expansion) targets `passed ∈ [3126 + 191 − 8, 3126 + 191 + 8] = [3309, 3325]`.
