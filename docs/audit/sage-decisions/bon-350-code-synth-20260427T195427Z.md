## Phase 2 — Adversarial filter on the 8 B innovations

### Innovation 1 — `_normalize_role` private helper extraction

**B's claim:** Extract `strip().lower()` + `isinstance(role, str)` guard into a named helper for grep-locatability and isolated unit-testability.

**Sage memo cite:** §D7 lines 184-186 — A inlines this as `normalized = role.strip().lower() if isinstance(role, str) else ""`. Memo does NOT specify helper extraction; both shapes are compliant.

**Adversarial check:** A's inline form is exactly one line. B's helper is 17 LOC including docstring. Net cost: +16 LOC for zero observable behavior change. Test coverage: identical (A and B both pass `test_resolver.py::TestResolverNormalization` × 3 + `TestResolverFallback` × 3).

**Verdict: NOT load-bearing.** Default conservative.

### Innovation 2 — `_canonical_for` private helper extraction

**B's claim:** Extract canonical-vs-gamified resolution into a named helper for clarity.

**Sage memo cite:** §D7 resolution-order table (lines 428-438) specifies the algorithm, not its decomposition. Both A's inline `try/except + GAMIFIED_TO_GENERIC.get()` and B's helper-with-frozenset-probe pass the same fallback chain.

**Adversarial check:** A's inline shape is 6 LOC. B's helper is 18 LOC. Net cost: +12 LOC for zero observable behavior change. The helper is callable from no other site (sole caller is `resolve_model_for_role`).

**Verdict: NOT load-bearing.** Default conservative.

### Innovation 3 — `frozenset` membership probe vs `try/except`

**B's claim:** "Pythonic guidance prefers explicit membership probes for known-non-error cases" over exception-as-flow. `_CANONICAL_VALUES = frozenset(role.value for role in AgentRole)` snapshot at module load, then `if normalized in _CANONICAL_VALUES`.

**Sage memo cite:** §D7 step 2 says "Try AgentRole(normalized) — canonical generic name. On ValueError, try GAMIFIED_TO_GENERIC[normalized]." Memo specifies *order* and *outcome*, not the EAFP-vs-LBYL choice.

**Adversarial check:** Both shapes correctly implement the §D7 fallback chain. A's `try: AgentRole(normalized) / except ValueError:` is the canonical Python EAFP idiom for enum membership; B's frozenset snapshot adds a module-level cache to avoid the exception path. Performance: B is ~2× faster on the unknown-role path (avoid ValueError construction), ~equivalent on the known-role path. The resolver is called once per stage dispatch; absolute throughput is irrelevant. Footprint: B adds a module-level frozenset that duplicates the AgentRole enum keyset — a redundant data structure that must be kept in sync if AgentRole is ever extended.

**Verdict: NOT load-bearing.** Memo §D7 says "Try AgentRole(...)" — A's try/except is the more direct memo-faithful reading. B's optimization is real but marginal and adds a sync hazard. Default conservative.

### Innovation 4 — Named `_LAST_RESORT_DEFAULT_TIER` constant

**B's claim:** Centralize the BALANCED fallback into one named constant so a future fallback-policy change lands in one place.

**Sage memo cite:** §D7 step 4 says "fall back to ModelTier.BALANCED (no raise)." Memo references `ModelTier.BALANCED` directly. A inlines `ModelTier.BALANCED` at the two fallback sites.

**Adversarial check:** A inlines BALANCED at 2 sites; B references `_LAST_RESORT_DEFAULT_TIER` at 2 sites. Both are equivalent. The future-change argument is forward-looking but speculative — if/when the fallback policy changes, that's a NEW ticket with its own contract. Premature abstraction.

**Verdict: NOT load-bearing today.** D-FT eligible if a fallback-policy change ever comes (D-FT G.1 below).

**Sage memo cite:** §D7 specifies steps 4–5 in prose; both shapes implement them.

**Adversarial check:** A's two-line if/else is more readable for the same observable behavior. B's conditional expression is denser. Test coverage identical.

### Innovation 7 — `collections.abc.Mapping` vs `typing.Mapping`

## Recommended Sage memo §D-CL note (carry forward to BON-351)

**Pattern: BON-NNN-reference scrub during new-file ports.**

When Sage memo §D2 contains a sample module docstring with `BON-NNN` ticket references, and the v0.1 repo carries `tests/unit/test_no_bon_ref_in_src_sweep.py` (or equivalent) that bans `BON-NNN` substrings in any `.py` under `src/bonfire/`, Warriors will discover a forced contradiction during GREEN-phase. The canonical resolution is to PARAPHRASE the BON-NNN references in the actual docstring (e.g., "BON-351's dispatch integration" → "the dispatch engine integration" or "the dispatch integration layer"). Properties:

1. **Runtime semantics preserved** — docstring is descriptive only, never inspected.
2. **Source bytes never contain** the banned `BON-NNN` literal.
3. **Sweep test passes** because it scans rendered file text.
4. **Sage §D2 contract intent preserved** — the docstring still describes the module's role and forward-references the consuming subsystem.

**Sage should pre-emptively paraphrase BON-NNN references in §D2 docstring samples** of any future port memo where the v0.1 repo has the BON-NNN sweep test active.

**Precedent:** BON-350 (this ticket). v0.1 sweep test = `test_no_bon_ref_in_src_sweep`. Sage memo BON-350 §D2 lines 92, 97 contained `-- BON-350` and `BON-351's dispatch integration`. Resolution = paraphrase in actual `tiers.py` docstring (both Warriors converged independently).

This extends the BON-348 D-CL pattern note (rename-sweep guard during cli composition-root ports) to cover BON-NNN-reference sweep tests for new files. File as D-FT G.4.
