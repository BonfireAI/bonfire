### Q2 ‚Äî Classifier algorithm: structured front-matter, prose pattern-match, or hybrid?

**Options:**
- **(a) Sentinel front-matter block** ‚Äî Sage memos MUST include a YAML/TOML front-matter region with explicit `xfail_defers:` array of `{flag, condition_module, condition_attr}` records; classifier parses structurally.
- **(b) Pattern-match free-form prose** ‚Äî Sage memos use a canonical heading like `## DEFER via xfail` (per ticket text) and the classifier regex-scans for that heading + bullet patterns underneath.
- **(c) Hybrid** ‚Äî accept either; prefer front-matter, fall back to prose when absent. Classifier emits `parse_source: Literal["front_matter", "prose", "absent"]` for forensics.

**Sage proposal: (c) hybrid, prefer (a) for new memos.**

**Reasoning:**
1. The ticket text says "parses Sage decision log for 'DEFER via xfail' sections" (free-form prose). That's option (b). But option (b) alone is fragile: every Sage memo must hew to the same heading literal, and a typo silently masks the marker. The XML-vs-stdout fail-safe in BON-519 ¬ßD4 (`parse_pytest_stdout_fallback`) is the precedent ‚Äî structured first, prose second.
2. Existing Sage memos under `docs/audit/sage-decisions/` (35+ files) do NOT currently use a front-matter block; they're free-form Markdown with `## ¬ßA`, `## ¬ßD-CL.*` section headers. Forcing retrofit is OOS for v0.1.
3. Adding an OPTIONAL `<!-- bonfire:defers -->` HTML-comment block (or YAML front-matter delimited by `---`) at memo top, parsed when present, lets new memos opt into structured semantics without breaking existing memos. Old memos parsed via prose fallback.
4. Determinism requirement (ticket: "deterministic decision-log scanning only") rules out LLM-backed parsing. Both (a) and (b) are deterministic; (c) is the union.

**Anchor:** Ticket "Out of scope: LLM-backed classification". BON-519 ¬ßD4 fail-safe pattern (`parse_pytest_junit_xml` + `parse_pytest_stdout_fallback`).

### Q4 ‚Äî Correction-agent shape: full Sage subagent or restricted "xfail-edit-only" agent?

**Options:**
- **(a) Full Sage subagent re-dispatched** ‚Äî re-invoke the Sage role (synthesizer) with full read/write access; trust the agent to scope its edits.
- **(b) Restricted "xfail-corrector" agent** ‚Äî new narrow agent role with tools constrained to `Read`, `Edit` on a whitelisted file set, and `Write` denied. Cannot touch `src/`, `tests/<body>`, fixtures, assertions; can only edit `xfail` decorators / flag definitions / Sage memo decision-log.
- **(c) Pure-deterministic correction (no agent)** ‚Äî generate the corrected xfail markers programmatically from the classifier's parsed `DeferRecord` set + the actual failing test files. No LLM in the correction loop.

**Sage proposal: (b) restricted "xfail-corrector" agent, with tool-policy enforcement at dispatch site.**

**Reasoning:**
1. Option (c) is tempting (cheapest, fastest, deterministic) but fails the "under-specified" case: when a Sage memo says `xfail(condition=not _ANALYST_PRESENT)` and the actual failure is because BOTH `_ANALYST_PRESENT` AND `_VERIFIER_PRESENT` are missing, deterministic correction has no reliable way to MAP the failing test to the missing flag set without re-reading the Sage memo's reasoning prose. An LLM trained on the memo handles this; a regex-based corrector either over-corrects (adds every flag in the memo) or under-corrects.
2. Option (a) (full Sage) is too permissive ‚Äî Sage's tool surface includes Write; a runaway correction could rewrite test bodies or src/. The ticket explicitly forbids touching src/, test bodies, fixtures, assertions.
3. Option (b) ‚Äî narrow agent with hard-locked tools ‚Äî is the right shape. Tool policy enforcement happens at the dispatch site (per BON-118 lesson, see `feedback_full_pipeline_protocol.md`): the SageCorrectionBounce handler dispatches with `DispatchOptions(allowed_tools=("Read", "Edit"), denied_tools=("Write", "Bash"))` and a path allowlist (`docs/audit/sage-decisions/*.md` + a whitelisted set of test files containing `@pytest.mark.xfail` decorators).

**Anchor:** `src/bonfire/protocols.py` `DispatchOptions`; `src/bonfire/handlers/wizard.py:43` (review agent runs read-only ‚Äî same shape); BON-519 ¬ßD-CL.4 ("Pure-function classifier ... I/O lives in the handler shell") for separation of concerns; ticket "Edits ONLY xfail decorators / flag definitions / decision log; never src/, never test bodies/fixtures/assertions".

**Consequences:**
- New `AgentRole` member NOT introduced. Reuse `AgentRole.SYNTHESIZER` (already shipped, `roles.py:40`) ‚Äî the corrector IS a Sage doing surgical work, not a new role. Display layer: gamified "Sage" appears in CLI output for the correction stage (consistent with the role name).
- Dispatch site: `SageCorrectionBounceHandler.handle()` constructs `DispatchOptions(allowed_tools=frozenset({"Read", "Edit"}))`. Bash is implicitly denied (not in allowed). Write is implicitly denied.
- Path allowlist enforced via a pre-dispatch hook (or via the agent prompt ‚Äî Sage proposal: belt-and-suspenders, BOTH a prompt-level discipline rule AND a post-dispatch verification step that diffs the correction branch and rejects any file outside the allowlist).
- Post-dispatch verification: the handler reads the correction commit's `git diff --name-only HEAD~1 HEAD` from the correction worktree; if any path is outside `{docs/audit/sage-decisions/*.md, tests/**/conftest.py, tests/**/*.py}` AND that test file's diff hunks touch any line NOT a `@pytest.mark.xfail` decorator or flag-definition assignment, classifier returns `correction_failed`.
- Correction agent SMEAC discipline: SMEAC explicitly forbids touching test bodies/assertions/fixtures (mirrors BON-519 SMEAC discipline for boundary-respecting agents).

**Sub-decision Q4a ‚Äî naming the correction sub-stage:** the auto-dispatched agent is internally a "Sage correction sub-agent". Sage proposal: keep the agent role string as `"sage"` (gamified) backed by `AgentRole.SYNTHESIZER`; the SMEAC variant carries a `correction_mode=True` flag in the prompt. No new role string, no new role enum member. Anta to ratify.

### Q5 ‚Äî Loop semantics: max correction attempts before escalation?

**Options:**
- **(a) 1 attempt** ‚Äî single bounce. If the correction fails, escalate to Wizard immediately.
- **(b) 2 attempts** ‚Äî give the corrector one retry with the post-mortem of attempt 1.
- **(c) 3 attempts** (mirror Warrior `max_iterations=3`) ‚Äî match Warrior's TDD retry budget.
- **(d) Configurable** ‚Äî `BonfireSettings.pipeline.sage_correction_max_attempts: int = 1`.

**Sage proposal: (a) 1 attempt for v0.1; expose as configurable setting (default 1) for forward compatibility.**

**Reasoning:**
1. The under-specified-xfail symptom is a NARROW class of failure. The corrector either gets it right by reading the memo + failure list (high probability with a focused agent), or the memo lacks the information needed to specify correctly (in which case retrying with the same inputs won't help).
2. Warrior's `max_iterations=3` exists because Warrior is doing creative implementation work ‚Äî three swings at making a contract green. Sage-correction is purely surgical edits; it's closer to a code-formatter than a creative agent. One attempt is the right budget.
3. Token cost discipline: each correction attempt is a full Opus dispatch (Sage role). Two attempts double cost; three triple it. For a v0.1 keystone, 1-attempt + escalate-to-Wizard is the cheapest right answer.
4. Configurability: expose `sage_correction_max_attempts` on `BonfireSettings` to let power users tune. Default 1 keeps the v0.1 ship contract honest.
5. Counter (option b/c): if Anta wants more attempts later, the path is a setting bump, not a code change. Build the seam.

**Anchor:** `src/bonfire/workflows/standard.py:58` (`max_iterations=3` for Warrior ‚Äî the comparison case); `src/bonfire/models/config.py` (settings home); ticket "Pipeline stage SageCorrectionBounce ‚Äî auto-dispatches Sage-correction".

**Consequences:**
- `StageSpec` for `sage_correction_bounce` uses `max_iterations=1`. The handler internally does NOT loop; loop semantics are surfaced via the StageSpec contract so the engine handles retry orchestration uniformly.
- New setting: `BonfireSettings.pipeline.sage_correction_max_attempts: int = Field(default=1, ge=1, le=5)`.
- Handler reads the setting via constructor injection (not at handle-time); test fixtures override.
- On exhaustion of attempts: classifier returns `correction_failed`, handler returns FAILED envelope with `ErrorDetail(error_type="sage_correction_exhausted")`, gate `sage_correction_resolved` fails error, pipeline halts with escalation message naming the failing test set + the original Sage memo path.

**Sub-decision Q5a ‚Äî bounce-back vs in-stage retry:** Sage proposal: in-stage retry (handler loops up to `max_attempts`), NOT a workflow-level bounce edge. Reasoning: the engine's `on_gate_failure` semantic (cf. `standard.py:65, 79`) bounces to a previous stage and re-runs IT. Bouncing to "self" is an in-handler concern. ### Q6 ‚Äî Decision-log schema: canonical "DEFER via xfail" header or pattern-match free-form?

**Options:**
- **(a) Strict canonical header** ‚Äî Sage memos MUST contain a literal `## DEFER via xfail` H2 heading; classifier scans only for that heading.
- **(b) Tolerant prose pattern** ‚Äî accept variants: `## DEFER`, `### xfail defers`, `## DEFER via xfail`, `## ¬ßD-DEFER`, etc. Regex with case-insensitive + whitespace-tolerant matching.
- **(c) Front-matter only** ‚Äî see Q2 (c). Prose scanning is fallback only.

**Sage proposal: combination ‚Äî Q2's hybrid (front-matter preferred, prose fallback) PLUS a single canonical prose heading literal `## DEFER via xfail` (matching ticket text verbatim).**

**Reasoning:**
1. Per Q2, front-matter is the structured path. When prose is the source of truth, a single canonical literal heading prevents ambiguity. Anchor the literal to the ticket: `## DEFER via xfail` (case-insensitive match, no other variants accepted).
2. Reject pattern-tolerance (option b): every alternative spelling is a hidden-failure surface. If a Sage memo uses `## DEFER` instead of `## DEFER via xfail`, it should be flagged on Wizard review, not silently parsed.
3. The bullet schema under the heading is also locked: each defer record is a Markdown list item of the form:
   ```
   - flag: <FLAG_NAME>, module: <module.path>, attr: <Module.Attribute>
   ```
   The classifier regex matches `^-\s+flag:\s*(\S+),\s*module:\s*(\S+),\s*attr:\s*(\S+)\s*$` after a `## DEFER via xfail` heading.
4. The schema lock is documented in `docs/pipeline-stages.md` ¬ßSageCorrectionBounce (consumed by Sage memo authors going forward).

**Anchor:** Ticket text "Sage decision log for 'DEFER via xfail' sections"; `docs/audit/sage-decisions/bon-350-sage-20260427T182947Z.md` for the prevailing memo style (free-form Markdown, no front-matter); BON-519 ¬ßD7 for documentation precedent.

**Consequences:**
- Classifier rejects (with `parse_source="absent"`) any memo without either front-matter OR a `## DEFER via xfail` heading.
- The strict heading literal is a constant in the classifier module: `_DEFER_HEADING_RE = re.compile(r"^##\s+DEFER\s+via\s+xfail\s*$", re.IGNORECASE | re.MULTILINE)`.
- Bullet record regex: `_DEFER_RECORD_RE` with named groups `flag`, `module`, `attr`.
- Documentation: a "Sage memo defer schema" subsection in `docs/pipeline-stages.md` showing the canonical front-matter AND prose syntaxes, with a one-paragraph rationale citing this Q.
- Test surface: classifier unit tests cover (i) front-matter only, (ii) prose only, (iii) both (front-matter wins), (iv) canonical heading with malformed bullets (bullets ignored, parse_source="prose", records empty), (v) ALL absent (parse_source="absent"), (vi) wrong heading variant (e.g., `## DEFER`) ‚Äî treated as absent.

### Q7 ‚Äî Branch ownership: where do correction commits land?

**Options:**
- **(a) Original Sage branch** ‚Äî correction commit cherry-picked onto the same branch the Sage memo lives on (typically the warrior/feature branch).
- **(b) New `*-sage-correction` branch** ‚Äî corrections land on a new ephemeral branch; main feature branch remains pristine.
- **(c) Mixed: edit memo on Sage branch, edit test files on warrior branch** ‚Äî split the surgical edits by file domain.

**Sage proposal: (a) original Sage branch ‚Äî same branch the upstream pipeline is operating on.**

In v0.1 standard pipeline, the upstream stages (Knight, Warrior, Prover, Bard, Wizard) all operate on a single feature branch. The Sage memo lives in `docs/audit/sage-decisions/bon-XXX-sage-*.md` on that same branch. Correction commits MUST land on the same branch so Bard's PR captures both the original work AND the correction in one diff.

**Reasoning:**
1. The ticket text says "Correction agent SMEAC ‚Äî targets original Sage branch". This is option (a) literally.
2. The post-correction merge step in the ticket ("Post-correction merge + re-verify ‚Äî cherry-pick xfail fixes, re-run pytest") only makes sense if the correction commit is on the same branch as the Warrior commit. A separate branch would require a merge step BEFORE re-verify, which adds complexity and a new conflict surface.
3. v0.1 has NO push protection or branch protection rules on the agent-feature branches (`antawari/bon-XXX-*`). Pushing a correction commit is the same primitive as Warrior pushing a GREEN commit. No new git auth surface.
4. Pre-commit hooks (ruff, etc.) MUST run on correction commits same as on Warrior commits. Sage proposal: the correction agent runs `git commit` from the worktree without `--no-verify`. If a hook fails, the correction is treated as `correction_failed` and pipeline escalates.
5. Worktree discipline: the correction agent operates inside the SAME worktree as the upstream pipeline (no new worktree primitive needed). This avoids the BON-519 ScratchWorktreeFactory surface entirely ‚Äî BON-513's concern is a single-branch in-place edit, not a simulated-merge tip.

**Anchor:** Ticket "Correction agent SMEAC ‚Äî targets original Sage branch"; `feedback_pre_commit_hooks` discipline; `feedback_no_economic_cuts.md` (correction commits go through the same hooks as feature commits).

**Consequences:**
- Handler does NOT create a new worktree. Operates in the worktree associated with the upstream stage envelope (passed via `prior_results` or envelope metadata ‚Äî Sage B's ¬ßD2 specifies the extraction path).
- Correction commit message format: `bon-XXX: sage correction (xfail under-marking)` followed by a body listing the failing tests and the corrected `xfail` flags. No `Co-Authored-By` footer (correction is a Bonfire-internal mechanism, not human-attributed).
- Push step: `git push origin <branch>` ‚Äî same primitive as Warrior's push (Bard typically owns push, but Warrior pushes too). The correction handler invokes push only if commit succeeds AND classifier confirms re-verify is green.
- Pre-commit hook failures: handled via the `RuntimeError` exit-code path in the subprocess wrapper; `proc.kill()` + `await proc.wait()` on timeout (per BON-519 ¬ßD-CL.7 hazard #2).
- No `--no-verify`, no `--no-gpg-sign`, no `-c commit.gpgsign=false` (per project memory: skip-hooks discipline).

**Sub-decision Q7a ‚Äî re-verify scope:** after correction, re-run pytest on the SAME test file set the Warrior cycle ran (NOT full-suite ‚Äî that's BON-519 MergePreflight's job, downstream). ### Q9 ‚Äî Gate vs Stage: does the classifier produce a Gate or a Stage?

**Options:**
- **(a) Stage-only** ‚Äî `SageCorrectionBounceHandler` is a full StageHandler; its `handle()` runs classifier + correction + re-verify; no separate gate.
- **(b) Stage + Gate** ‚Äî handler writes classifier result to envelope metadata; a separate `SageCorrectionResolvedGate` reads metadata and produces gate verdict.
- **(c) Gate-only** ‚Äî classifier runs as part of a Gate; correction is a separate stage triggered by gate failure.

**Sage proposal: (b) Stage + Gate.**

**Reasoning:**
1. BON-519 ¬ßA Q1 + ¬ßD6 established the precedent: a deterministic verification mechanism produces BOTH a handler (the I/O orchestrator) AND a gate (the engine-readable verdict). `MergePreflightHandler` writes `META_PREFLIGHT_CLASSIFICATION` + `META_PREFLIGHT_TEST_DEBT_NOTED`; `MergePreflightGate` reads them.
2. Gates are the engine's pluggable decision primitive (`src/bonfire/engine/gates.py:33-103` ‚Äî six existing gate classes follow the same shape: read envelope, return GateResult). Reusing this pattern keeps the engine surface uniform.
3. Stage-only (option a) hides the verdict from the engine's gate-aware logic: pipeline-level escalation, severity reporting, GateChain composition (`gates.py:126-139`) all work on Gate verdicts. Burying the classification inside the handler forfeits these.
4. Gate-only (option c) inverts ownership awkwardly: gates are stateless evaluators, but the correction work is stateful (commit, push, re-verify). Stage owns the work; gate owns the verdict.

**Anchor:** `src/bonfire/engine/gates.py:33-103` (six existing gates); BON-519 ¬ßD6 ("File 3: `src/bonfire/engine/gates.py` ‚Äî new `MergePreflightGate`"); `src/bonfire/protocols.py:175-184` (QualityGate Protocol).

- New gate: `SageCorrectionResolvedGate` in `src/bonfire/engine/gates.py`.

## ¬ßD2 ‚Äî Classifier Algorithm + Verdicts

**Decision:** The classifier is **pure-deterministic, no LLM, no I/O at evaluate time**, mirroring BON-519 ¬ßD4's `classify_pytest_run` pure-function discipline (BON-519 ¬ßD4 lines 435-456). Out-of-scope per ticket: LLM-backed classification. The classifier reads three inputs (already-resident in memory) and returns a verdict by first-match-wins decision tree.

**Public types (`src/bonfire/verify/classifier.py`):**

```python
class ClassifierVerdict(StrEnum):
    SAGE_UNDER_MARKED = "sage_under_marked"
    WARRIOR_BUG = "warrior_bug"
    INDETERMINATE = "indeterminate"   # decision log absent, malformed, or no DEFER section


@dataclass(frozen=True)
class FailingTest:
    file_path: str            # repo-relative, e.g. "tests/unit/test_wizard_handler.py"
    classname: str
    name: str                 # may include parametrize id ("test_x[case-3]")
    message: str              # truncated failure message
    failure_kind: Literal["xfail_strict_unexpected_pass", "real_failure", "error", "collection_error"]


@dataclass(frozen=True)
class ClassificationResult:
    verdict: ClassifierVerdict
    candidate_test: FailingTest | None        # the single failing test that drove the verdict
    suspected_flag_name: str | None           # e.g. "_PARSER_HELPERS_PRESENT"
    suspected_missing_deps: tuple[str, ...]   # e.g. ("_EXTRA_META_PRESENT",) ‚Äî flags Sage forgot to AND-into the condition
    decision_log_entry: DecisionLogEntry | None
    rationale: str                            # short, machine-grepable; one of a fixed set of phrases
```

**Classifier signature:**

```python
class Classifier:
    """Deterministic Warrior-failure classifier.

    No I/O. No subprocess. No LLM. Construct with the three deterministic
    inputs and call `.classify()`.
    """

    def __init__(
        self,
        *,
        warrior_failures: tuple[FailingTest, ...],
        decision_log_entries: tuple[DecisionLogEntry, ...],
        flag_definitions: tuple[FlagDefinition, ...],
    ) -> None: ...

    def classify(self) -> ClassificationResult: ...
```

**Decision tree (FIRST MATCH WINS ‚Äî mirrors BON-519 ¬ßD4 line 447 stance):**

```
1. warrior_failures == ()                        -> INDETERMINATE  (rationale="no_failures")
2. decision_log_entries == ()                    -> INDETERMINATE  (rationale="decision_log_absent")
3. ANY failing test has failure_kind="collection_error"
                                                 -> WARRIOR_BUG    (rationale="collection_error")
4. Find candidate: the FIRST failing test whose file_path + test name
   appear in some entry e ‚àà decision_log_entries (matched by
   `e.target_test_module` exact-string match against `failing.file_path`,
   AND `e.target_test_name_substring` ‚äÜ `failing.name`).
   No candidate found -> WARRIOR_BUG (rationale="no_decision_log_match")
5. From candidate's matched entry e, read e.referenced_flags = set of
   `_X_PRESENT`-shaped names cited in the DEFER section's prose or in the
   xfail decorator's `condition=` expression.
   Read flag_definitions = parsed `_X_PRESENT = (...)` records from the
   test module's top-level (set up by parser, see ¬ßD5).
6. Compute deps_satisfied(flag) for each cited flag:
     - flag.dep_count == number of distinct module/attr lookups inside
       its initializer (e.g. `try: from m import X, Y` -> 2 deps).
     - For the candidate's referenced flags, if Sage's DEFER prose names
       MORE referenced symbols than `flag.dep_count`, mark the flag
       UNDER-SPECIFIED.
7. Any cited flag is UNDER-SPECIFIED -> SAGE_UNDER_MARKED
                                       (rationale="under_marked_flag",
                                        suspected_flag_name=flag.name,
                                        suspected_missing_deps=flag.missing_deps)
8. Otherwise (flag deps complete, but real failure) -> WARRIOR_BUG
                                       (rationale="flag_well_specified_but_failure_real")
```

**Key edge cases (Knight tests must cover ‚Äî mirrors BON-519 ¬ßD4 edge-cases table):**

| Scenario | Inputs | Expected verdict |
|---|---|---|
| 1 failure, decision-log entry covers test, single-flag condition citing only `_PARSER_HELPERS_PRESENT`, but DEFER prose names both `_parse_verdict` AND `_extra_meta` | flag.referenced=2, flag.dep_count=1 | SAGE_UNDER_MARKED |
| 1 failure, decision-log covers test, condition cites two flags AND-ed, both flags fully specified | flags well-formed | WARRIOR_BUG (rationale=flag_well_specified) |
| 1 failure, no decision-log entry mentions this test | decision_log non-empty, no match | WARRIOR_BUG (rationale=no_decision_log_match) |
| pytest collection error before any test runs | failures=(error,), kind=collection_error | WARRIOR_BUG |
| Decision log present but malformed (no parseable DEFER section) | decision_log_entries=() | INDETERMINATE |
| Multiple failures, only one matches a DEFER entry; others don't | mixed | First-match-wins on candidate; if first match yields SAGE_UNDER_MARKED return that. (Sub-case: ALL must match a DEFER for single-bounce. Mixed -> falls through to WARRIOR_BUG, rationale="multiple_failures_only_partial_match") |
| `pytest.mark.xfail(strict=True)` raised XPASS (the absorption-trap inverse) | failure_kind=xfail_strict_unexpected_pass | SAGE_UNDER_MARKED (rationale="xpass_under_strict") ‚Äî this is also Sage's bug, opposite direction |

**What the classifier deliberately DOES NOT do (out-of-scope, deterministic-only stance):**

- Does not call any LLM (per ticket "Out of scope: LLM-backed classification").
- Does not infer "Sage probably meant flag X" from test source ‚Äî only reads the parsed `decision_log_entries` and parsed `flag_definitions`. If Sage's memo doesn't name the missing dep, the rationale is `under_marked_flag` and `suspected_missing_deps` lists every dep referenced in the test body that the flag's initializer doesn't cover (mechanical AST diff, not semantic guess).
- Does not differentiate "near-miss" from "wildly wrong" ‚Äî any under-specification is SAGE_UNDER_MARKED.
- Does not generalize to non-xfail markers (skipif, skip) ‚Äî out-of-scope per ticket "Generalizing to other author/reviewer boundaries".

## ¬ßD5 ‚Äî Decision-Log Schema (Classifier Input Format)

**Decision:** The Sage decision-log files under `docs/audit/sage-decisions/bon-XXX-sage-*.md` already follow a conventional structure (see e.g. BON-519 ¬ßA-¬ßB-¬ßD pattern). For BON-513 the classifier reads ONE specific subsection convention: the **"DEFER via xfail" block**. We standardize that block's shape now (this memo IS the schema lock) so the parser has a deterministic grammar to chew on, mirroring BON-519 ¬ßD9's "memory artifact schema lock" stance (BON-519 ¬ßD9 lines 681-688).

**Block grammar (Markdown subsection inside any ¬ßA/¬ßB/¬ßD section):**

```
### DEFER via xfail ‚Äî <free-form title>

- **Test module:** `tests/<path>/test_<file>.py`
- **Test name (substring or full id):** `test_<name>`
   (or `test_<name>[<param>]` when parametrized)
- **Flag:** `_<NAME>_PRESENT`
- **Condition:** `condition = not (_X_PRESENT and _Y_PRESENT)`
   (exact Python expression; the parser treats this as the source of
   truth for what the flag is SUPPOSED to AND together)
- **Referenced symbols (gap surface):**
   - `bonfire.module.thing_a`
   - `bonfire.module.thing_b`
- **Reason:** <free-form prose>
- **Strict:** `True` | `False`

> Optional CORRECTED footer (added by the auto-bounce; never authored
> by humans):
> CORRECTED 2026-04-28T21:00:00Z: widened _PARSER_HELPERS_PRESENT to
> AND _EXTRA_META_PRESENT.
```

**Parsed dataclasses (`src/bonfire/verify/decision_log.py`):**

```python
@dataclass(frozen=True)
class DecisionLogEntry:
    target_test_module: str             # repo-relative path from "Test module:"
    target_test_name_substring: str     # value from "Test name:" line
    referenced_flags: tuple[str, ...]   # extracted from "Condition:" expression
    referenced_symbols: tuple[str, ...] # bullets under "Referenced symbols:"
    condition_expression: str           # raw RHS of "Condition:" (for diff)
    strict: bool
    reason: str
    section_anchor: str                 # markdown line offset for round-trip edits
    corrected_at: str | None            # ISO timestamp from CORRECTED footer, if any


@dataclass(frozen=True)
class FlagDefinition:
    """Parsed `_X_PRESENT = ...` block from a test-module's source."""
    name: str                                  # e.g. "_PARSER_HELPERS_PRESENT"
    imported_symbols: tuple[str, ...]          # symbols probed in try/import
    imported_module: str                       # e.g. "bonfire.handlers.wizard"
    dep_count: int                             # len(imported_symbols)
    source_lineno: int                         # for downstream Edit precision
    source_file: str                           # repo-relative path
    raw_block: str                             # original 3-7 line block

    @property
    def referenced_extras(self) -> tuple[str, ...]:
        """Symbols mentioned ANYWHERE in the test body that aren't in
        imported_symbols (this is computed by the test-source AST scan
        in classifier.py ‚Äî NOT in this dataclass)."""
        ...   # returns tuple captured at construction time
```

**Header conventions (parser landmarks):**

The parser uses these grep-anchors (in priority order, mirroring how `wizard.py:71-79` uses anchored regex):

1. `^### DEFER via xfail` ‚Äî section start (case-insensitive, flexible whitespace).
2. `^- \*\*Test module:\*\*\s*\`(?P<path>[^\`]+)\`` ‚Äî module path.
3. `^- \*\*Test name(?: \(.*\))?:\*\*\s*\`(?P<name>[^\`]+)\`` ‚Äî test name/substring.
4. `^- \*\*Flag:\*\*\s*\`(?P<flag>_[A-Z_]+_PRESENT)\`` ‚Äî primary flag.
5. `^- \*\*Condition:\*\*\s*\`(?P<expr>[^\`]+)\`` ‚Äî condition expression.
6. `^- \*\*Referenced symbols.*:\*\*$` followed by `^\s*-\s*\`(?P<sym>[^\`]+)\`` bullets.
7. `^- \*\*Strict:\*\*\s*\`(?P<bool>True|False)\`` ‚Äî strict flag.
8. `^> CORRECTED (?P<ts>[^:]+):` ‚Äî corrected-footer detection.

**Fail-safe behavior (parser-level):**

- Missing required field (test module / test name / flag) -> entry skipped, parser logs `WARN decision_log.entry_skipped reason=missing_<field>`. Classifier sees `decision_log_entries=()` for that section and returns `INDETERMINATE` per ¬ßD2 step 2.
- Multiple "DEFER via xfail" subsections in one memo are all parsed; classifier scans all of them for the candidate match in ¬ßD2 step 4.
- Markdown structural drift (missing bullets, alternate punctuation) -> parser is intentionally strict-but-forgiving on whitespace, strict on the bullet-prefix anchors (`- **Field:**`). This forces Sages to follow the schema; tolerance breeds drift (mirrors BON-519 ¬ßD5 "frozen schema" stance, BON-519 ¬ßD5 line 506).
- Encoding / UTF-8 BOM tolerated (parser opens with `errors="replace"`).

**Why Markdown and not JSON-in-fenced-block?**

- The decision logs are sellable documentation per `feedback_sellable_documentation.md` (memory ref). Markdown bullets are reviewable by humans; JSON would force a separate doc surface.
- Adding a JSON block alongside is a D-FT (post-v0.1 if drift becomes painful).
- The grammar above is greppable and round-trippable by the correction agent without needing a parser library ‚Äî it can `Edit` the bullet line by line.

**Reference seeds for the parser unit tests:** Knight tests fixture files include synthetic memos with (a) one well-formed DEFER block, (b) two DEFER blocks, (c) malformed (missing Flag bullet), (d) corrected-footer present, (e) Cyrillic/zero-width-char noise injected (no semantic meaning, parser must ignore ‚Äî mirrors `test_security_hooks_blindspots.py` Cyrillic discipline). The synthetic fixtures live under `tests/fixtures/sage_decision_logs/`.

## ¬ßD7 ‚Äî Re-Verify Loop (Cherry-Pick + Re-Run + Re-Classify)

**Decision:** The re-verify loop is a private async helper inside `SageCorrectionBounceHandler` (not a top-level pipeline stage). Mirrors BON-519 ¬ßD3's "scratch worktree primitive is a helper of the handler, not its own stage" stance (BON-519 ¬ßD3 line 298). The loop runs at most ONCE per pipeline pass (`MAX_BOUNCE_ATTEMPTS = 1`); a second failure escalates rather than re-bouncing.

**Loop pseudocode (inside `handle` after correction-agent verdict CORRECTED):**

```
async def _re_verify_after_correction(
    self,
    *,
    warrior_branch: str,
    correction_branch: str | None,    # may be None if agent committed in-place
    repo_path: Path,
    pytest_command: tuple[str, ...],
    pytest_timeout_seconds: int | None,
) -> tuple[ClassificationResult, str]:   # (re-run classification, pytest_stdout)
    1. If correction_branch is not None and != warrior_branch:
         await _git("cherry-pick", correction_branch, cwd=repo_path)
       (subagent SHOULD have committed on warrior_branch directly per
        SMEAC ¬ßExecution step 6; this branch is the safety net for the
        case where the agent created a side branch.)
    2. proc = await asyncio.create_subprocess_exec(
            *pytest_command,
            cwd=repo_path,
            stdout=PIPE, stderr=PIPE,
       )
       try:
           stdout, stderr = await asyncio.wait_for(
               proc.communicate(),
               timeout=pytest_timeout_seconds,
           )
       except asyncio.TimeoutError:
           proc.kill()
           await proc.wait()             # subprocess teardown contract
           raise PytestTimeoutError(...) # caught by handler outer try
    3. Parse stdout into FailingTest tuple (reuse the same parser as
       step 6 in ¬ßD3 handle flow).
    4. classifier = Classifier(
           warrior_failures=parsed_failures,
           decision_log_entries=<re-parse the corrected memo>,
           flag_definitions=<re-parse the corrected test module>,
       )
       result = classifier.classify()
    5. Return (result, stdout.decode(errors='replace'))
```

**Subprocess discipline (HARD constraints from project rules):**

- `pytest_command: tuple[str, ...]` ‚Äî `shell=False` always. (Per Wizard SMEAC constraint and `feedback_no_economic_cuts` no-shortcut law.)
- Subprocess teardown on timeout: `proc.kill()` + `await proc.wait()`. (Mirrors BON-519 ¬ßD2 line 294 error-handling stance.)
- Cherry-pick on conflict: catch `subprocess.CalledProcessError` from `_git("cherry-pick", ...)` and `_git("cherry-pick", "--abort")` to leave the branch clean before raising `RuntimeError("cherry-pick conflict")` for the handler outer try to absorb.

**Result branching (mirrors BON-519 ¬ßD2 step 11 envelope-build):**

| Re-classify result | Outcome | Envelope shape |
|---|---|---|
| `result.verdict == INDETERMINATE` AND `failures == ()` (i.e. green) | continue pipeline | `with_status(COMPLETED)` + `META_SAGE_BOUNCE_VERDICT="corrected"` + `META_SAGE_BOUNCE_ATTEMPTS=1` |
| `result.verdict == SAGE_UNDER_MARKED` AND `attempts >= MAX_BOUNCE_ATTEMPTS` | escalate | see ¬ßD8 |
| `result.verdict == WARRIOR_BUG` after correction | escalate | see ¬ßD8 (rationale="warrior_bug_after_correction") |
| Any exception during re-verify | error envelope | `with_error(ErrorDetail(error_type="...", message="...", stage_name=stage.name))` |

**Idempotence + cleanup:**

- Re-verify NEVER mutates anything outside the warrior branch and the pytest cache. The classifier, decision-log parser, and flag-definition parser are pure. The pytest run mutates only the local git index/worktree if test ran (via fixtures); standard pytest isolation applies.
- We do NOT create scratch worktrees here (that's BON-519's tool). We re-run pytest in the existing Warrior worktree ‚Äî the correction agent already committed the corrected xfail there.
- If a test fixture happens to leave debris (e.g. tmp files), the next pipeline stage's CWD is unchanged. v0.1 accepts the small risk of fixture-leftover noise; cleanup is the test author's job, not the handler's.

**`event_bus.emit` on re-verify (forward-compat hooks):**

The handler emits these events when an event_bus is provided (mirroring `wizard.py:266-269`):

- `SageBounceClassified(verdict, candidate_test, attempts, cost_usd)` ‚Äî after first classification.
- `SageCorrectionDispatched(memo_path, suspected_flag_name, missing_deps)` ‚Äî before agent dispatch.
- `SageCorrectionVerdict(verdict_tag, cost_usd)` ‚Äî after agent verdict parse.
- `SageBounceReVerified(verdict, pytest_returncode, pytest_duration_seconds)` ‚Äî after re-run pytest.
- `SageBounceEscalated(rationale)` ‚Äî when escalating to Wizard.

These event types live in `src/bonfire/events/types.py` (assumed extension; if absent in v0.1 they degrade to log-only ‚Äî the handler `await self._emit(event)` no-ops if `_bus is None`).

## ¬ßD8 ‚Äî Escalation to Wizard

**Decision:** Escalation is **non-fatal** ‚Äî the envelope returns COMPLETED with structured metadata, allowing the pipeline's existing `bard -> wizard` arc to proceed. Wizard's review prompt (built by `wizard.py:191` `_build_review_prompt`) gets a *new context block* describing the bounce attempt's outcome, so Wizard can factor "this was bounced once and still failed" into its verdict. Mirrors BON-519 ¬ßD8 lines 593-598 stance ‚Äî the gate FAILS, but the envelope itself isn't a poison-pill; downstream stages can read the metadata and decide.

**Three escalation paths:**

| Trigger | `META_SAGE_BOUNCE_VERDICT` value | Pipeline downstream behavior |
|---|---|---|
| `attempts >= MAX_BOUNCE_ATTEMPTS` and re-verify still fails | `"escalate_max_attempts"` | Bard runs (PR is opened so Wizard can see it); Wizard sees `META_SAGE_BOUNCE_VERDICT="escalate_max_attempts"` in envelope.metadata and adds a "bounce-attempt context" block to its review prompt. |
| Correction agent emitted `<verdict>UNFIXABLE</verdict>` | `"escalated"` (rationale="agent_self_unfixable") | Same as above. |
| Re-classifier returned `WARRIOR_BUG` after correction | `"escalated"` (rationale="warrior_bug_after_correction") | Same as above. |
| Re-verify raised an exception (cherry-pick conflict, pytest crash) | envelope `with_error(...)` (FAILED) | Pipeline halts. The standard error envelope flow applies ‚Äî this is NOT escalation, it's a handler error. |

**Envelope shape on escalation (NOT error):**

```python
envelope.with_metadata(
    sage_bounce_verdict="escalated",      # or "escalate_max_attempts"
    sage_bounce_attempts=str(attempts),
    sage_bounce_classification=json.dumps({
        "verdict": "sage_under_marked",
        "candidate_test_file": "tests/unit/test_wizard_handler.py",
        "candidate_test_name": "test_parse_verdict_extra_meta",
        "suspected_flag_name": "_PARSER_HELPERS_PRESENT",
        "suspected_missing_deps": ["_EXTRA_META_PRESENT"],
        "rationale": "agent_self_unfixable",
        "attempts": 1,
        "pytest_stdout_tail": "<last 2KB>",
    }),
    correction_branch=correction_branch_or_empty_string,
    original_sage_memo_path=memo_path,
).with_status(TaskStatus.COMPLETED).with_result(
    "sage_correction_bounce: escalated to wizard "
    "(verdict=escalated, attempts=1, candidate=test_parse_verdict_extra_meta)"
)
```

**Wizard prompt augmentation (downstream effect ‚Äî separate D-FT to land in Wizard):**

A *small, additive* edit to `wizard.py:191` `_build_review_prompt` reads the metadata when present and prepends a context section:

```
## Bounce-attempt context

This PR was processed by the Sage correction-bounce stage.
- Verdict: {sage_bounce_verdict}
- Attempts: {sage_bounce_attempts}
- Suspected flag: {suspected_flag_name}
- Suspected missing deps: {suspected_missing_deps}
- Rationale: {rationale}

Factor this into your review. If the bounce verdict is "corrected", the
xfail markers have already been widened ‚Äî the diff you're reviewing
includes that correction.
```

**Sage decision:** the Wizard-prompt augmentation is a **D-FT child of BON-513**, NOT a v0.1 hard requirement. v0.1 ships the bounce stage with metadata; Wizard reads or ignores. Knight integration tests assert metadata is present; they do NOT yet assert Wizard prompt includes the new section. This keeps the change-surface tight per BON-612 discipline.

**Max-attempt counter discipline:**

- `META_SAGE_BOUNCE_ATTEMPTS` is stringified-int (envelope metadata is `dict[str, Any]`; `prior_results: dict[str, str]` forces stringification at boundary).
- Counter is reset to `"0"` on every NEW pipeline pass (the engine constructs fresh prior_results from each stage's result; the counter only persists within one stage's `handle` call across the re-verify loop).
- For multi-pass scenarios (rare; would only happen if a downstream gate bounces back to warrior, which would itself reset Knight/Warrior context), the counter is REPLACED, not accumulated. v0.1 treats each pipeline pass as a fresh bounce budget.

**Why max_attempts=1?**

- The classifier is deterministic. If after one widening the test still fails, either (a) the agent widened the wrong flag, (b) there's a real Warrior bug, or (c) the under-specification is structurally wider than xfail can express. Cases (a) and (c) need human judgment (Wizard); case (b) is by definition not a bounce target. Looping more than once just burns tokens.
- Future ticket can lift this if patterns emerge (D-FT candidate).

## Failure classification

The classifier returns one of three deterministic verdicts:

| Verdict | Meaning | Pipeline action |
|---|---|---|
| `green` | Warrior verdict is COMPLETED with passing tests | Stage no-op; metadata `META_CORRECTION_SKIPPED_REASON="warrior_green"`. |

## Correction cycle

When the verdict is `sage_under_marked`:

1. Handler extracts the missing-deps set (cited - sage-specified).
2. Dispatches a Sage-correction agent with `tools=["Read", "Edit", "Grep"]`. Agent edits xfail decorators, dep-presence flags, and the Sage decision log to over-specify. NO `Write`, NO `Bash`. Scope-limited by tool restriction.
3. Cherry-picks the correction commit onto the warrior branch.
4. Re-runs pytest with the same args the Warrior used.
5. On re-verify pass: `metadata[META_CORRECTION_VERDICT]="corrected"`, pipeline proceeds.
6. On re-verify still-fail: `metadata[META_CORRECTION_VERDICT]="escalated"`, `META_CORRECTION_ESCALATED=True`; pipeline proceeds to Wizard with the escalation flag.

## ¬ßD-CL.7 ‚Äî superpowers:code-reviewer Focus Areas (12 cross-cutting items)

(Outside-the-contract concerns ‚Äî bugs the per-handler-test-suite cannot catch. BON-519 ¬ßD-CL.7 ships 10 items; BON-513 ships 12 because the correction-cycle adds the cherry-pick + re-verify axes. Each item lists explicit failure-shape + suggested mitigation.)

1. **Async cancellation propagation in correction cycle.** *Failure shape:* outer pipeline cancels the handler mid-Sage-dispatch; the backend task leaks and continues consuming budget after the envelope is rejected. *Mitigation:* wrap the `await self._backend.execute(...)` in a `try / except asyncio.CancelledError` block that explicitly cancels the inner task and re-raises. Verify by inspecting whether `execute_with_retry` already provides this guarantee; if not, wrap it.
2. **Subprocess teardown on re-verify timeout.** *Failure shape:* re-verify pytest hangs (e.g., a test imports a module that opens a TCP connection); subprocess outlives the handler; worktree corruption on next acquire. *Mitigation:* on timeout, `proc.kill()` THEN `await proc.wait()` (BON-519 ¬ßD-CL.7 #2 verbatim). NEVER just `proc.terminate()` without await ‚Äî kernel race.
3. **Envelope-leak on FAILED return paths.** *Failure shape:* error path constructs `with_error(...)` but does not preserve `metadata[META_CLASSIFIER_VERDICT]` set earlier, so downstream stages (Wizard) cannot read why the correction failed. *Mitigation:* every FAILED return path uses `model_copy(update={"metadata": {**envelope.metadata, ...}, "error": ...})` instead of bare `with_error` (mirror `bard.py` line 254‚Äì273).
4. **Race condition: parallel pipelines invoking sage_correction on overlapping warrior branches.** *Failure shape:* two pipelines cherry-pick to the same branch and produce ambiguous git state. *Mitigation:* handler reads `envelope.working_dir` (worktree-scoped); composition root MUST allocate a unique worktree per pipeline run (BON-519's `feedback_worktrees_for_parallel.md` discipline). Code reviewer verifies the handler NEVER calls `os.getcwd()` or `os.chdir()`.
5. **File-write atomicity for decision-log corrections.** *Failure shape:* Sage edits the decision log; partial write corrupts the file when interrupted. *Mitigation:* the Sage-correction agent uses `Edit` (read-modify-write atomic by Claude SDK contract). Code reviewer verifies the handler does NOT itself rewrite the decision log via `open(..., "w")`. The agent does the editing ‚Äî the handler orchestrates only.
6. **Classifier non-determinism via dict ordering.** *Failure shape:* `parse_sage_decision_log` returns a `frozenset` (correct), but `Classification.cited_deps` accidentally returns a `list` whose order varies by Python version ‚Üí test flake. *Mitigation:* every set-shape field on `Classification` is explicitly `frozenset[str]` in the dataclass annotation; classifier never returns mutable types. Code reviewer greps for `list` annotations on `Classification` and rejects.
7. **Idempotency of cherry-pick.** *Failure shape:* cherry-pick fails mid-apply, leaves working tree in MERGING state. Subsequent re-verify pytest runs against a corrupt tree. *Mitigation:* on cherry-pick failure, handler runs `git cherry-pick --abort` (or `git reset --hard HEAD`) BEFORE returning FAILED.
