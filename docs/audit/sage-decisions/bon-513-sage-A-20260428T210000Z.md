### Q2 — Classifier algorithm: structured front-matter, prose pattern-match, or hybrid?

**Options:**
- **(a) Sentinel front-matter block** — Sage memos MUST include a YAML/TOML front-matter region with explicit `xfail_defers:` array of `{flag, condition_module, condition_attr}` records; classifier parses structurally.
- **(b) Pattern-match free-form prose** — Sage memos use a canonical heading like `## DEFER via xfail` (per ticket text) and the classifier regex-scans for that heading + bullet patterns underneath.
- **(c) Hybrid** — accept either; prefer front-matter, fall back to prose when absent. Classifier emits `parse_source: Literal["front_matter", "prose", "absent"]` for forensics.

**Sage proposal: (c) hybrid, prefer (a) for new memos.**

**Reasoning:**
1. The ticket text says "parses Sage decision log for 'DEFER via xfail' sections" (free-form prose). That's option (b). But option (b) alone is fragile: every Sage memo must hew to the same heading literal, and a typo silently masks the marker. The XML-vs-stdout fail-safe in BON-519 §D4 (`parse_pytest_stdout_fallback`) is the precedent — structured first, prose second.
2. Existing Sage memos under `docs/audit/sage-decisions/` (35+ files) do NOT currently use a front-matter block; they're free-form Markdown with `## §A`, `## §D-CL.*` section headers. Forcing retrofit is OOS for v0.1.
3. Adding an OPTIONAL `<!-- bonfire:defers -->` HTML-comment block (or YAML front-matter delimited by `---`) at memo top, parsed when present, lets new memos opt into structured semantics without breaking existing memos. Old memos parsed via prose fallback.
4. Determinism requirement (ticket: "deterministic decision-log scanning only") rules out LLM-backed parsing. Both (a) and (b) are deterministic; (c) is the union.

**Anchor:** Ticket "Out of scope: LLM-backed classification". BON-519 §D4 fail-safe pattern (`parse_pytest_junit_xml` + `parse_pytest_stdout_fallback`).

**Consequences:**
- Two parser functions in `src/bonfire/verify/classifier.py`:
  - `parse_defer_front_matter(memo_text: str) -> tuple[DeferRecord, ...]` — parses `<!-- bonfire:defers -->...<!-- /bonfire:defers -->` HTML-comment block (YAML inside) OR fenced ```yaml bonfire:defers ... ``` block.
  - `parse_defer_prose(memo_text: str) -> tuple[DeferRecord, ...]` — regex-scans for `## DEFER via xfail` (case-insensitive) and bullet lines like `- flag: X_PRESENT, module: bonfire.x, attr: X`.
- A canonical `DeferRecord` dataclass: `(flag_name: str, module: str, attr: str, source: Literal["front_matter", "prose"])`.
- `BounceClassification.parse_source: Literal["front_matter", "prose", "absent"]` for forensics.
- The "absent" case (no DEFER section in memo) means the Warrior failure is NOT classifiable as `SAGE_UNDER_MARKED` — fall through to `WARRIOR_BUG` (escalate to Wizard).
- Sage tooling discipline: starting BON-513+, new Sage memos SHOULD use the structured front-matter block. Documented in `docs/pipeline-stages.md` §SageCorrectionBounce.

**Sub-decision Q2a:** Front-matter format. Sage proposal: HTML-comment-wrapped YAML to coexist with Markdown rendering on GitHub. Format:
```

- flag: ANALYST_PRESENT
  module: bonfire.agent.roles
```

### Q4 — Correction-agent shape: full Sage subagent or restricted "xfail-edit-only" agent?

**Options:**
- **(a) Full Sage subagent re-dispatched** — re-invoke the Sage role (synthesizer) with full read/write access; trust the agent to scope its edits.
- **(b) Restricted "xfail-corrector" agent** — new narrow agent role with tools constrained to `Read`, `Edit` on a whitelisted file set, and `Write` denied. Cannot touch `src/`, `tests/<body>`, fixtures, assertions; can only edit `xfail` decorators / flag definitions / Sage memo decision-log.
- **(c) Pure-deterministic correction (no agent)** — generate the corrected xfail markers programmatically from the classifier's parsed `DeferRecord` set + the actual failing test files. No LLM in the correction loop.

**Sage proposal: (b) restricted "xfail-corrector" agent, with tool-policy enforcement at dispatch site.**

**Reasoning:**
1. Option (c) is tempting (cheapest, fastest, deterministic) but fails the "under-specified" case: when a Sage memo says `xfail(condition=not _ANALYST_PRESENT)` and the actual failure is because BOTH `_ANALYST_PRESENT` AND `_VERIFIER_PRESENT` are missing, deterministic correction has no reliable way to MAP the failing test to the missing flag set without re-reading the Sage memo's reasoning prose. An LLM trained on the memo handles this; a regex-based corrector either over-corrects (adds every flag in the memo) or under-corrects.
2. Option (a) (full Sage) is too permissive — Sage's tool surface includes Write; a runaway correction could rewrite test bodies or src/. The ticket explicitly forbids touching src/, test bodies, fixtures, assertions.
3. Option (b) — narrow agent with hard-locked tools — is the right shape. Tool policy enforcement happens at the dispatch site (per BON-118 lesson, see `feedback_full_pipeline_protocol.md`): the SageCorrectionBounce handler dispatches with `DispatchOptions(allowed_tools=("Read", "Edit"), denied_tools=("Write", "Bash"))` and a path allowlist (`docs/audit/sage-decisions/*.md` + a whitelisted set of test files containing `@pytest.mark.xfail` decorators).

**Anchor:** `src/bonfire/protocols.py` `DispatchOptions`; `src/bonfire/handlers/wizard.py:43` (review agent runs read-only — same shape); BON-519 §D-CL.4 ("Pure-function classifier ... I/O lives in the handler shell") for separation of concerns; ticket "Edits ONLY xfail decorators / flag definitions / decision log; never src/, never test bodies/fixtures/assertions".

**Consequences:**
- New `AgentRole` member NOT introduced. Reuse `AgentRole.SYNTHESIZER` (already shipped, `roles.py:40`) — the corrector IS a Sage doing surgical work, not a new role. Display layer: gamified "Sage" appears in CLI output for the correction stage (consistent with the role name).
- Dispatch site: `SageCorrectionBounceHandler.handle()` constructs `DispatchOptions(allowed_tools=frozenset({"Read", "Edit"}))`. Bash is implicitly denied (not in allowed). Write is implicitly denied.
- Path allowlist enforced via a pre-dispatch hook (or via the agent prompt — Sage proposal: belt-and-suspenders, BOTH a prompt-level discipline rule AND a post-dispatch verification step that diffs the correction branch and rejects any file outside the allowlist).
- Post-dispatch verification: the handler reads the correction commit's `git diff --name-only HEAD~1 HEAD` from the correction worktree; if any path is outside `{docs/audit/sage-decisions/*.md, tests/**/conftest.py, tests/**/*.py}` AND that test file's diff hunks touch any line NOT a `@pytest.mark.xfail` decorator or flag-definition assignment, classifier returns `correction_failed`.
- Correction agent SMEAC discipline: SMEAC explicitly forbids touching test bodies/assertions/fixtures (mirrors BON-519 SMEAC discipline for boundary-respecting agents).

**Sub-decision Q4a — naming the correction sub-stage:** the auto-dispatched agent is internally a "Sage correction sub-agent". Sage proposal: keep the agent role string as `"sage"` (gamified) backed by `AgentRole.SYNTHESIZER`; the SMEAC variant carries a `correction_mode=True` flag in the prompt. No new role string, no new role enum member. Anta to ratify.

### Q5 — Loop semantics: max correction attempts before escalation?

**Options:**
- **(a) 1 attempt** — single bounce. If the correction fails, escalate to Wizard immediately.
- **(b) 2 attempts** — give the corrector one retry with the post-mortem of attempt 1.
- **(c) 3 attempts** (mirror Warrior `max_iterations=3`) — match Warrior's TDD retry budget.
- **(d) Configurable** — `BonfireSettings.pipeline.sage_correction_max_attempts: int = 1`.

**Sage proposal: (a) 1 attempt for v0.1; expose as configurable setting (default 1) for forward compatibility.**

**Reasoning:**
1. The under-specified-xfail symptom is a NARROW class of failure. The corrector either gets it right by reading the memo + failure list (high probability with a focused agent), or the memo lacks the information needed to specify correctly (in which case retrying with the same inputs won't help).
2. Warrior's `max_iterations=3` exists because Warrior is doing creative implementation work — three swings at making a contract green. Sage-correction is purely surgical edits; it's closer to a code-formatter than a creative agent. One attempt is the right budget.
3. Token cost discipline: each correction attempt is a full Opus dispatch (Sage role). Two attempts double cost; three triple it. For a v0.1 keystone, 1-attempt + escalate-to-Wizard is the cheapest right answer.
4. Configurability: expose `sage_correction_max_attempts` on `BonfireSettings` to let power users tune. Default 1 keeps the v0.1 ship contract honest.
5. Counter (option b/c): if Anta wants more attempts later, the path is a setting bump, not a code change. Build the seam.

**Anchor:** `src/bonfire/workflows/standard.py:58` (`max_iterations=3` for Warrior — the comparison case); `src/bonfire/models/config.py` (settings home); ticket "Pipeline stage SageCorrectionBounce — auto-dispatches Sage-correction".

**Consequences:**
- `StageSpec` for `sage_correction_bounce` uses `max_iterations=1`. The handler internally does NOT loop; loop semantics are surfaced via the StageSpec contract so the engine handles retry orchestration uniformly.
- New setting: `BonfireSettings.pipeline.sage_correction_max_attempts: int = Field(default=1, ge=1, le=5)`.
- Handler reads the setting via constructor injection (not at handle-time); test fixtures override.
- On exhaustion of attempts: classifier returns `correction_failed`, handler returns FAILED envelope with `ErrorDetail(error_type="sage_correction_exhausted")`, gate `sage_correction_resolved` fails error, pipeline halts with escalation message naming the failing test set + the original Sage memo path.

**Sub-decision Q5a — bounce-back vs in-stage retry:** Sage proposal: in-stage retry (handler loops up to `max_attempts`), NOT a workflow-level bounce edge. Reasoning: the engine's `on_gate_failure` semantic (cf. `standard.py:65, 79`) bounces to a previous stage and re-runs IT. Bouncing to "self" is an in-handler concern. Engine bounce edges are reserved for cross-stage retries (Warrior on test failure, Warrior on Wizard rejection).

### Q6 — Decision-log schema: canonical "DEFER via xfail" header or pattern-match free-form?

**Options:**
- **(a) Strict canonical header** — Sage memos MUST contain a literal `## DEFER via xfail` H2 heading; classifier scans only for that heading.
- **(b) Tolerant prose pattern** — accept variants: `## DEFER`, `### xfail defers`, `## DEFER via xfail`, `## §D-DEFER`, etc. Regex with case-insensitive + whitespace-tolerant matching.
- **(c) Front-matter only** — see Q2 (c). Prose scanning is fallback only.

**Sage proposal: combination — Q2's hybrid (front-matter preferred, prose fallback) PLUS a single canonical prose heading literal `## DEFER via xfail` (matching ticket text verbatim).**

**Reasoning:**
1. Per Q2, front-matter is the structured path. When prose is the source of truth, a single canonical literal heading prevents ambiguity. Anchor the literal to the ticket: `## DEFER via xfail` (case-insensitive match, no other variants accepted).
2. Reject pattern-tolerance (option b): every alternative spelling is a hidden-failure surface. If a Sage memo uses `## DEFER` instead of `## DEFER via xfail`, it should be flagged on Wizard review, not silently parsed.
3. The bullet schema under the heading is also locked: each defer record is a Markdown list item of the form:
   ```
   - flag: <FLAG_NAME>, module: <module.path>, attr: <Module.Attribute>
   ```
   The classifier regex matches `^-\s+flag:\s*(\S+),\s*module:\s*(\S+),\s*attr:\s*(\S+)\s*$` after a `## DEFER via xfail` heading.
4. The schema lock is documented in `docs/pipeline-stages.md` §SageCorrectionBounce (consumed by Sage memo authors going forward).

**Anchor:** Ticket text "Sage decision log for 'DEFER via xfail' sections"; `docs/audit/sage-decisions/bon-350-sage-20260427T182947Z.md` for the prevailing memo style (free-form Markdown, no front-matter); BON-519 §D7 for documentation precedent.

**Consequences:**
- Classifier rejects (with `parse_source="absent"`) any memo without either front-matter OR a `## DEFER via xfail` heading.
- The strict heading literal is a constant in the classifier module: `_DEFER_HEADING_RE = re.compile(r"^##\s+DEFER\s+via\s+xfail\s*$", re.IGNORECASE | re.MULTILINE)`.
- Bullet record regex: `_DEFER_RECORD_RE` with named groups `flag`, `module`, `attr`.
- Documentation: a "Sage memo defer schema" subsection in `docs/pipeline-stages.md` showing the canonical front-matter AND prose syntaxes, with a one-paragraph rationale citing this Q.
- Test surface: classifier unit tests cover (i) front-matter only, (ii) prose only, (iii) both (front-matter wins), (iv) canonical heading with malformed bullets (bullets ignored, parse_source="prose", records empty), (v) ALL absent (parse_source="absent"), (vi) wrong heading variant (e.g., `## DEFER`) — treated as absent.

The Sage memo lives in `docs/audit/sage-decisions/bon-XXX-sage-*.md` on that same branch. Correction commits MUST land on the same branch so Bard's PR captures both the original work AND the correction in one diff.

The post-correction merge step in the ticket ("Post-correction merge + re-verify — cherry-pick xfail fixes, re-run pytest") only makes sense if the correction commit is on the same branch as the Warrior commit. ### Q8 — Synthetic-ticket fixture: programmatic factory or stored fixture data?

**Options:**
- **(a) Programmatic factory** — fixture function builds a synthetic Sage memo + Warrior failure list in-memory; integration test uses a temp git repo + temp file writes.
- **(b) Stored fixture data** — JSON/Markdown files under `tests/fixtures/sage_correction/` with hand-authored realistic memos and failure XML.
- **(c) Hybrid** — base templates stored as fixtures, parametrize-and-render them via a factory.

**Sage proposal: (c) hybrid.**

**Reasoning:**
1. Pure factory (a) keeps test code self-contained but produces unrealistic memos — synthetic memos don't exercise real prose patterns. The classifier regex must work on REAL Sage memo text from `docs/audit/sage-decisions/`, not on lab-grown text.
2. Pure stored fixture (b) is more realistic but creates "magic file" dependencies — every test refers to a fixture path; debugging a regex requires opening the fixture file. And maintenance: if the schema changes, every fixture file must be updated.

### Q9 — Gate vs Stage: does the classifier produce a Gate or a Stage?

**Options:**
- **(a) Stage-only** — `SageCorrectionBounceHandler` is a full StageHandler; its `handle()` runs classifier + correction + re-verify; no separate gate.
- **(b) Stage + Gate** — handler writes classifier result to envelope metadata; a separate `SageCorrectionResolvedGate` reads metadata and produces gate verdict.
- **(c) Gate-only** — classifier runs as part of a Gate; correction is a separate stage triggered by gate failure.

**Sage proposal: (b) Stage + Gate.**

**Reasoning:**
1. BON-519 §A Q1 + §D6 established the precedent: a deterministic verification mechanism produces BOTH a handler (the I/O orchestrator) AND a gate (the engine-readable verdict). `MergePreflightHandler` writes `META_PREFLIGHT_CLASSIFICATION` + `META_PREFLIGHT_TEST_DEBT_NOTED`; `MergePreflightGate` reads them.
2. Gates are the engine's pluggable decision primitive (`src/bonfire/engine/gates.py:33-103` — six existing gate classes follow the same shape: read envelope, return GateResult). Reusing this pattern keeps the engine surface uniform.
3. Stage-only (option a) hides the verdict from the engine's gate-aware logic: pipeline-level escalation, severity reporting, GateChain composition (`gates.py:126-139`) all work on Gate verdicts. Burying the classification inside the handler forfeits these.
4. Gate-only (option c) inverts ownership awkwardly: gates are stateless evaluators, but the correction work is stateful (commit, push, re-verify). Stage owns the work; gate owns the verdict.

**Anchor:** `src/bonfire/engine/gates.py:33-103` (six existing gates); BON-519 §D6 ("File 3: `src/bonfire/engine/gates.py` — new `MergePreflightGate`"); `src/bonfire/protocols.py:175-184` (QualityGate Protocol).

**Consequences:**
- New gate: `SageCorrectionResolvedGate` in `src/bonfire/engine/gates.py`.
  ```python
  class SageCorrectionResolvedGate:
      """Passes when bounce verdict is GREEN, WARRIOR_BUG (escalate), or
      SAGE_UNDER_MARKED + corrected. Fails on AMBIGUOUS or correction_failed."""
      async def evaluate(self, envelope, context) -> GateResult: ...
  ```
- Engine `__all__` extension: `__all__ += ["SageCorrectionResolvedGate"]` in `src/bonfire/engine/__init__.py` (mirrors BON-519 §D6 File 4).
- Metadata key: `META_BOUNCE_VERDICT` in `src/bonfire/models/envelope.py:174` (after `META_TICKET_REF` line 173). Value is `BounceVerdict` StrEnum value.
- Pipeline composition root reads gate via `stage.gates=["sage_correction_resolved"]` (Q3 already specifies this).
- `WARRIOR_BUG` verdict + gate: this is the genuine-bug escalate path. Gate result is `passed=True, severity="warning"` (gate passes — pipeline continues to Bard with a Wizard escalation flag) OR `passed=False, severity="error"` (gate blocks pipeline; Wizard intervention required). **Sage proposal: `passed=True, severity="warning"`** — pipeline proceeds to Bard so Wizard sees the PR with full context; Wizard's review surface displays the bounce verdict as a warning. Anta to ratify (alternative is hard-block).

**Sub-decision Q9a — `AMBIGUOUS` handling:** when classifier verdict is `AMBIGUOUS` (Q1a), gate returns `passed=False, severity="error"` — pipeline halts. This forces Wizard inspection rather than auto-bouncing on uncertain inputs. Conservative default; Anta to ratify.
