## Failure classification

The classifier returns one of three deterministic verdicts:

| Verdict | Meaning | Pipeline action |
|---|---|---|
| `green` | Warrior verdict is COMPLETED with passing tests | Stage no-op; metadata `META_CORRECTION_SKIPPED_REASON="warrior_green"`. |
| `sage_under_marked` | Failing tests cite deps that the Sage decision log did NOT enumerate in its DEFER-via-xfail section | Auto-dispatch Sage-correction agent → cherry-pick → re-verify. |
| `warrior_bug` | Failing tests cite NO under-marked deps (Sage specified them all, or they're unconditional failures) | Skip correction; mark `META_CORRECTION_ESCALATED=True`; pipeline proceeds to Wizard with the escalation flag (Wizard sees the bounce and reviews the warrior failure as-is). |

## Correction cycle

When the verdict is `sage_under_marked`:

1. Handler extracts the missing-deps set (cited - sage-specified).
2. Dispatches a Sage-correction agent with `tools=["Read", "Edit", "Grep"]`. Agent edits xfail decorators, dep-presence flags, and the Sage decision log to over-specify. NO `Write`, NO `Bash`. Scope-limited by tool restriction.
3. Cherry-picks the correction commit onto the warrior branch.
4. Re-runs pytest with the same args the Warrior used.
5. On re-verify pass: `metadata[META_CORRECTION_VERDICT]="corrected"`, pipeline proceeds.
6. On re-verify still-fail: `metadata[META_CORRECTION_VERDICT]="escalated"`, `META_CORRECTION_ESCALATED=True`; pipeline proceeds to Wizard with the escalation flag.

## §D-CL.7 — superpowers:code-reviewer Focus Areas (12 cross-cutting items)

(Outside-the-contract concerns — bugs the per-handler-test-suite cannot catch. BON-519 §D-CL.7 ships 10 items; BON-513 ships 12 because the correction-cycle adds the cherry-pick + re-verify axes. Each item lists explicit failure-shape + suggested mitigation.)

1. **Async cancellation propagation in correction cycle.** *Failure shape:* outer pipeline cancels the handler mid-Sage-dispatch; the backend task leaks and continues consuming budget after the envelope is rejected. *Mitigation:* wrap the `await self._backend.execute(...)` in a `try / except asyncio.CancelledError` block that explicitly cancels the inner task and re-raises. Verify by inspecting whether `execute_with_retry` already provides this guarantee; if not, wrap it.
2. **Subprocess teardown on re-verify timeout.** *Failure shape:* re-verify pytest hangs (e.g., a test imports a module that opens a TCP connection); subprocess outlives the handler; worktree corruption on next acquire. *Mitigation:* on timeout, `proc.kill()` THEN `await proc.wait()` (BON-519 §D-CL.7 #2 verbatim). NEVER just `proc.terminate()` without await — kernel race.
3. **Envelope-leak on FAILED return paths.** *Failure shape:* error path constructs `with_error(...)` but does not preserve `metadata[META_CLASSIFIER_VERDICT]` set earlier, so downstream stages (Wizard) cannot read why the correction failed. *Mitigation:* every FAILED return path uses `model_copy(update={"metadata": {**envelope.metadata, ...}, "error": ...})` instead of bare `with_error` (mirror `bard.py` line 254–273).
4. **Race condition: parallel pipelines invoking sage_correction on overlapping warrior branches.** *Failure shape:* two pipelines cherry-pick to the same branch and produce ambiguous git state. *Mitigation:* handler reads `envelope.working_dir` (worktree-scoped); composition root MUST allocate a unique worktree per pipeline run (BON-519's `feedback_worktrees_for_parallel.md` discipline). Code reviewer verifies the handler NEVER calls `os.getcwd()` or `os.chdir()`.
5. **File-write atomicity for decision-log corrections.** *Failure shape:* Sage edits the decision log; partial write corrupts the file when interrupted. *Mitigation:* the Sage-correction agent uses `Edit` (read-modify-write atomic by Claude SDK contract). Code reviewer verifies the handler does NOT itself rewrite the decision log via `open(..., "w")`. The agent does the editing — the handler orchestrates only.
6. **Classifier non-determinism via dict ordering.** *Failure shape:* `parse_sage_decision_log` returns a `frozenset` (correct), but `Classification.cited_deps` accidentally returns a `list` whose order varies by Python version → test flake. *Mitigation:* every set-shape field on `Classification` is explicitly `frozenset[str]` in the dataclass annotation; classifier never returns mutable types. Code reviewer greps for `list` annotations on `Classification` and rejects.
7. **Idempotency of cherry-pick.** *Failure shape:* cherry-pick fails mid-apply, leaves working tree in MERGING state. Subsequent re-verify pytest runs against a corrupt tree. *Mitigation:* on cherry-pick failure, handler runs `git cherry-pick --abort` (or `git reset --hard HEAD`) BEFORE returning FAILED. Code reviewer verifies the abort path exists in the except-block.
8. **Gate-block path coverage.** *Failure shape:* `SageCorrectionGate` returns `passed=True` for the escalation envelope (`META_CORRECTION_ESCALATED=True`), allowing pipeline to proceed. If the Wizard handler does not also see the escalation flag, the bounce is silently dropped. *Mitigation:* Wizard SMEAC verification category #5 explicitly checks Wizard reads `META_CORRECTION_ESCALATED` from `prior_results`. Code reviewer confirms the wiring.
9. **Subprocess argument injection (BON-519 §D-CL.7 #8 verbatim).** *Failure shape:* pytest args read from `prior_results["warrior"]` could in principle contain shell metacharacters if a malicious upstream injected them. *Mitigation:* args are passed as `tuple[str, ...]` to `asyncio.create_subprocess_exec` (NEVER as a single shell string, NEVER `shell=True`). Code reviewer greps for `shell=True` and rejects any occurrence in `sage_correction.py`.
10. **Signal conventions across re-verify subprocess and parent.** *Failure shape:* SIGTERM to the parent does not propagate to the pytest subprocess; pytest survives parent exit. *Mitigation:* spawn pytest with `start_new_session=False` (default) so it shares the process group. On graceful shutdown, parent's signal hits the group. Code reviewer verifies `start_new_session` is NOT True.
11. **Path-guard discipline (BON-519 §D-CL.7 #7 adapted).** *Failure shape:* handler emits absolute paths in error messages or in `metadata[META_CORRECTION_VERDICT]`-adjacent JSON. Per `path_guard.py` discipline these MUST be repo-relative. *Mitigation:* all path serialization runs through a `_relpath(p)` helper that calls `Path(p).relative_to(repo_root)` with a fallback to `Path(p).name` for paths outside the tree. Code reviewer verifies no raw `str(Path(...))` calls in error/metadata builders.
12. **xfail-marker over-specification regression test.** *Failure shape:* a future Sage memo for a different ticket under-marks an xfail in BON-513's own test files; the BON-513 classifier should classify its own tests if dispatched against them. *Mitigation:* Knight B writes ONE meta-regression test that constructs a synthetic warrior failure citing a dep NOT in BON-513's own decision log (this very memo file) and asserts the classifier returns SAGE_UNDER_MARKED. The classifier eats its own dogfood. Code reviewer verifies this meta-test exists and is not xfail-marked.

**Mitigation cross-references:** Items #1, #2, #4, #9, #10 are the BON-519 §D-CL.7 patterns (cancellation, subprocess kill, race, injection, signal) applied to BON-513's surface. Items #3, #5, #6, #7, #8, #11, #12 are BON-513-specific. Code reviewer SMEAC dispatched via `superpowers:code-reviewer` on the PR cites this §D-CL.7 numbered list verbatim.
