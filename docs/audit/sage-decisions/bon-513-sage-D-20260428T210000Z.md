## §D2 — Classifier Algorithm + Verdicts

**Decision:** The classifier is **pure-deterministic, no LLM, no I/O at evaluate time**, mirroring BON-519 §D4's `classify_pytest_run` pure-function discipline (BON-519 §D4 lines 435-456). Out-of-scope per ticket: LLM-backed classification. The classifier reads three inputs (already-resident in memory) and returns a verdict by first-match-wins decision tree.

**Public types (`src/bonfire/verify/classifier.py`):**

```python
class ClassifierVerdict(StrEnum):
    SAGE_UNDER_MARKED = "sage_under_marked"
    WARRIOR_BUG = "warrior_bug"
    INDETERMINATE = "indeterminate"   # decision log absent, malformed, or no DEFER section


@dataclass(frozen=True)
class FailingTest:
    file_path: str            # repo-relative, e.g. "tests/unit/test_wizard_handler.py"
    classname: str
    name: str                 # may include parametrize id ("test_x[case-3]")
    message: str              # truncated failure message
    failure_kind: Literal["xfail_strict_unexpected_pass", "real_failure", "error", "collection_error"]


@dataclass(frozen=True)
class ClassificationResult:
    verdict: ClassifierVerdict
    candidate_test: FailingTest | None        # the single failing test that drove the verdict
    suspected_flag_name: str | None           # e.g. "_PARSER_HELPERS_PRESENT"
    suspected_missing_deps: tuple[str, ...]   # e.g. ("_EXTRA_META_PRESENT",) — flags Sage forgot to AND-into the condition
    decision_log_entry: DecisionLogEntry | None
    rationale: str                            # short, machine-grepable; one of a fixed set of phrases
```

**Classifier signature:**

```python
class Classifier:
    """Deterministic Warrior-failure classifier.

    No I/O. No subprocess. No LLM. Construct with the three deterministic
    inputs and call `.classify()`.
    """

    def __init__(
        self,
        *,
        warrior_failures: tuple[FailingTest, ...],
        decision_log_entries: tuple[DecisionLogEntry, ...],
        flag_definitions: tuple[FlagDefinition, ...],
    ) -> None: ...

    def classify(self) -> ClassificationResult: ...
```

**Decision tree (FIRST MATCH WINS — mirrors BON-519 §D4 line 447 stance):**

```
1. warrior_failures == ()                        -> INDETERMINATE  (rationale="no_failures")
2. decision_log_entries == ()                    -> INDETERMINATE  (rationale="decision_log_absent")
3. ANY failing test has failure_kind="collection_error"
                                                 -> WARRIOR_BUG    (rationale="collection_error")
4. Find candidate: the FIRST failing test whose file_path + test name
   appear in some entry e ∈ decision_log_entries (matched by
   `e.target_test_module` exact-string match against `failing.file_path`,
   AND `e.target_test_name_substring` ⊆ `failing.name`).
   No candidate found -> WARRIOR_BUG (rationale="no_decision_log_match")
5. From candidate's matched entry e, read e.referenced_flags = set of
   `_X_PRESENT`-shaped names cited in the DEFER section's prose or in the
   xfail decorator's `condition=` expression.
   Read flag_definitions = parsed `_X_PRESENT = (...)` records from the
   test module's top-level (set up by parser, see §D5).
6. Compute deps_satisfied(flag) for each cited flag:
     - flag.dep_count == number of distinct module/attr lookups inside
       its initializer (e.g. `try: from m import X, Y` -> 2 deps).
     - For the candidate's referenced flags, if Sage's DEFER prose names
       MORE referenced symbols than `flag.dep_count`, mark the flag
       UNDER-SPECIFIED.
7. Any cited flag is UNDER-SPECIFIED -> SAGE_UNDER_MARKED
                                       (rationale="under_marked_flag",
                                        suspected_flag_name=flag.name,
                                        suspected_missing_deps=flag.missing_deps)
8. Otherwise (flag deps complete, but real failure) -> WARRIOR_BUG
                                       (rationale="flag_well_specified_but_failure_real")
```

**Key edge cases (Knight tests must cover — mirrors BON-519 §D4 edge-cases table):**

| Scenario | Inputs | Expected verdict |
|---|---|---|
| 1 failure, decision-log entry covers test, single-flag condition citing only `_PARSER_HELPERS_PRESENT`, but DEFER prose names both `_parse_verdict` AND `_extra_meta` | flag.referenced=2, flag.dep_count=1 | SAGE_UNDER_MARKED |
| 1 failure, decision-log covers test, condition cites two flags AND-ed, both flags fully specified | flags well-formed | WARRIOR_BUG (rationale=flag_well_specified) |
| 1 failure, no decision-log entry mentions this test | decision_log non-empty, no match | WARRIOR_BUG (rationale=no_decision_log_match) |
| pytest collection error before any test runs | failures=(error,), kind=collection_error | WARRIOR_BUG |
| Decision log present but malformed (no parseable DEFER section) | decision_log_entries=() | INDETERMINATE |
| Multiple failures, only one matches a DEFER entry; others don't | mixed | First-match-wins on candidate; if first match yields SAGE_UNDER_MARKED return that. (Sub-case: ALL must match a DEFER for single-bounce. Mixed -> falls through to WARRIOR_BUG, rationale="multiple_failures_only_partial_match") |
| `pytest.mark.xfail(strict=True)` raised XPASS (the absorption-trap inverse) | failure_kind=xfail_strict_unexpected_pass | SAGE_UNDER_MARKED (rationale="xpass_under_strict") — this is also Sage's bug, opposite direction |

**What the classifier deliberately DOES NOT do (out-of-scope, deterministic-only stance):**

- Does not call any LLM (per ticket "Out of scope: LLM-backed classification").
- Does not infer "Sage probably meant flag X" from test source — only reads the parsed `decision_log_entries` and parsed `flag_definitions`. If Sage's memo doesn't name the missing dep, the rationale is `under_marked_flag` and `suspected_missing_deps` lists every dep referenced in the test body that the flag's initializer doesn't cover (mechanical AST diff, not semantic guess).
- Does not differentiate "near-miss" from "wildly wrong" — any under-specification is SAGE_UNDER_MARKED.
- Does not generalize to non-xfail markers (skipif, skip) — out-of-scope per ticket "Generalizing to other author/reviewer boundaries".

## §D3 — `StageHandler` Protocol Conformance

**Decision:** `SageCorrectionBounceHandler` conforms to the `StageHandler` Protocol at `src/bonfire/protocols.py:188` exactly as the four core handlers do. The conformance shape is **identical** to BON-519's `MergePreflightHandler` precedent (BON-519 §D2 lines 229-294): outer try/except-Exception, never raises, structured `ErrorDetail` on every failure path, prior_results consumption pattern mirrors `wizard.py:279` PR-number extraction.

**Class signature:**

```python
class SageCorrectionBounceHandler:
    """Pipeline stage: deterministic auto-bounce of under-specified xfail markers.

    Runs between Warrior verify and Wizard review. Reads Warrior's failing
    tests + the originating Sage decision log; classifies as
    SAGE_UNDER_MARKED or WARRIOR_BUG; on SAGE_UNDER_MARKED, dispatches a
    Sage-correction subagent whose ONLY edit scope is xfail decorators,
    flag definitions, and the decision log itself; cherry-picks the
    correction commits onto the Warrior branch; re-runs pytest; classifies
    the re-run; green -> continue; still failing -> escalate to Wizard.

    Never raises -- StageHandler Protocol contract (protocols.py:188).
    """

    def __init__(
        self,
        *,
        backend: AgentBackend | None,
        github_client: Any,
        config: PipelineConfig,
        repo_path: Path,
        pytest_command: tuple[str, ...] = ("pytest", "tests/"),
        pytest_timeout_seconds: int | None = 600,
        max_attempts: int = MAX_BOUNCE_ATTEMPTS,
        event_bus: EventBus | None = None,
        settings: BonfireSettings | None = None,
    ) -> None: ...

    async def handle(
        self,
        stage: StageSpec,
        envelope: Envelope,
        prior_results: dict[str, str],
    ) -> Envelope: ...
```

**`handle` flow (pseudocode — mirrors BON-519 §D2 lines 265-292):**

```
try:
    1. Read attempt counter:
       attempts = int(prior_results.get(META_SAGE_BOUNCE_ATTEMPTS, "0"))
       If attempts >= MAX_BOUNCE_ATTEMPTS:
           return envelope.with_metadata(
               sage_bounce_verdict="escalate_max_attempts",
               sage_bounce_attempts=attempts,
           ).with_status(COMPLETED)   # let Wizard handle, don't re-bounce
    2. Read Warrior outcome from prior_results["warrior"] (the result
       string follows the existing TestPassGate-friendly format that
       includes "FAILED" lines parseable by the same regex as
       merge_preflight: see gates.py:20 _NONZERO_FAILED_RE).
    3. If Warrior result indicates GREEN (no FAILED indicators) ->
       return envelope COMPLETED with metadata
       sage_bounce_verdict="not_needed_warrior_green" (early-return).
    4. Resolve original Sage memo path:
         path = prior_results.get(META_ORIGINAL_SAGE_MEMO_PATH)
                or envelope.metadata.get(META_ORIGINAL_SAGE_MEMO_PATH)
       If absent -> envelope.with_metadata(sage_bounce_verdict="indeterminate_no_memo")
                                .with_status(COMPLETED)  # Wizard decides
    5. Parse decision log: parse_decision_log(memo_path) -> entries.
       Parse flag definitions: scan failing-test modules for flag init
       blocks (see §D5 schema).
    6. Parse Warrior failures (parse pytest stdout via the same regex
       merge_preflight uses; reuse if exposed publicly post-BON-519 or
       reimplement locally — D-FT to consolidate).
    7. classifier = Classifier(warrior_failures=..., decision_log_entries=...,
                                flag_definitions=...)
       result = classifier.classify()
    8. Branch on result.verdict:
       SAGE_UNDER_MARKED -> dispatch correction subagent (see §D4),
                            cherry-pick its commits onto Warrior branch,
                            re-run pytest, re-classify;
                            green -> COMPLETED with sage_bounce_verdict="corrected"
                            still failing -> escalate (see §D8)
       WARRIOR_BUG       -> COMPLETED with sage_bounce_verdict="warrior_bug",
                            envelope unchanged (Wizard will reject)
       INDETERMINATE     -> COMPLETED with sage_bounce_verdict="indeterminate",
                            metadata captures rationale; Wizard decides
    9. Always increment attempts counter in returned metadata.
    10. Always return -- never raise.
except Exception as exc:
    return envelope.with_error(
        ErrorDetail(error_type=type(exc).__name__, message=str(exc),
                    stage_name=stage.name)
    )
```

## §D4 — Correction Agent SMEAC Contract

**Decision:** The auto-dispatched correction subagent receives a **handler-authored SMEAC** with hard-locked edit scope. Mirrors the `wizard.py` discipline of "fail-safe body is handler-authored, not agent-authored" (`wizard.py:18-19`, `_render_fail_safe_body` at line 212). The correction agent NEVER edits source code; the SMEAC enforces this with explicit allow-list and deny-list.

**SMEAC builder location:** `src/bonfire/handlers/sage_correction_smeac.py`. Pure function `build_correction_smeac(...) -> str` (no I/O, no LLM, deterministic given inputs). Dispatched via `bonfire.dispatch.runner.execute_with_retry` exactly like `wizard.py:338` does.

**SMEAC inputs (handler passes to builder):**

| Input | Source |
|---|---|
| `classification: ClassificationResult` | from §D2 classifier |
| `original_memo_path: str` | from envelope/prior_results lookup |
| `warrior_branch: str` | the branch carrying RED + Warrior's GREEN attempt |
| `failing_test_path: str` | `classification.candidate_test.file_path` |
| `failing_test_name: str` | `classification.candidate_test.name` |
| `suspected_flag_name: str` | `classification.suspected_flag_name` |
| `suspected_missing_deps: tuple[str, ...]` | what Sage forgot to AND-into the flag |
| `pytest_stdout_tail: str` | last 2KB of Warrior's pytest output (for context) |

**SMEAC sections (the prompt the agent receives):**

```
## Situation
You are correcting an under-specified xfail marker. The Warrior's pytest
revealed a failure that should have been deferred via xfail but wasn't —
because the flag (`{suspected_flag_name}`) checks a smaller set of
dependencies than the test actually requires.

Original Sage memo: {original_memo_path}
Failing test: {failing_test_path}::{failing_test_name}
Suspected missing deps: {suspected_missing_deps_list}

## Mission
Edit the xfail flag definition (and only the xfail flag definition) so it
ALSO checks the missing dependencies. Update the corresponding "DEFER via
xfail" section of the original Sage memo to reflect the corrected scope.
Do NOT touch source code under src/. Do NOT touch test bodies. Do NOT
touch fixtures. Do NOT touch assertions.

## Execution
1. Read the failing test module: {failing_test_path}.
2. Locate the flag definition `{suspected_flag_name}`. The expected shape
   is a `try: from <module> import <symbol[, symbol]>; <FLAG> = True
   except ImportError: <FLAG> = False` block at module top.
3. Widen the import list to ALSO probe these symbols (one of):
     {suspected_missing_deps_compact}
4. Add the corresponding `<other_flag> = True` definitions and AND them
   into the xfail decorator's `condition=not _X_PRESENT` expression so
   `condition = not (_X_PRESENT and _Y_PRESENT)`.
5. Open the original Sage memo at {original_memo_path}. Locate the
   "## DEFER via xfail" subsection that names this test. Replace its
   condition-citation line with the corrected expression. Keep the rest
   of the memo verbatim. Add a one-line append-only correction note:
   `> CORRECTED {iso_timestamp_utc}: widened {flag} to AND
   {missing_deps}.`
6. Commit on the SAME branch (`{warrior_branch}`) with message:
   `bon-513: correct under-specified xfail for
   {failing_test_path}::{failing_test_name}` (no Linear-ID-as-name; the
   ID is fine in commit message body, NOT in src/ — sweep test forbids).

## Administration / Logistics
- Tools allowed: `Read`, `Edit`, `Write` (Write only for the memo if it
  doesn't yet exist — by construction it should exist).
- Tools forbidden: `Bash` (cherry-pick happens in the parent handler),
  any web tool, any source edit under `src/`.
- Working directory: {repo_path}.
- Branch: stay on `{warrior_branch}`. Do NOT create a new branch.
- Cost: budget-capped per stage config (max_budget_usd).
- Time: a single dispatch turn; max_turns=3.

## Constraints (HARD)
- ALLOW-LIST (the ONLY paths you may Edit):
    * `{failing_test_path}` (xfail flag block + decorator only)
    * `{original_memo_path}` (the DEFER section + correction note only)
- DENY-LIST (must NOT Edit, NOT Read with intent to change):
    * Any path under `src/`
    * Any test body (only the flag-definition block + the decorator)
    * Any fixture (`conftest.py`, `tests/fixtures/*`)
    * Any assertion line of any test
- If the corrected flag would still under-specify the test, FAIL the
  task by emitting `<verdict>UNFIXABLE</verdict>` and explain why; do
  NOT make a partial commit.

## Verdict format (REQUIRED)
End your response with one of:
  <verdict>CORRECTED</verdict>     — flag widened, memo updated, committed.
  <verdict>UNFIXABLE</verdict>     — analysis shows under-specification
                                     is wider than xfail can express;
                                     escalate to Wizard.
```

## §D5 — Decision-Log Schema (Classifier Input Format)

**Decision:** The Sage decision-log files under `docs/audit/sage-decisions/bon-XXX-sage-*.md` already follow a conventional structure (see e.g. BON-519 §A-§B-§D pattern). For BON-513 the classifier reads ONE specific subsection convention: the **"DEFER via xfail" block**. We standardize that block's shape now (this memo IS the schema lock) so the parser has a deterministic grammar to chew on, mirroring BON-519 §D9's "memory artifact schema lock" stance (BON-519 §D9 lines 681-688).

**Header conventions (parser landmarks):**

The parser uses these grep-anchors (in priority order, mirroring how `wizard.py:71-79` uses anchored regex):

1. `^### DEFER via xfail` — section start (case-insensitive, flexible whitespace).
2. `^- \*\*Test module:\*\*\s*\`(?P<path>[^\`]+)\`` — module path.
3. `^- \*\*Test name(?: \(.*\))?:\*\*\s*\`(?P<name>[^\`]+)\`` — test name/substring.
4. `^- \*\*Flag:\*\*\s*\`(?P<flag>_[A-Z_]+_PRESENT)\`` — primary flag.
5. `^- \*\*Condition:\*\*\s*\`(?P<expr>[^\`]+)\`` — condition expression.
6. `^- \*\*Referenced symbols.*:\*\*$` followed by `^\s*-\s*\`(?P<sym>[^\`]+)\`` bullets.
7. `^- \*\*Strict:\*\*\s*\`(?P<bool>True|False)\`` — strict flag.
8. `^> CORRECTED (?P<ts>[^:]+):` — corrected-footer detection.

**Fail-safe behavior (parser-level):**

- Missing required field (test module / test name / flag) -> entry skipped, parser logs `WARN decision_log.entry_skipped reason=missing_<field>`. Classifier sees `decision_log_entries=()` for that section and returns `INDETERMINATE` per §D2 step 2.
- Multiple "DEFER via xfail" subsections in one memo are all parsed; classifier scans all of them for the candidate match in §D2 step 4.
- Markdown structural drift (missing bullets, alternate punctuation) -> parser is intentionally strict-but-forgiving on whitespace, strict on the bullet-prefix anchors (`- **Field:**`). This forces Sages to follow the schema; tolerance breeds drift (mirrors BON-519 §D5 "frozen schema" stance, BON-519 §D5 line 506).
- Encoding / UTF-8 BOM tolerated (parser opens with `errors="replace"`).

**Why Markdown and not JSON-in-fenced-block?**

- The decision logs are sellable documentation per `feedback_sellable_documentation.md` (memory ref). Markdown bullets are reviewable by humans; JSON would force a separate doc surface.
- Adding a JSON block alongside is a D-FT (post-v0.1 if drift becomes painful).
- The grammar above is greppable and round-trippable by the correction agent without needing a parser library — it can `Edit` the bullet line by line.

## §D7 — Re-Verify Loop (Cherry-Pick + Re-Run + Re-Classify)

**Decision:** The re-verify loop is a private async helper inside `SageCorrectionBounceHandler` (not a top-level pipeline stage). Mirrors BON-519 §D3's "scratch worktree primitive is a helper of the handler, not its own stage" stance (BON-519 §D3 line 298). The loop runs at most ONCE per pipeline pass (`MAX_BOUNCE_ATTEMPTS = 1`); a second failure escalates rather than re-bouncing.

**Loop pseudocode (inside `handle` after correction-agent verdict CORRECTED):**

```
async def _re_verify_after_correction(
    self,
    *,
    warrior_branch: str,
    correction_branch: str | None,    # may be None if agent committed in-place
    repo_path: Path,
    pytest_command: tuple[str, ...],
    pytest_timeout_seconds: int | None,
) -> tuple[ClassificationResult, str]:   # (re-run classification, pytest_stdout)
    1. If correction_branch is not None and != warrior_branch:
         await _git("cherry-pick", correction_branch, cwd=repo_path)
       (subagent SHOULD have committed on warrior_branch directly per
        SMEAC §Execution step 6; this branch is the safety net for the
        case where the agent created a side branch.)
    2. proc = await asyncio.create_subprocess_exec(
            *pytest_command,
            cwd=repo_path,
            stdout=PIPE, stderr=PIPE,
       )
       try:
           stdout, stderr = await asyncio.wait_for(
               proc.communicate(),
               timeout=pytest_timeout_seconds,
           )
       except asyncio.TimeoutError:
           proc.kill()
           await proc.wait()             # subprocess teardown contract
           raise PytestTimeoutError(...) # caught by handler outer try
    3. Parse stdout into FailingTest tuple (reuse the same parser as
       step 6 in §D3 handle flow).
    4. classifier = Classifier(
           warrior_failures=parsed_failures,
           decision_log_entries=<re-parse the corrected memo>,
           flag_definitions=<re-parse the corrected test module>,
       )
       result = classifier.classify()
    5. Return (result, stdout.decode(errors='replace'))
```

**Subprocess discipline (HARD constraints from project rules):**

- `pytest_command: tuple[str, ...]` — `shell=False` always. (Per Wizard SMEAC constraint and `feedback_no_economic_cuts` no-shortcut law.)
- Subprocess teardown on timeout: `proc.kill()` + `await proc.wait()`. (Mirrors BON-519 §D2 line 294 error-handling stance.)
- Cherry-pick on conflict: catch `subprocess.CalledProcessError` from `_git("cherry-pick", ...)` and `_git("cherry-pick", "--abort")` to leave the branch clean before raising `RuntimeError("cherry-pick conflict")` for the handler outer try to absorb.

**Result branching (mirrors BON-519 §D2 step 11 envelope-build):**

| Re-classify result | Outcome | Envelope shape |
|---|---|---|
| `result.verdict == INDETERMINATE` AND `failures == ()` (i.e. green) | continue pipeline | `with_status(COMPLETED)` + `META_SAGE_BOUNCE_VERDICT="corrected"` + `META_SAGE_BOUNCE_ATTEMPTS=1` |
| `result.verdict == SAGE_UNDER_MARKED` AND `attempts >= MAX_BOUNCE_ATTEMPTS` | escalate | see §D8 |
| `result.verdict == WARRIOR_BUG` after correction | escalate | see §D8 (rationale="warrior_bug_after_correction") |
| Any exception during re-verify | error envelope | `with_error(ErrorDetail(error_type="...", message="...", stage_name=stage.name))` |

**Idempotence + cleanup:**

- Re-verify NEVER mutates anything outside the warrior branch and the pytest cache. The classifier, decision-log parser, and flag-definition parser are pure. The pytest run mutates only the local git index/worktree if test ran (via fixtures); standard pytest isolation applies.
- We do NOT create scratch worktrees here (that's BON-519's tool). We re-run pytest in the existing Warrior worktree — the correction agent already committed the corrected xfail there.
- If a test fixture happens to leave debris (e.g. tmp files), the next pipeline stage's CWD is unchanged. v0.1 accepts the small risk of fixture-leftover noise; cleanup is the test author's job, not the handler's.

**`event_bus.emit` on re-verify (forward-compat hooks):**

The handler emits these events when an event_bus is provided (mirroring `wizard.py:266-269`):

- `SageBounceClassified(verdict, candidate_test, attempts, cost_usd)` — after first classification.
- `SageCorrectionDispatched(memo_path, suspected_flag_name, missing_deps)` — before agent dispatch.
- `SageCorrectionVerdict(verdict_tag, cost_usd)` — after agent verdict parse.
- `SageBounceReVerified(verdict, pytest_returncode, pytest_duration_seconds)` — after re-run pytest.
- `SageBounceEscalated(rationale)` — when escalating to Wizard.

These event types live in `src/bonfire/events/types.py` (assumed extension; if absent in v0.1 they degrade to log-only — the handler `await self._emit(event)` no-ops if `_bus is None`).

## §D8 — Escalation to Wizard

**Decision:** Escalation is **non-fatal** — the envelope returns COMPLETED with structured metadata, allowing the pipeline's existing `bard -> wizard` arc to proceed. Wizard's review prompt (built by `wizard.py:191` `_build_review_prompt`) gets a *new context block* describing the bounce attempt's outcome, so Wizard can factor "this was bounced once and still failed" into its verdict. Mirrors BON-519 §D8 lines 593-598 stance — the gate FAILS, but the envelope itself isn't a poison-pill; downstream stages can read the metadata and decide.

**Three escalation paths:**

| Trigger | `META_SAGE_BOUNCE_VERDICT` value | Pipeline downstream behavior |
|---|---|---|
| `attempts >= MAX_BOUNCE_ATTEMPTS` and re-verify still fails | `"escalate_max_attempts"` | Bard runs (PR is opened so Wizard can see it); Wizard sees `META_SAGE_BOUNCE_VERDICT="escalate_max_attempts"` in envelope.metadata and adds a "bounce-attempt context" block to its review prompt. |
| Correction agent emitted `<verdict>UNFIXABLE</verdict>` | `"escalated"` (rationale="agent_self_unfixable") | Same as above. |
| Re-classifier returned `WARRIOR_BUG` after correction | `"escalated"` (rationale="warrior_bug_after_correction") | Same as above. |
| Re-verify raised an exception (cherry-pick conflict, pytest crash) | envelope `with_error(...)` (FAILED) | Pipeline halts. The standard error envelope flow applies — this is NOT escalation, it's a handler error. |

**Envelope shape on escalation (NOT error):**

```python
envelope.with_metadata(
    sage_bounce_verdict="escalated",      # or "escalate_max_attempts"
    sage_bounce_attempts=str(attempts),
    sage_bounce_classification=json.dumps({
        "verdict": "sage_under_marked",
        "candidate_test_file": "tests/unit/test_wizard_handler.py",
        "candidate_test_name": "test_parse_verdict_extra_meta",
        "suspected_flag_name": "_PARSER_HELPERS_PRESENT",
        "suspected_missing_deps": ["_EXTRA_META_PRESENT"],
        "rationale": "agent_self_unfixable",
        "attempts": 1,
        "pytest_stdout_tail": "<last 2KB>",
    }),
    correction_branch=correction_branch_or_empty_string,
    original_sage_memo_path=memo_path,
).with_status(TaskStatus.COMPLETED).with_result(
    "sage_correction_bounce: escalated to wizard "
    "(verdict=escalated, attempts=1, candidate=test_parse_verdict_extra_meta)"
)
```

**Wizard prompt augmentation (downstream effect — separate D-FT to land in Wizard):**

A *small, additive* edit to `wizard.py:191` `_build_review_prompt` reads the metadata when present and prepends a context section:

This PR was processed by the Sage correction-bounce stage.
- Verdict: {sage_bounce_verdict}
- Attempts: {sage_bounce_attempts}
- Suspected flag: {suspected_flag_name}
- Suspected missing deps: {suspected_missing_deps}

Factor this into your review. If the bounce verdict is "corrected", the

**Sage decision:** the Wizard-prompt augmentation is a **D-FT child of BON-513**, NOT a v0.1 hard requirement. v0.1 ships the bounce stage with metadata; Wizard reads or ignores. Knight integration tests assert metadata is present; they do NOT yet assert Wizard prompt includes the new section. This keeps the change-surface tight per BON-612 discipline.

**Max-attempt counter discipline:**

- `META_SAGE_BOUNCE_ATTEMPTS` is stringified-int (envelope metadata is `dict[str, Any]`; `prior_results: dict[str, str]` forces stringification at boundary).
- Counter is reset to `"0"` on every NEW pipeline pass (the engine constructs fresh prior_results from each stage's result; the counter only persists within one stage's `handle` call across the re-verify loop).
- For multi-pass scenarios (rare; would only happen if a downstream gate bounces back to warrior, which would itself reset Knight/Warrior context), the counter is REPLACED, not accumulated. v0.1 treats each pipeline pass as a fresh bounce budget.

**Why max_attempts=1?**

- The classifier is deterministic. If after one widening the test still fails, either (a) the agent widened the wrong flag, (b) there's a real Warrior bug, or (c) the under-specification is structurally wider than xfail can express. Cases (a) and (c) need human judgment (Wizard); case (b) is by definition not a bounce target. Looping more than once just burns tokens.
- Future ticket can lift this if patterns emerge (D-FT candidate).
