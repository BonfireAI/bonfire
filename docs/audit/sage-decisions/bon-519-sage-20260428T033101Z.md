### Q2 — Stage trigger semantics: when does `MergePreflight` fire?

**Options:**
- **(a) Wizard-approve gate** — fires after the `wizard` stage produces an APPROVE verdict, before `herald` (which merges).
- **(b) Bard PR-creation preview** — fires immediately after `bard` opens the PR.
- **(c) Both** — once after Bard (early signal), once before Herald (final gate).

**Sage proposal: (a) — between `wizard` and `herald`.**

**Reasoning:**
1. The Wizard's review is the most expensive analysis. Running preflight before Wizard wastes Wizard cycles when preflight will block anyway. But — running preflight BEFORE Wizard also gives Wizard cross-wave context, which is valuable.
2. Counter: **(a) is correct** because Wizard reviews the PR diff as code (semantic concerns), not as merged-tip behavior. Merging-tip behavior is the merge stage's concern. Wizard-then-preflight is the right sequence.
3. The ticket's "Trigger: Wizard review approves PR → before merge button" pins this — (a) is ticket-literal.

**Consequences:**
- New `WorkflowPlan` stage between `wizard` and `herald` in `bonfire.workflows.standard.standard_build()`.
- `herald.depends_on = ["merge_preflight"]` (was `["wizard"]`).
- `merge_preflight.depends_on = ["wizard"]`.
- A new gate `merge_preflight_passed` is registered. On failure, halts pipeline (no bounce-back this PR — bounce destinations are BON-513 territory).
- Wizard verdict is read from `prior_results["wizard"]` (verdict metadata via `META_REVIEW_VERDICT`). Preflight skips itself (returns early-COMPLETED) if Wizard's verdict was not `approve`.

### Q4 — Failure classifier algorithm?

**Sage proposal:** Deterministic 4-way classifier. Returns one of:
- `green` — all tests passed
- `pre_existing_debt` — failing tests also fail on `origin/<base>` HEAD (NO PR diff applied)
- `cross_wave_interaction` — failing test files intersect with the file-set of an open sibling PR's diff
- `pure_warrior_bug` — neither of the above; the PR's own diff causes the failure

**Algorithm (in order, first match wins):**

```
1. Parse pytest JUnit XML output (--junit-xml=<path>); extract failing test
   nodes (file path, class, name).
2. If failing-test-file-set is empty -> return green.
3. Compute base_failures: run pytest on origin/<base> HEAD in scratch worktree
   (CACHED across preflight invocations within a session via session_id key).
   If a failing test from step 1 is in base_failures -> classify as pre_existing_debt.
4. Compute sibling_files: union of file-paths from open PRs' diffs targeting
   same base (gh pr list filtered, gh pr diff per PR -> file paths).
5. If any failing test's file (the test file itself) intersects sibling_files
   OR any failing test's *traceback files* intersect sibling_files
   -> classify as cross_wave_interaction.
6. Otherwise -> classify as pure_warrior_bug.
```

**Edge cases:**
- **Empty PR list / API error** — sibling detection degrades gracefully: assume no siblings, classify as `pure_warrior_bug` if step 5 had no data. The classifier records `sibling_detection_status: "skipped" | "ok" | "error"` in the result for forensics.
- **Pytest collection error (no XML emitted)** — return a special classification `pytest_collection_error` with raw stderr. This is a 5th class. Pipeline blocks merge.
- **Parametrize-generated failures** — file path is the same as the parametrize source; classifier handles correctly because it operates on file paths, not test IDs.
- **JUnit XML parser fail-safe** — if XML is malformed, fall back to scanning pytest stdout for `FAILED <path>::` lines via regex. Never fail-open into `green`.

**Return type:**
```python
@dataclass(frozen=True)
class PreflightClassification:
    verdict: Literal["green", "cross_wave_interaction", "pure_warrior_bug",
                     "pre_existing_debt", "pytest_collection_error"]
    failing_tests: tuple[FailingTest, ...]   # parsed from JUnit XML
    sibling_pr_numbers: tuple[int, ...]      # PRs whose files intersected
    sibling_detection_status: Literal["ok", "skipped", "error"]
    pytest_returncode: int
    pytest_duration_seconds: float
    pytest_stdout_tail: str                  # last ~2KB for forensics
```

### Q5 — Sibling-batch detection algorithm?

**Sage proposal:**

```
1. Get current PR's base branch (B) and number (N).
2. List open PRs targeting B: gh pr list -R <repo> --base B --state open --json number,headRefName,files
3. Filter: exclude PR N itself.
4. For each remaining PR M:
     - Read M's changed file set from the JSON (or gh pr view N --json files)
     - Add files to sibling_files dict {pr_number: frozenset[str]}
5. Return sibling_files.
```

**Heuristic refinement (NOT in v0.1 scope, flagged as D-FT in §D-CL.8):** weight by PR age, by reviewer overlap, by Sage-decision-doc cross-references. v0.1 uses raw open-PR list — every open PR against same base is a candidate sibling.

**Authentication:** Reuses `GitHubClient` (existing). Adds a new method `list_open_prs(base: str) -> list[PRSummary]` — see §D5 for signature.

**Mocking:** `MockGitHubClient.list_open_prs` returns canned data; tests parametrize the canned result.

### Q6 — Cross-wave reconciliation lane behavior?

**Sage proposal for v0.1: BLOCK + surface to Wizard.**

When classifier returns `cross_wave_interaction`:
1. The handler returns `TaskStatus.FAILED` with `ErrorDetail(error_type="cross_wave_interaction", message=<human-summary>, stage_name="merge_preflight")`.
2. The envelope metadata carries the full classifier result as JSON in a new metadata key `META_PREFLIGHT_CLASSIFICATION` (added in `src/bonfire/models/envelope.py`).
3. The pipeline halts (no auto-dispatch to Sage reconciliation — that's BON-513).
4. The human-readable error message includes: failing test count, sibling PR numbers, and a one-liner: "Cross-wave interaction detected with PR #M. Run BON-513 reconciliation lane (post-v0.1) or close one PR + re-run."

**For pure_warrior_bug:** Same shape, `error_type="pure_warrior_bug"`, message escalates to Wizard ("preflight blocks merge: pure-Warrior bug; re-run Warrior cycle").

**For pre_existing_debt:** Sage proposal — **ALLOW MERGE WITH ANNOTATION**. The handler returns `TaskStatus.COMPLETED` with metadata `META_PREFLIGHT_TEST_DEBT_NOTED=true` and the result string includes the debt summary. The Herald posts a comment on the PR noting the debt. **Anta to ratify** — alternative is "block always, debt is debt." ### Q7 — Test-cost budget (full-suite pytest ~28s on v0.1)?

**Sage proposal: Acceptable. No timeout cap in v0.1.**

**Reasoning:**
1. Per `feedback_no_economic_cuts.md`, time and tokens are investments.
2. 28s + sibling-PR queries (~5s for `gh pr list` + diff) = ~35s per merge gate. This is below the human-attention threshold and tiny vs the upstream LLM stages (minutes each).
3. **One-time amortization:** the `origin/<base>` baseline pytest is cached per session_id. After the first preflight in a session, base-failure data is reused. The classifier's `pre_existing_debt` step is essentially free for subsequent PRs.
4. The handler accepts a `pytest_timeout_seconds: int | None = 600` constructor kwarg as a safety net (10-minute hard ceiling). On timeout, classifier returns `pytest_collection_error` and pipeline blocks.

## §D2 — `MergePreflightHandler` Class Signature

```python
class MergePreflightHandler:
    """Pipeline stage: full-suite pytest against simulated merged tip.

    Runs between Wizard approve and Herald merge. Creates scratch worktree
    at origin/<base>, applies PR diff (and sibling-PR diffs if siblings
    are open), runs pytest, classifies failures deterministically, blocks
    merge on cross-wave interaction or pure-warrior-bug.

    Never raises -- StageHandler Protocol contract (protocols.py:195).
    """

    def __init__(
        self,
        *,
        github_client: GitHubClient | MockGitHubClient,
        scratch_worktree_factory: ScratchWorktreeFactory,  # see D3
        repo_path: Path,
        base_branch: str = "master",
        pytest_command: tuple[str, ...] = ("pytest", "tests/"),
        pytest_timeout_seconds: int | None = 600,
        sibling_detection: bool = True,
        # session-scoped cache of origin/<base> failures, keyed by base SHA
        baseline_cache: dict[str, frozenset[str]] | None = None,
    ) -> None: ...

    async def handle(
        self,
        stage: StageSpec,
        envelope: Envelope,
        prior_results: dict[str, str],
    ) -> Envelope: ...
```

**`handle` flow (pseudocode):**

```
1. Read PR number from prior_results / envelope.metadata (mirror BardHandler/HeraldHandler extraction).
   On missing -> envelope.with_error(ErrorDetail("ValueError", "No PR number found", ...))
2. Read Wizard verdict from prior_results[META_REVIEW_VERDICT].
   If not "approve" -> envelope.with_result("preflight: skipped (wizard verdict not approve)") COMPLETED early-return.
3. Resolve sibling PRs (Q5 algorithm) -> sibling_pr_numbers, sibling_files.
4. Acquire scratch worktree via factory at origin/<base_branch>.
   Use try/finally to GUARANTEE teardown.
5. Apply PR N diff: gh pr diff N -> patch -> git apply --3way (or merge --no-ff --no-commit if siblings present).
   On apply failure -> classify "merge_conflict" (5th-class, pipeline blocks).
6. (Sibling-batch) If sibling_pr_numbers non-empty: apply each sibling's diff to the same scratch worktree.
   Order: deterministic by PR number ascending. Each apply guarded by try/except RuntimeError.
7. Run pytest in scratch worktree. Capture JUnit XML to <worktree>/preflight-junit.xml.
8. Parse XML -> failing_tests.
9. (cache miss) Run baseline pytest on origin/<base_branch> in a separate scratch worktree (NO diffs applied).
   Cache the failing-test-file-set per base SHA in baseline_cache.
10. Classify (Q4 algorithm) -> PreflightClassification.
11. Build result envelope:
    - green -> COMPLETED, result="preflight: PASSED (N tests, M.MMs)", metadata[META_PREFLIGHT_CLASSIFICATION]=json
    - pre_existing_debt -> COMPLETED with metadata[META_PREFLIGHT_TEST_DEBT_NOTED]=True, result includes debt summary
    - cross_wave_interaction -> FAILED, ErrorDetail("cross_wave_interaction", ...)
    - pure_warrior_bug -> FAILED, ErrorDetail("pure_warrior_bug", ...)
    - pytest_collection_error / merge_conflict -> FAILED, ErrorDetail with type literal
12. Cleanup scratch worktrees (finally block).
13. Return envelope.
```

**Error handling:** ALL exceptions (subprocess, gh CLI, git, pytest, JUnit parser) are caught at the outer `try` (mirroring BardHandler line 254). Returns FAILED envelope with `ErrorDetail`, never raises.

```python
    Fail-safe: returns () if XML is missing or malformed. Caller must
    handle the empty case as MERGE_CONFLICT or PYTEST_COLLECTION_ERROR
    based on the pytest returncode.
    """

def parse_pytest_stdout_fallback(stdout: str) -> tuple[FailingTest, ...]:
    """Regex-scan FAILED lines from pytest stdout when JUnit XML is absent.

    Pattern: r'^FAILED ([^:]+)::([^:]+(?:::[^:]+)?)' (file::class::name)
    """

def classify_pytest_run(
    *,
    pytest_returncode: int,
    pytest_duration_seconds: float,
    pytest_stdout: str,
    failing_tests: tuple[FailingTest, ...],
    sibling_files: dict[int, frozenset[str]],   # {pr_n: file_set}
    baseline_failures: frozenset[str],          # {test_file_paths}
    sibling_detection_status: Literal["ok", "skipped", "error"],
) -> PreflightClassification:
    """Pure deterministic classifier. No I/O, no subprocess, no network.

    Order of checks (FIRST MATCH WINS):
    1. returncode != 0 AND failing_tests is empty -> PYTEST_COLLECTION_ERROR
    2. failing_tests is empty AND returncode == 0 -> GREEN
    3. ALL failing-test file paths are in baseline_failures -> PRE_EXISTING_DEBT
       (NOT 'any' -- ALL. If even one failure is novel, fall through.)
    4. ANY failing-test file path or any traceback_files entry intersects
       union(sibling_files.values()) -> CROSS_WAVE_INTERACTION
    5. Otherwise -> PURE_WARRIOR_BUG
    """
```

**Edge cases (Knight tests must cover):**

| Scenario | Input | Expected verdict |
|---|---|---|
| All tests pass | empty failures, rc=0 | GREEN |
| pytest crashes before collection | empty failures, rc=4 | PYTEST_COLLECTION_ERROR |
| 1 failure, also fails on base | 1 failing in baseline | PRE_EXISTING_DEBT |
| 1 failure, NOT in baseline, file in sibling PR | 1 failing, file ∈ sibling | CROSS_WAVE_INTERACTION |
| 1 failure, NOT in baseline, file NOT in sibling | 1 failing, file ∉ sibling | PURE_WARRIOR_BUG |
| 2 failures: 1 baseline + 1 novel | mixed | falls through to step 4/5 (NOT pre-existing-debt) |
| Sibling detection errored | sibling_status="error" | step 4 yields no intersection -> PURE_WARRIOR_BUG with status flagged |
| Traceback names a file in sibling, but failing test file does not | trace has sibling file | CROSS_WAVE_INTERACTION (per step 4 "or traceback_files") |

```markdown
When a pipeline lacks a deterministic preflight stage, the only signal of cross-wave interaction is post-merge red main. The Wizard then either:
- (a) scrambles to revert + re-coordinate (high cost, audit-trail wobble)
- (b) misses the breakage until the next full-suite run, ships latent bugs
- (c) patches test files directly (the absorption trap)

None are first-class. The right answer is `MergePreflightHandler` -- a deterministic StageHandler that:
1. Creates a scratch worktree at origin/<base>.
2. Applies the PR diff (and any open sibling-PR diffs).
3. Runs full-suite pytest with JUnit XML output.
4. Classifies failures: GREEN / PRE_EXISTING_DEBT / CROSS_WAVE_INTERACTION / PURE_WARRIOR_BUG / PYTEST_COLLECTION_ERROR / MERGE_CONFLICT.
5. Blocks merge unless GREEN or PRE_EXISTING_DEBT (which surfaces a debt annotation).

## How to apply

When designing pipeline stages that gate a destructive operation (merge, deploy, release):
1. The gate MUST be deterministic. No LLM judgment on this gate (LLM classification is a future ticket; deterministic-only ships first).
2. Sibling-batch awareness is required when 2+ PRs target the same base. Apply ALL diffs in the scratch worktree, not just the candidate PR.
3. The classifier MUST distinguish pre-existing debt from PR-introduced regression. Cache the baseline-failure set per base SHA.
4. Cleanup of the scratch worktree is GUARANTEED via try/finally. A cleanup failure NEVER masks the original handler error.
5. The handler NEVER raises (StageHandler Protocol contract). All exceptions become FAILED envelopes with structured ErrorDetail.
```

## §D-CL.7 — superpowers:code-reviewer Focus Areas

(Outside-the-contract concerns — bugs the per-handler-test-suite cannot catch.)

1. **Race conditions in scratch worktree creation** — two concurrent preflights with same `pr_number` and same nanosecond. Verify the 8-hex random suffix is generated per-acquire (not per-handler-instance).
2. **Resource leak on pytest timeout** — if `pytest_timeout_seconds` fires, the subprocess MUST be killed (not just awaited); verify `proc.kill()` is called and `proc.wait()` is awaited before the worktree is torn down.
3. **JUnit XML path traversal safety** — the parser reads `<worktree>/preflight-junit.xml`. If pytest is configured to write the XML to a different path (via existing `pyproject.toml` settings), the parser silently misses failures. **Code reviewer:** verify the handler explicitly passes `--junit-xml=<known-path>` to pytest, overriding any project config.
4. **Sibling-batch ordering determinism** — what if two siblings' diffs touch the same file? Sage decision: ascending PR number wins (later PR's diff takes precedence on conflict via `git apply --3way`). Code reviewer verifies this is documented and tested.
5. **gh CLI auth failure** — if `gh pr list` fails due to auth (403), the handler classifies sibling_status="error". Verify this does NOT leak credentials in the error message.
6. **Envelope size bloat** — `META_PREFLIGHT_CLASSIFICATION` JSON could be large (full failing-tests list). Verify the JSON is bounded (truncate `pytest_stdout_tail` to 2KB; truncate `failing_tests` to 100 entries with overflow flag).
7. **Path-guard discipline** — the handler emits paths in error messages. Verify these are repo-relative (never absolute) per `path_guard.py` discipline.
8. **Subprocess argument injection** — pytest_command and gh args are not user-controlled in v0.1, but verify they are passed as `tuple[str, ...]` to `create_subprocess_exec` (never as a single shell string).
9. **Worktree branch deletion vs remaining checkouts** — `git branch -D` of the ephemeral branch will fail if the worktree is still present. Verify the cleanup order is `worktree remove` THEN `branch -D`, mirroring `WorktreeManager.cleanup`.
10. **Test file path normalization** — JUnit XML reports test file paths as classnames in dotted form on some pytest versions. Verify the parser normalizes to forward-slash file paths (`tests/unit/test_x.py`, not `tests.unit.test_x`).
