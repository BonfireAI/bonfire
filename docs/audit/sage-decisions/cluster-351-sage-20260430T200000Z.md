## B — Convergence map

### B.1 Where all three lenses agree

1. **The four D-FTs collapse to TWO independent axes.** All three scouts
   independently identify that pipeline-asymmetry D-FT and model-resolver
   D-FT are the same code path (`pipeline.py:451`/`456` envelope.model
   write); per-instance settings D-FT and TOML-fallback raise D-FT both
   touch the `BonfireSettings()` fallback in three constructors. This is
   unanimous and load-bearing for the design.

2. **`pipeline.py:456` is dead-weight in pipeline-mode dispatch.**
   - Artisan: "Nothing downstream reads `envelope.model` in pipeline-mode."
   - Psychologist: "the `pipeline.py:451` write is purely cosmetic in
     pipeline-mode dispatch" — cost ledger sources `options.model`.
   - Machinist: "`envelope.model` is set, never read in pipeline-mode
     dispatch. Cost telemetry is sourced from `options.model`."
   Cost ledger path:
   `DispatchOptions.model → runner.py:192 model=options.model →
   DispatchCompleted → DispatchRecord.model → analyzer.model_costs()`.
   `envelope.model` is not on this path.

3. **Composition-root threading beats module-scope memoization for
   settings.** Artisan flags memoization as "anti-aligned with
   testability"; Psychologist cites the FastAPI DI / SPEC-1 lazy-init
   convention; Machinist quantifies test-isolation + fork-safety hazards.
   Path (a) — thread settings through a single composition root — is the
   unanimous recommendation.

4. **The CLI is greenfield for settings threading.** Artisan finds zero
   `BonfireSettings` references in `src/bonfire/cli/`. Psychologist
   confirms "no production composition root" exists (cited from a prior
   Sage memo). Machinist counts `BonfireSettings()` calls in
   `cli/commands/`: zero. **Refactor sets the precedent; doesn't
   retrofit one.**

5. **Resolver perf is irrelevant.** Machinist: ~1–5 µs per call, ~10–50 µs
   per 10-stage pipeline. Optimization is not the motivator for any
   axis-1 fix; correctness + single-source-of-truth are.

## C — Architectural design: the two-axis decomposition

The four D-FTs sort into two orthogonal axes. Treating them as four
independent fixes produces four interfering diffs; treating them as two
coordinated diffs lands all four cleanly.

### C.1 Axis 1 — model-resolver helper (model-resolver D-FT + pipeline-asymmetry D-FT)

**Seam name.** `bonfire.engine.model_resolver` (new module).

**Module path.** `src/bonfire/engine/model_resolver.py`. Public-named
(no leading underscore). Per ADR-001 module-roster discipline, a new
engine submodule is an additive change to the public-tree shape.
Re-export from `bonfire.engine.__init__` is **NOT** required for v0.1
— callers reach the helper via fully-qualified import. The pure
primitive `resolve_model_for_role` continues to live at
`bonfire.agent.tiers` (per `bon-350-sage-20260427T182947Z.md` D-CL.1)
and remains the public-surface primitive.

**Proposed signature (exact types locked per `feedback_sage_lock_types`):**

```python
# src/bonfire/engine/model_resolver.py
from __future__ import annotations
from typing import TYPE_CHECKING

from bonfire.agent.tiers import resolve_model_for_role

if TYPE_CHECKING:
    from bonfire.models.config import BonfireSettings, PipelineConfig


def resolve_dispatch_model(
    *,
    explicit_override: str,
    role: str,
    settings: "BonfireSettings",
    config: "PipelineConfig",
) -> str:
    """Return the model string for a dispatch call site.

    Three-tier precedence (locked by Sage cluster-350 D-CL.1 and
    ratified by cluster-351):

        1. ``explicit_override`` — per-stage / per-envelope escape hatch.
           Empty string falls through to (2).
        2. ``resolve_model_for_role(role, settings)`` — role-based
           routing via ModelTier (returns string from settings.models).
        3. ``config.model`` — pipeline default (PipelineConfig.model).

    Pure synchronous function. Never raises on string input.

    Returns:
        Non-empty model string suitable for ``DispatchOptions.model``.
        Empty string is only returned if (1), (2), and (3) are ALL
        empty — defensive return value preserves today's executor
        contract.
    """
    return (
        explicit_override
        or resolve_model_for_role(role, settings)
        or config.model
    )
```

**Affected call sites (file:line):**

1. `src/bonfire/engine/executor.py:271-282` — replace inline `or` chain
   in `DispatchOptions(model=...)` with `resolve_dispatch_model(...)`.
   `envelope.model` is passed as `explicit_override` (line 273 today),
   preserving the existing executor contract: the envelope built at
   `executor.py:196` (`model=stage.model_override or ""`) carries the
   override; helper resolves the rest.

2. `src/bonfire/engine/pipeline.py:502-513` — replace inline `or` chain
   with `resolve_dispatch_model(explicit_override=spec.model_override,
   role=spec.role, settings=self._settings, config=self._config)`.
   **AND** drop the `pipeline.py:456` envelope.model write — see
   Decision D below for the binding semantic.

3. `src/bonfire/handlers/wizard.py:308-317` — replace inline `or` chain
   in `review_envelope = Envelope(model=...)` with
   `resolve_dispatch_model(explicit_override=stage.model_override or "",
   role=ROLE.value, settings=self._settings, config=self._config)`.
   Note the canonical `ROLE.value` ("reviewer") is preserved at this
   call site per the existing
   `tests/unit/test_wizard_handler.py:586` assertion contract.

**What the helper deliberately does NOT do** (Artisan D.4):

- Does NOT instantiate `BonfireSettings()`. Settings flow in as a
  parameter; the helper is pure.
- Does NOT normalize role strings. `resolve_model_for_role` already
  normalizes (`tiers.py:99`).
- Does NOT cache. Memoization is separable from precedence.

### C.2 Axis 2 — composition-root settings threading (per-instance settings D-FT + TOML-fallback raise D-FT)

**Seam name.** `bonfire.engine.factory` (new module) — a thin
factory function the CLI command (or any external composition root)
calls to build a `BonfireSettings` once per pipeline run.

**Module path.** `src/bonfire/engine/factory.py`. Adopting an engine
submodule (rather than a `cli/factory.py`) keeps the factory close to
the engine it serves and lets test code import it without touching the
CLI surface. Per Machinist G1, this resolves the chicken-and-egg
question (no `bonfire run` command exists yet); the factory is
greenfield-additive.

**Proposed signature (exact types locked):**

```python
# src/bonfire/engine/factory.py
from __future__ import annotations
import logging
from typing import TYPE_CHECKING

if TYPE_CHECKING:
    from bonfire.models.config import BonfireSettings

logger = logging.getLogger(__name__)


def load_settings_or_default() -> "BonfireSettings":
    """Build a ``BonfireSettings`` instance for a pipeline run.

    Reads ``bonfire.toml`` from cwd and ``BONFIRE_*`` env vars per
    pydantic-settings source priority. On a *load* failure (malformed
    TOML, env-var coercion error, validator failure), emits a stderr
    warning and falls back to a defaults-only ``BonfireSettings``
    via ``model_construct`` (bypasses validation; safe for the alpha).

    Caller-facing contract:
        - Returns a fully-constructed ``BonfireSettings``.
        - NEVER raises. Catches ``(ValidationError, TOMLDecodeError,
          OSError)`` and warns to ``logger.warning``.
        - Pass the returned instance into ``PipelineEngine``,
          ``StageExecutor``, ``WizardHandler`` constructors via
          ``settings=`` to avoid the re-load.
    """
    from bonfire.models.config import BonfireSettings
    try:
        return BonfireSettings()
    except Exception as exc:  # noqa: BLE001 — wraps tomllib + pydantic types
        logger.warning(
            "Failed to load BonfireSettings from bonfire.toml/env (%s); "
            "falling back to schema defaults. Pass settings= explicitly "
            "to suppress this warning.",
            exc,
        )
        return BonfireSettings.model_construct()
```

**Affected call sites (file:line):**

1. `src/bonfire/engine/pipeline.py:106-116` — keep
   `settings: BonfireSettings | None = None` parameter; replace the
   silent `_BonfireSettings()` fallback (line 116) with a call to
   `load_settings_or_default()` from the new factory module. Same
   net behavior, but now warning-instrumented.

2. `src/bonfire/engine/executor.py:80-92` — same swap (line 92).

3. `src/bonfire/handlers/wizard.py:256-264` — same swap (line 264).

4. `src/bonfire/cli/commands/` — **NO CHANGES** in this refactor.
   The CLI doesn't yet construct any of these three classes; when a
   `bonfire run` command lands later, it will call
   `load_settings_or_default()` at the top of the command, then
   thread the result into engine + handler constructors.

**Rationale for keeping the optional kwarg.** Per F (risk surface),
40+ existing tests construct without `settings=`. Removing the kwarg
default would force a 40+ test-site diff late in the v0.1 release
train (Decision E).

## D — Public-vs-internal `envelope.model` contract decision

**Decision: `envelope.model` is INTERNAL — a per-call-site record-
keeping field, not a cross-stage public contract.**

**Locked semantic:** "`envelope.model` is the value the call site stamped
on construction; downstream code reads `options.model` (or, equivalently,
`event.model` on `Dispatch*` events) for the model that was actually
dispatched."

**Three-bullet justification:**

1. **Cost telemetry already authoritatively reads `options.model`, not
   `envelope.model`.** The path is `DispatchOptions.model →
   runner.py:109 (DispatchStarted) → runner.py:192 (DispatchCompleted)
   → DispatchRecord.model → analyzer.model_costs()`. All three scouts
   confirmed this independently. There is no observable downstream
   consumer of `envelope.model` in pipeline-mode dispatch today, so the
   line `pipeline.py:456` write is dead weight (Psychologist D-DEAD,
   Artisan A.4, Machinist E).

2. **The Wizard's only "external" read of `review_envelope.model` is a
   self-feed.** `wizard.py:362` reads `review_envelope.model` two lines
   after writing it (line 311-315). That's a local round-trip the
   handler controls, not a contract on envelopes flowing across stages.
   The fail-safe body posted to GitHub renders the model name from a
   value the same handler stamped — it could equally read its own
   resolver result with no observable difference.

3. **`Envelope` is a value-type message between stages, not a snapshot
   of dispatch state.** Stamping the resolved model into the envelope
   in pipeline-mode (path-a "wire-through") would create a new public
   contract — "an envelope's `model` field is the model that was
   dispatched" — that no consumer asks for and that the executor's
   empty-string sentinel pattern (`executor.py:196`) actively
   contradicts. Locking the field as internal aligns the two engines
   on the simpler reading: envelope.model is whatever the call site
   stamped, and **the resolver result lives in `DispatchOptions.model`
   only**.

**Concrete implication for Axis 1:** Drop the `pipeline.py:456`
envelope.model write. Replace `model=spec.model_override or
self._config.model` with `model=spec.model_override or ""`. This
mirrors the executor's contract at `executor.py:196` and removes the
asymmetry surface. The pipeline's options-build at line 502-507
becomes the single source of truth for what gets dispatched.

**Docstring hook (Knight RED test asserts this):** the `Envelope.model`
field docstring (`src/bonfire/models/envelope.py:76`) gains a sentence
clarifying "stamped by the constructing call site; downstream
consumers read `DispatchOptions.model` or `Dispatch*.model` events for
the resolved dispatch model."

## E — Settings constructor doctrine

**Decision: keep `settings: BonfireSettings | None = None` optional;
on `None`, call `load_settings_or_default()` (factory in C.2) which
warns + falls back to `BonfireSettings.model_construct()` on load
failure.**

**Justification against pre-v0.1.0 alpha window per `docs/release-policy.md`:**

1. **Required-arg breaks 90+ existing constructor sites in one diff.**
   Psychologist B counted ~90 test invocations across
   `test_engine_executor.py`, `test_engine_pipeline.py`,
   `test_engine_pipeline_tool_policy.py`, `test_wizard_handler.py`.
   The Wave 9.1 E2E gate is the v0.1.0 blocker (per repo
   `docs/release-policy.md` line 39), and a 90-test refactor late in
   the release train is a test-stability risk worse than the
   silent-default ergonomics it would fix.

2. **Warn-on-fallback aligns with existing repo doctrine.** Per
   Psychologist E (BON-599-pattern), the codebase's existing pattern
   is **loud-fail-soft**: catch, log, proceed with a safe substitute
   (see `wizard.py:_parse_verdict`, `pipeline.py:464-480`,
   `wizard.py:392-405`). Silent raise from a constructor is the
   exception, not the rule. Wrap+warn at the three fallback sites
   restores doctrinal consistency with one factory function.

3. **Pre-v0.1.0 alpha is exactly the window for a deprecation path.**
   The current release-train rules (`docs/release-policy.md`) classify
   pre-release publications as name-reservation only — wheels print a
   notice and exit. Library callers do not yet exist in the wild, so
   the kwarg can stay optional today and tighten to required at v0.2.
   This memo files that v0.2 promotion as a follow-up (see K).

**The two non-doctrinal alternatives explicitly rejected:**

- **Required-arg now.** Breaks 90+ tests; correct architecture, wrong
  release window.
- **Status-quo silent raise.** Inconsistent with repo's loud-fail-soft
  doctrine; users hit a raw `pydantic_core.ValidationError` from a
  constructor they thought was inert.

**Exception types pinned in docstrings.** The constructor docstrings on
`PipelineEngine.__init__`, `StageExecutor.__init__`,
`WizardHandler.__init__` document that with `settings=None` the
factory may emit a warning when loading defaults. The factory
docstring pins the caught types: `ValidationError, TOMLDecodeError,
OSError`. (See Psychologist F1 docstring template; the Knight test
asserts the warning is emitted on a malformed TOML.)
