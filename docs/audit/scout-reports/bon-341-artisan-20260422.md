## 3. Six Question Answers

### Q1 — LanceDB ship-or-defer?

**Recommendation: SHIP.** Gate behind the existing `[knowledge]` extra.

**Evidence:**
- `pyproject.toml:36` already declares `knowledge = ["lancedb>=0.15", "ollama>=0.4"]` — the extra is LIVE in v0.0.0a1. Deferring would regress existing packaging intent.
- LanceDB on PyPI: `0.30.2` current, minimum stable floor `0.15` (already pinned). Healthy release cadence (version history spans 0.1 → 0.30). Python 3.12 compat verified via `pyproject.toml:11` (requires-python = ">=3.12") + LanceDB 0.15+ ships 3.12 wheels.
- Lazy-import architecture (`vault/backend.py:119,135` → `import lancedb`, `import pyarrow`) means `pip install bonfire-ai` with no extras does NOT trigger a LanceDB load. Zero footprint for base users.
- V1 ships both backends side-by-side (`vault/__init__.py:37-46`); the pattern is proven.

**Trade-off:** LanceDB adds ~150MB to the `[knowledge]` install (pyarrow pulls it in). Acceptable for an explicit opt-in extra; unacceptable as a base dep.

**Critical change vs v1:** Flip the default. V1 `vault/__init__.py:18` has `backend: str = "lancedb"`. Public v0.1 must have `backend: str = "memory"` so first-contact users see no import errors when `[knowledge]` isn't installed.

### Q2 — Embeddings: ship in v0.1 or stay lazy/xfailed?

**Recommendation: SHIP ollama_embedder + mock_embedder, default to mock.**

**Evidence:**
- Ollama Python client on PyPI: `0.6.1` current. Pinned at `>=0.4` in the `[knowledge]` extra.
- `vault/ollama_embedder.py:34-39` already handles missing `ollama` gracefully: `ImportError` at `__init__` with helpful message (`"Install it with: pip install ollama"`). No import-time failure.
- `vault/mock_embedder.py` has zero deps. Deterministic. Safe default.
- `vault/embeddings.py:23` already defaults `provider="mock"` — keep this default.

**No xfail needed.** Mock backend exercises the embedder contract fully. Integration tests for Ollama live behind `@pytest.mark.live` (already declared in `pyproject.toml:58`).

### Q3 — Minimum surface to flip xfails in `test_architect_handler.py`?

**Hard requirement:** All three modules must land for full green — `bonfire.knowledge.memory` (`_VAULT_XFAIL`), `bonfire.knowledge.chunker` (`_CHUNKER_XFAIL`), `bonfire.knowledge.scanner` (`_SCANNER_XFAIL`), plus the handler's lazy-import path. Partial ports will leave some xfails stuck.

**xfail decorator usage count (from `grep -c`):**
- `@_HANDLER_XFAIL`: 19 uses
- `@_VAULT_XFAIL`: 4 uses
- `@_CHUNKER_XFAIL`: 6 uses
- `@_SCANNER_XFAIL`: 13 uses
- Total decorator applications: 42 (some tests carry multiple stacked xfails).

**Distinct tests that flip when modules land:**
- Tests with a single xfail decorator flip independently. Tests with stacked decorators (e.g. `@_HANDLER_XFAIL + @_SCANNER_XFAIL + @_CHUNKER_XFAIL + @_VAULT_XFAIL` on `test_returns_completed_envelope_with_json_summary` at line 436-440) need ALL conditions satisfied.
- `_HANDLER_XFAIL` condition is `not _HANDLER_PRESENT`. `ArchitectHandler` is ALREADY present (`handlers/architect.py` exists). So `_HANDLER_PRESENT = True` already and that decorator is a no-op in current runs — xpass, not xfail.
- Baseline report: 2833 passed / 45 xfailed / 17 xpassed. The 17 xpassed likely include handler tests whose `_HANDLER_XFAIL` evaluates False (handler present) but stacked conditions also eval False.

**Smallest footprint to flip all xfails tied to BON-341:**
1. `knowledge/memory.py` (50 LOC) → flips 4 `@_VAULT_XFAIL` tests.
2. `knowledge/chunker.py` (190 LOC) + `knowledge/hasher.py` (31 LOC) → flips 6 `@_CHUNKER_XFAIL` tests.
3. `knowledge/scanner.py` (237 LOC) → flips 13 `@_SCANNER_XFAIL` tests.
4. Update `handlers/architect.py:80-82` import paths → handler behavior tests (currently xpassing as handler-present but failing inside because `bonfire.vault.*` doesn't exist) will go fully green.

**S007 warning:** Under-specified xfails caused a Session 007 reconciliation cycle. Sage MUST lock the expected test count delta in the contract block. Suggested metric: "After BON-341 lands, `pytest tests/unit/test_architect_handler.py` should report N passed / 0 xfailed / 0 xpassed, where N ≈ 42 distinct test functions."

### Q4 — Does `fingerprinter.py` belong in scan/?

**Recommendation: YES, port + rename to `scan/tech_scanner.py`.**

**Evidence of active usage in v1:**
- `bonfire/front_door/scanners/project_structure.py:18`: `from bonfire.scanners.fingerprinter import TechFingerprinter` — wraps fingerprinter as streaming onboarding scanner.
- Linear spec names "TechScanner" (not TechFingerprinter) — ADR-001 line 61 explicitly renames `TechFingerprinter → TechScanner` because class name should match module name. So the "TechScanner" in the Linear spec IS this file after rename.
- Dedicated v1 test: `tests/unit/test_fingerprinter_innovative.py` — not architectural debris.

**Naming collision warning for Sage:**
- V1 `vault/scanner.py` contains `ProjectScanner`. Public destination: `knowledge/scanner.py`.
- V1 `scanners/fingerprinter.py` contains `TechFingerprinter → TechScanner`. Public destination: `scan/tech_scanner.py`.
- Three "scanner" concepts coexist: `knowledge.scanner.ProjectScanner` (AST-level file scanner), `scan.tech_scanner.TechScanner` (dependency/language detection), and a future `onboard/scanners/*.py` (user-facing streaming scanners). Sage should NOT try to merge these. They're separate.

**File rename rationale:** Keeping `scan/fingerprinter.py` with class `TechScanner` inside would create mismatch between file stem and class name. ADR-001's "Matches module name" rationale drives both the class rename AND the file rename to `tech_scanner.py`.

**Port plan:**

**Test impact:**
