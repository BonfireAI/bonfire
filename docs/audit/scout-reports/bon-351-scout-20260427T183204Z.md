## B — Cost analyzer

### `src/bonfire/cost/models.py` — `DispatchRecord`
Defined at line 14-22:
```python
class DispatchRecord(BaseModel):
    type: Literal["dispatch"] = "dispatch"
    timestamp: float
    session_id: str
    agent_name: str
    cost_usd: float
    duration_seconds: float
```
**No `model` field. No `tier` field. No `role` field.** Records key on `(timestamp, session_id, agent_name)`; agent_name is gamified-role-ish but historically was just the envelope's `agent_name` string (see `cost/consumer.py:33`).

`PipelineRecord` at line 25-32: aggregate per session (`total_cost_usd`, `stages_completed`). Same — no model/tier breakdown.

`SessionCost` at line 36-49: contains a `dispatches: list[DispatchRecord]` (line 42), so any per-model split would need to live on `DispatchRecord` first.

`AgentCost` at line 52-58: aggregates by `agent_name`, NOT by model.

`DEFAULT_LEDGER_PATH = Path.home() / ".bonfire" / "cost" / "cost_ledger.jsonl"` (line 11).

### `src/bonfire/cost/consumer.py` — `CostLedgerConsumer`
Subscribes (line 24-27) to `DispatchCompleted` and `PipelineCompleted` only. Builds a `DispatchRecord` from the event at line 30-37 — currently passes `timestamp`, `session_id`, `agent_name`, `cost_usd`, `duration_seconds`. **`DispatchCompleted.model` does NOT exist; `model` is only on `DispatchStarted` (events.py:127-130).** A per-model cost slice today would have to correlate `DispatchStarted` with `DispatchCompleted` by session+agent — they have no shared dispatch ID.

Append mode: `open("a", encoding="utf-8")` line 51 — JSONL append-only. Schema migration cost: zero — adding a nullable `model` field on `DispatchRecord` is backward-compatible (existing rows would parse with default). But `DispatchCompleted` event would also need a `model` field to feed the consumer.

### `src/bonfire/cost/analyzer.py` — `CostAnalyzer`
Five public methods:
- `cumulative_cost() -> float` (line 65) — sums `PipelineRecord.total_cost_usd`
- `session_cost(session_id) -> SessionCost | None` (line 70)
- `agent_costs() -> list[AgentCost]` (line 114) — **groups by `agent_name`**, not by model
- `all_sessions() -> list[SessionCost]` (line 138)
- `all_records() -> list[DispatchRecord | PipelineRecord]` (line 195)

There is **no method that aggregates per-model**. Adding `model_costs()` would be additive. `_read_records()` (line 27) tolerates malformed lines, unknown types, missing required fields — so adding fields is safe.

### "Per-model-tier spend" implication
For BON-351's "Cost tracking reports per-model-tier spend":
1. `DispatchCompleted` event must carry `model` (today only `DispatchStarted` does).
2. `DispatchRecord` must gain `model: str` (and likely `tier: str` if BON-350 introduces tiers).
3. `CostAnalyzer` must gain a per-model aggregator (e.g. `model_costs() -> list[ModelCost]`).
4. `consumer._on_dispatch_completed` must read the new event field.

Schema is forward-friendly (Pydantic permissive; analyzer's `_read_records` skips unknown types but accepts unknown fields per default Pydantic behavior — verify in Sage).

## C — Envelope + role flow

### `src/bonfire/models/envelope.py` — `Envelope`
Frozen Pydantic at line 64-162. Relevant fields:
- `agent_name: str = ""` (line 75) — gamified-role-ish, set by callers

No `role` field on the envelope itself. The role lives in `metadata["role"]` for the wizard handler (`handlers/wizard.py:307`: `metadata={"role": ROLE.value}`).

### Role visibility at dispatch time
The **role IS visible at dispatch time** via `DispatchOptions.role` (locked by BON-337 D4). The wiring path:
- `models/plan.py:46-63` — `StageSpec` has `role: str = ""` (line 53) and `model_override: str | None` (line 60).
- `engine/executor.py:259-281` — `_dispatch_backend` (line 259) builds `DispatchOptions` with `role=stage.role` (line 271) and `model=envelope.model or self._config.model` (line 266).
- `engine/pipeline.py:497-504` — pipeline-mode dispatch builds `DispatchOptions` with `role=spec.role` (line 503) and `model=spec.model_override or self._config.model` (line 498).
- `handlers/wizard.py:315-322` — wizard handler builds `DispatchOptions` but **DOES NOT set `role=`** (line 315-322; `role` defaults to `""`). This is the only call site where role is currently invisible to the backend. **Coupling note for BON-351:** if the wizard role tier should drive its model, this site needs the role string passed in.

Source of truth for the model today (precedence order, observed at executor.py:266 / pipeline.py:498):
1. `envelope.model` / `spec.model_override` (per-stage override)
2. `self._config.model` (PipelineConfig — `models/config.py:43` defaults to `"claude-sonnet-4-6"`)
3. `""` empty fallback (the SDK then picks its own default)

No tier-aware resolution today. `PipelineConfig.tier: str = "free"` exists at `models/config.py:42` but is **never read** by dispatch (it's a placeholder mirroring `TierGate`).
