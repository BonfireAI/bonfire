## TL;DR

The ticket as written assumes commands that don't exist on `v0.1`. The CLI surface today is `init / scan / status / resume / handoff / persona / cost`. There is **no `bonfire dispatch`**, and `bonfire scan` is a blocking browser-WebSocket flow — it cannot be smoke-tested headlessly without surgery to the command. The realistic v0.1.0 smoke gate is therefore narrower than the ticket text suggests, and three of the six ticket bullets fold into a "scaffold-now, fill-when-the-CLI-lands" pattern.

The recommended plan ships in two forms:

1. **`tests/smoke/smoke_test.py`** — a standalone Python script (NOT in `tests/` proper, runs in an isolated venv against a built wheel, uses zero pytest scaffolding). Pattern lifted directly from `uv` and `ruff`. Validates: wheel installs, console script `bonfire` is on PATH, `bonfire --version` and `bonfire --help` exit 0, and the package surface imports cleanly.
2. **`.github/workflows/smoke.yml`** — a separate workflow (not folded into `ci.yml`) that runs `python -m build`, then runs the smoke script against both `dist/*.whl` and `dist/*.tar.gz`.

The "real provider key" leg of the ticket is **deferred** to a manual tag-time gate, run from Anta's box (`tests/e2e/scripts/e2e-box.sh`), per `docs/release-gates.md` §"E2E box · Local execution only · Never runs in CI." This is consistent with the existing release-gate doctrine, which is older than this ticket and explicitly forbids CI from holding the API key.

D-FTs file the gaps for v0.1.1.

## 1. Research findings

Four reference patterns from the wider ecosystem.

### 1.1 `uv` (astral-sh) — the modern canonical pattern

Source: [docs.astral.sh/uv/guides/package/ — Building and publishing](https://docs.astral.sh/uv/guides/package/) and [docs.astral.sh/uv/guides/integration/github/](https://docs.astral.sh/uv/guides/integration/github/).

The recommended uv release workflow runs a pair of smoke steps **after** `uv build` and **before** `uv publish`:

```yaml
- name: Smoke test (wheel)
  run: uv run --isolated --no-project --with dist/*.whl tests/smoke_test.py

- name: Smoke test (source distribution)
  run: uv run --isolated --no-project --with dist/*.tar.gz tests/smoke_test.py
```

`--isolated --no-project` forces a fresh venv with **only** the built artifact — guarantees you're testing the wheel, not the editable source tree. This catches packaging-data bugs (missing `include = ["src/bonfire/onboard/ui.html"]`-class problems, missing entry-point wiring, missed namespace packages) that `pip install -e .` will silently mask.

`smoke_test.py` itself is intentionally minimal: import the package, call the public API, exit 0. **Not** a pytest run.

### 1.2 `ruff` (astral-sh) — wheel test pattern in CI

Source: [github.com/astral-sh/ruff `.github/workflows/ci.yaml`](https://github.com/astral-sh/ruff/blob/main/.github/workflows/ci.yaml) (verified via WebFetch).

```bash
pip install --force-reinstall --find-links dist "${PACKAGE_NAME}"
ruff --help
python -m ruff --help
```

Three validations: install from local wheel, console script works (`ruff --help`), module entry works (`python -m ruff --help`). The `--force-reinstall --find-links dist` shape is robust against prior cached wheels — important when a CI runner caches pip across jobs.

### 1.3 Hynek `build-and-inspect-python-package` — packaging-lint

Source: [github.com/hynek/build-and-inspect-python-package](https://github.com/hynek/build-and-inspect-python-package) and the [GitHub Marketplace listing](https://github.com/marketplace/actions/build-and-inspect-a-python-package).

Two complementary checks the action runs:

- `check-wheel-contents` — fails if the wheel has missing `__init__.py`, byte-compiled stragglers, or weird permissions.
- `twine check dist/*` — fails if the README won't render on PyPI or the long-description metadata is malformed.

Hatchling's wheel build is not bug-free; running these once before first PyPI publish is cheap insurance. We adopt the action wholesale for the build step — no custom logic.

### 1.4 Scientific Python Development Guide — pure-Python wheel CI

Source: [learn.scientific-python.org/development/guides/gha-pure/](https://learn.scientific-python.org/development/guides/gha-pure/).

Their recommended shape: separate `build` job uploads `dist/` as an artifact; separate `test-installed` job downloads it, installs from `--find-links dist/` into a clean Python, runs the test suite against the *installed* package (not editable). This is heavier than uv's smoke-script pattern but catches a different bug class: tests that pass against `pip install -e .` but fail against the wheel because of a missing data file or a package-namespace gap.

For BON-354 we do **not** need the full reinstalled-test job (the test count is 3508 — too slow for every PR, and `tests/` already runs editable in `ci.yml`). We borrow only the `actions/upload-artifact` + `actions/download-artifact` shape, so the smoke job operates on the same built wheel rather than rebuilding.

### 1.5 PyPI publishing best practice — Trusted Publisher + TestPyPI staging

Sources: [pypi.org Trusted Publishers docs](https://docs.pypi.org/trusted-publishers/using-a-publisher/) and [packaging.python.org publishing-package-distribution-releases-using-github-actions-ci-cd-workflows](https://packaging.python.org/en/latest/guides/publishing-package-distribution-releases-using-github-actions-ci-cd-workflows/).

For a first-tag release the discipline is:

1. Build wheel + sdist on tag.
2. Upload to **TestPyPI** first via `pypa/gh-action-pypi-publish` with `repository-url: https://test.pypi.org/legacy/`.
3. Smoke-install from TestPyPI in a fresh runner: `pip install --index-url https://test.pypi.org/simple/ --extra-index-url https://pypi.org/simple/ bonfire-ai==0.1.0rc1`. Run smoke script.
4. Only on green — promote to real PyPI.

Trusted Publishers (OIDC) eliminates the long-lived API token. This is BON-355's lane (release-publish workflow), not BON-354's, but the smoke test must be re-runnable from a TestPyPI install to support that flow when it lands. Our `smoke_test.py` accepts `bonfire` from PATH — whichever venv it runs in — so it works for both `pip install dist/*.whl` and `pip install --index-url ... bonfire-ai`.

## 2. Recommended plan

Two files ship. One Warrior, ~30-45 minutes.

### 2.1 New file: `tests/smoke/__init__.py`

Empty (so the directory is importable; `tests/smoke/smoke_test.py` is run as a script, not collected by pytest).

### 2.2 New file: `tests/smoke/smoke_test.py`

A standalone Python script. **Runs in a clean venv with only the built wheel installed.** Uses stdlib only — no pytest, no fixtures.

Verification list (each is a single assertion or `subprocess.check_call`):

1. **Package imports.** `import bonfire` → version string non-empty, equals `bonfire.__version__`.
2. **Public protocols import.** `from bonfire.protocols import AgentBackend, VaultBackend, QualityGate, StageHandler, DispatchOptions, VaultEntry` — six names, no `ImportError`.
3. **Engine imports.** `from bonfire.engine import PipelineEngine, PipelineResult` — composition root surface.
4. **Console script is on PATH.** `shutil.which("bonfire")` returns a non-`None` path inside the venv's `bin/`.
5. **`bonfire --version` exits 0** and prints a string that contains `bonfire ` and a digit.
6. **`bonfire --help` exits 0** and lists each top-level command currently registered (`init`, `scan`, `status`, `resume`, `handoff`, `persona`, `cost`). Test assertion is "every command name appears in stdout" — it tolerates new commands being added.
7. **`bonfire init <tmpdir>` succeeds.** Exits 0. `<tmpdir>/bonfire.toml` exists. `<tmpdir>/.bonfire/` exists. (This is `bonfire init`, NOT `bonfire scan`. See §3.2.)
8. **Packaged data files are present.** `importlib.resources.files("bonfire.onboard").joinpath("ui.html").is_file()` returns `True`. (Without this, the wheel hatchling config in `pyproject.toml:54` is broken and `bonfire scan` will crash at runtime — caught here at smoke time.)
9. **Cleanup.** Tempdir is `tempfile.TemporaryDirectory()` — auto-removed.

The script exits 0 on full pass, prints a one-line `OK` summary; any assertion fires `sys.exit(1)` with a one-line `FAIL: <reason>`.

### 2.3 New file: `.github/workflows/smoke.yml`

Separate workflow. Triggers identical to `ci.yml` (`push: main`, `pull_request: main`). Single job:

```yaml
name: Smoke
on:
  push: { branches: [main, v0.1] }
  pull_request: { branches: [main, v0.1] }

jobs:
  smoke:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: { python-version: "3.12" }

      - name: Install build frontend
        run: python -m pip install --upgrade pip build

      - name: Build wheel + sdist
        run: python -m build

      - name: Inspect built artifacts (twine check)
        run: |
          python -m pip install twine
          twine check dist/*

      - name: Smoke (wheel) — clean venv
        run: |
          python -m venv .smoke-wheel
          .smoke-wheel/bin/pip install --upgrade pip
          .smoke-wheel/bin/pip install dist/*.whl
          .smoke-wheel/bin/python tests/smoke/smoke_test.py

      - name: Smoke (sdist) — clean venv
        run: |
          python -m venv .smoke-sdist
          .smoke-sdist/bin/pip install --upgrade pip
          .smoke-sdist/bin/pip install dist/*.tar.gz
          .smoke-sdist/bin/python tests/smoke/smoke_test.py

      - name: Upload built artifacts
        uses: actions/upload-artifact@v4
        with:
          name: dist
          path: dist/*
```

Notes:

- We do **not** add this to `ci.yml`. Reason: `ci.yml` runs `pip install -e ".[dev]"` and `pytest tests/` — the editable install masks every bug the smoke script exists to catch. Mixing the two in one job poisons the venv. Two workflows is the cleaner separation.
- We use `python -m build` (the PEP 517 frontend), not `hatch build`. Same wheel either way per the [PyPA discussion](https://github.com/pypa/hatch/discussions/1058); `build` has no extra runtime dep beyond `build` itself. Reproducibility is identical because the backend (`hatchling`, declared in `pyproject.toml:2`) is the same.
- Artifacts upload last so a green smoke run leaves a downloadable `dist/` for inspection. Useful when BON-355 wires the publish workflow.

### 2.4 No changes to `pyproject.toml`

Out of scope per the brief. BON-355 owns metadata edits.

### 2.5 Manual real-key gate — deferred to existing `tests/e2e/`

The ticket's bullet 5 ("`bonfire dispatch` with real provider key produces a working pipeline run") is satisfied by the **existing** local E2E box scaffold under `tests/e2e/`, governed by `docs/release-gates.md` §E2E box. That scaffold:

- Lives at `tests/e2e/scripts/e2e-box.sh` (per `docs/release-gates.md:48`).
- Runs locally on Anta's machine with `ANTHROPIC_API_KEY` from gitignored `.env` (per `release-gates.md:38, 95-99`).
- **Never runs in CI** (per `release-gates.md:36`).
- Is tracked by BON-424/425/429 (per `docs/release-gate-tickets.md:21-26`) — those tickets are the proper home for "real-key smoke" work, not BON-354.

BON-354's CI-level smoke does not include the real-key dispatch leg. Surfaced to Anta in §3.1.

## 3. Decision points (with recommendations)

### 3.1 Real-key vs mocked-dispatch for the smoke test

**Pick: Mocked-only in CI; real-key gate runs locally via `tests/e2e/` box (already scoped under BON-424).**

Rationale (in order of weight):

1. **`docs/release-gates.md:36-38` is older than this ticket and explicitly forbids CI from holding the API key.** Anta's note: *"API key stays on Anta's machine. Cost is observable in real time. No unattended runs draining the key."* This is policy, not preference — the BON-354 ticket text was written before, or in tension with, the release-gate doctrine. Honoring the doctrine is the cheaper repair.
2. **Cost / DOS posture.** A real-key smoke that fires on every PR is a DOS vector the day a contributor pushes 50 commits to one PR. Even at $0.10/run, a forgotten draft PR with a CI loop costs more than the test catches.
3. **The release-gate scaffold already exists for the real-key job.** Doubling it in CI is duplicate infrastructure with worse governance.
4. **Mocked CI smoke catches the bug class BON-354 actually targets:** packaging drift, wheel data files, console-script wiring, naming-vocab drift in the CLI surface. None of these need a real model call.

The "ticket promise" for real-key is met by the local box (`tests/e2e/scripts/e2e-box.sh`, BON-424/425/429), not by `.github/workflows/smoke.yml`. This split is documented in §7 D-FTs and surfaced to Anta in §4.

### 3.2 Target repo for `bonfire scan`

**Pick: Skip `bonfire scan` from the smoke test entirely; substitute `bonfire init <tmpdir>`.**

Rationale:

- `bonfire scan` (per `src/bonfire/cli/commands/scan.py:18-43`) is **interactive-only**: it starts a WebSocket server, awaits a browser connection (`server.client_connected.wait()`), and blocks until a human disconnects. There is no `--non-interactive` or `--scan-only` flag.
- A headless smoke test cannot drive `scan` without either:
  - patching the command to add a non-interactive mode (out of BON-354 scope; that's a behavior change), or
  - launching a headless browser (Playwright/headless Chrome) and scripting the WebSocket flow (massive cost; not justified for a smoke gate), or
  - mocking the websocket client (also out of scope; would mock the very thing the test claims to verify).
- `bonfire init` provides the "CLI scaffolds a project on disk" signal at zero cost. It exercises the same Typer wiring, it writes `bonfire.toml`, it creates `.bonfire/`. If `init` is wired correctly, the CLI loader is healthy.
- The "real `bonfire scan` works" assertion is delivered by the `importlib.resources` check on `ui.html` (smoke step 8) — without that data file, `scan` will crash on first start.

D-FT: a non-interactive `bonfire scan --non-interactive --target <path>` mode that emits VaultEntry records to stdout and exits is independently useful (and would unlock real CI smoke coverage of the scanner). Filed in §7.

### 3.3 Test framework

**Pick: stdlib script (`tests/smoke/smoke_test.py`), NOT pytest.**

Rationale:

- The smoke script runs in a venv that contains **only** `bonfire-ai` plus its install dependencies. Adding pytest to that venv defeats the purpose — you'd be testing whether `bonfire-ai + pytest + coverage + ruff + ...` co-installs, not whether `bonfire-ai` ships clean.
- Stdlib `subprocess` + `assert` + `sys.exit(1)` is sufficient and reads cleanly. ~80 lines.
- Same pattern uv and ruff use ([uv smoke_test.py](https://docs.astral.sh/uv/guides/package/), [ruff ci.yaml](https://github.com/astral-sh/ruff/blob/main/.github/workflows/ci.yaml)).
- pytest in `tests/unit/` and `tests/integration/` continues to be the editable-install test suite. Two test surfaces, each with its job. No overlap.

### 3.4 CI shape

**Pick: separate `.github/workflows/smoke.yml` workflow. Do NOT extend `ci.yml`.**

Rationale: see §2.3 note. Two reasons:

- `ci.yml` uses `pip install -e ".[dev]"` (`ci.yml:27`). The whole point of smoke is to install the **wheel**, not the editable tree. Different installation modes in one workflow is a debugging nightmare.
- Smoke and pytest report different things; mixing their pass/fail signal makes a red dot on a PR ambiguous. PR badges are clearest when each workflow has one job to do.

No `Makefile` target, no `tox` env. Local-dev smoke is `python -m build && python tests/smoke/smoke_test.py` — three lines a contributor can copy from the workflow file. Documented in CONTRIBUTING.md update in BON-355's lane (or as a tiny D-FT, see §7).

### 3.5 Fixture cleanup

**Pick: `tempfile.TemporaryDirectory()` in `smoke_test.py`; venvs go in repo root with names `.smoke-wheel/` and `.smoke-sdist/` and are added to `.gitignore`.**

Rationale:

- GitHub runners are ephemeral; nothing survives the job anyway. Cleanup in CI is cosmetic.
- For local-dev runs, the named venvs (`.smoke-wheel/`, `.smoke-sdist/`) are easy to nuke (`rm -rf .smoke-*`). They land at repo root rather than `/tmp/` because `/tmp/` has its own discipline (per Anta's `feedback_tmpfs_artifact_loss.md`).
- `tempfile.TemporaryDirectory()` for the `bonfire init <tmpdir>` step inside the script auto-cleans on script exit, even on failure.
- `.gitignore` already has `.venv/`. Add two lines: `.smoke-wheel/` and `.smoke-sdist/`.

## 5. Acceptance criteria for the Warrior

What "smoke test passes" looks like — pass/fail signals + timing budgets.

### Pass signals (all must hold, in order)

1. **`python -m build` exits 0.** Produces `dist/bonfire_ai-<version>-py3-none-any.whl` and `dist/bonfire_ai-<version>.tar.gz`. Cycle time: < 30 seconds on a 4-core runner.
2. **`twine check dist/*` exits 0.** No warnings about README rendering. Cycle time: < 5 seconds.
3. **Wheel install in clean venv exits 0.** `pip install dist/*.whl`. The console script `bonfire` lands in `.smoke-wheel/bin/`. Cycle time: < 60 seconds (most of it is dependency install — `claude-agent-sdk`, `pydantic`, etc.).
4. **`tests/smoke/smoke_test.py` exits 0.** All 9 verifications pass. Single `OK` line on stdout. Cycle time: < 10 seconds (Typer command dispatches are fast).
5. **Same against sdist.** `pip install dist/*.tar.gz` → `python tests/smoke/smoke_test.py` → exit 0.
6. **Total wall-clock: < 5 minutes** from `actions/checkout` to `actions/upload-artifact`. Anything longer is a regression and Wizard flags it.

### Fail signals (any one fails the smoke gate)

- `pip install dist/*.whl` errors out → wheel-build is broken, hatchling config drift.
- `shutil.which("bonfire")` returns `None` → `[project.scripts]` block is broken.
- `bonfire --version` non-zero → CLI composition root crashes on import.
- `bonfire --help` missing a registered command name → Typer registration drift (e.g., a new command added without `app.command(...)` registration).
- `importlib.resources.files("bonfire.onboard").joinpath("ui.html")` fails → `[tool.hatch.build.targets.wheel]` `include` block dropped a data file.
- `bonfire init <tmpdir>` non-zero → `init` command logic regression, OR a new dependency isn't in `dependencies = [...]`.
- `twine check` warnings about long-description content type → `pyproject.toml` README config broke (likely a markdown-vs-rst regression).
