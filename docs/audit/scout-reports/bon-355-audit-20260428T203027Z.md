### Item 1 — `pyproject.toml::version`

**Status: FIX-NEEDED (DECISION already made by `docs/release-policy.md`).**

**Current state.** `pyproject.toml:7` declares `version = "0.0.0a1"`.
`src/bonfire/__init__.py:13` declares `__version__ = "0.1.0"`. **These two
disagree today** — the source says 0.1.0, the package metadata says 0.0.0a1.
Whichever string `python -m build` produces will come from `pyproject.toml`
(hatchling reads pyproject), so a wheel built against this tree right now
would carry the alpha string while shipping `__version__ = "0.1.0"` at
runtime. That divergence is the single most embarrassing release-blocker
on this list and must be fixed in lockstep with the version bump.

**Recommendation: bump `pyproject.toml::version` to `"0.1.0"`** (not an
alpha or beta tag), and bump the `Development Status` classifier from
`3 - Alpha` to `4 - Beta` — exactly as `docs/release-policy.md:36-51`
prescribes. The release policy file is the authoritative spec for this
question; it explicitly states:

> `docs/release-policy.md:47` — "Version becomes `0.1.0`."
> `docs/release-policy.md:49-50` — "The PyPI classifier advances from
> `Development Status :: 3 - Alpha` to `Development Status :: 4 - Beta`."

That settles the three options laid out in the SMEAC:

- `0.1.0a1` (alpha) is **rejected** — release policy explicitly says we
  leave alpha behind at 0.1.0. Continuing to ship alpha when the wave-9
  trigger has fired contradicts the policy that the project itself
  authored.
- `0.1.0b1` (beta) is **rejected** — release policy specifies the
  *classifier* moves to Beta but the *version string* moves to plain
  `0.1.0`. PEP 440 beta numbering would force `pip install bonfire-ai`
  (without `--pre`) to ignore the release, which defeats the purpose of
  cutting a public 0.1.0.
- `0.1.0` (the policy's pick) is **accepted**. The classifier carries the
  beta semantics; the version string carries the public-stable semantics.
  This is the standard PEP 440 / packaging idiom and matches the policy.

**Fix manifest entry.** Single Wizard commit that updates `pyproject.toml:7`
to `version = "0.1.0"`, *and* item 4 below (classifier line 14), *and*
item 8 below (`__version__` sync), all in one atomic change.

### Item 8 — `src/bonfire/__init__.py::__version__`

**Status: FIX-NEEDED (sync mechanism missing).**

`src/bonfire/__init__.py:13` declares:

```python
__version__ = "0.1.0"
```

The symbol exists. **However, it is hard-coded** rather than read via
`importlib.metadata.version("bonfire-ai")`, which means there are now two
sources of truth for the version string:

- `pyproject.toml:7` (currently `"0.0.0a1"`)
- `src/bonfire/__init__.py:13` (currently `"0.1.0"`)

**These disagree today.** A single-source-of-truth fix is mandatory before
tag. Two acceptable patterns:

**Option A — Have `__init__.py` read from importlib.metadata (preferred).**

```python
from importlib.metadata import PackageNotFoundError, version as _pkg_version

try:
    __version__ = _pkg_version("bonfire-ai")
except PackageNotFoundError:
    # Editable / unbuilt fallback — keep in lockstep with pyproject.toml
    __version__ = "0.1.0"
```

Pros: the runtime version matches the *installed* metadata, so editable
installs and built wheels can never disagree at import time. The fallback
literal is a maintenance scab but it's only consulted when the package
metadata is absent (fresh checkouts before `pip install -e .`).

**Option B — Have `pyproject.toml` read the version dynamically.**

Hatchling supports `[tool.hatch.version]` with `path = "src/bonfire/__init__.py"`
and `dynamic = ["version"]` in `[project]`. Single source of truth lives in
`__init__.py`; pyproject reads from it.

**Recommendation: Option A.** It's the smaller diff (no `[tool.hatch.version]`
section, no `dynamic` declaration), it's the idiom most well-known Python
packages adopt, and it correctly defers to the installed metadata at runtime
which is the version a user actually has.

**Fix manifest entry.** Edit `src/bonfire/__init__.py:13` to the Option A
snippet above, and update `pyproject.toml:7` to `"0.1.0"` in the same
commit so editable-install fallback and metadata both agree.
