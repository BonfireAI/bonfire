## E — Three gating strategies (now moot)

If/when doctests land:

- **(a) Global flag + per-module `# doctest: +SKIP`.** Most idiomatic for a Pydantic-heavy library. Each impure module gates itself.
- **(b) Path list in pyproject `addopts`.** Explicit allow-list. Scales worst (every new pure module needs a config edit).
- **(c) `collect_ignore` in `conftest.py`.** Programmatic gating. Best for "all pure, except these" defaults.

The right strategy depends on the eventual ratio of pure to impure modules. Without inventory, no strategy can be locked.

## F — Recommended path

**PARK BON-604.** Suggested path forward in order of likely value:

1. **Park status, no scope change.** Re-evaluate when the first doctest lands organically (a Warrior writes one as documentation in a pure module). At that point, the activation has a tangible win.
2. **Re-scope to "add doctests + activate" (M-sized).** Drop doctests into 2-4 pure modules deliberately (e.g. `dispatch/tiers.py`, `cost/ledger.py`, `models/envelope.py`) AND activate `--doctest-modules` in the same PR. Larger ticket but ships the protection AND the content. If Anta wants the safety net, this is the right shape.
3. **Reject permanently.** v0.1 doesn't need doctest infrastructure pre-launch. Close BON-604 as won't-fix; revisit post-v0.1.0.

Conservative recommendation: **#1 (park).** The flag without content is configuration without protection.
